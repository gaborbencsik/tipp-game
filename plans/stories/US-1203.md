# US-1203: Automatikus adatszinkron cron job

> Epic: E12 – Automatikus adatszinkron

**Story:**
Mint **fejlesztő**, szeretnék **egy Render Cron Job-ot, amely percenként triggerel és a backend smart gate logikája alapján dönti el, hogy fusson-e a szinkron**, hogy **az API quota ne pazarlódjon, az adatok mindig naprakészek legyenek, és ne kelljen manuálisan indítani a sync-et**.

## Architektúra

**Hybrid megközelítés:**
- **Render Cron Job** → fix 1 perces intervallum, HTTP POST a backend-re
- **Backend gate** → DB-ből olvassa `last_successful_sync_at` + `sync_mode` → eldönti: skip vagy futtat

```
[Render Cron 1 min] → POST /api/internal/sync/tick (service token)
                          ↓
                    [Gate logic – 1 DB read]
                          ↓
              skip (too early / mode=off)  OR  runSync(...)
```

## Gate logika

| Mode | Feltétel a futáshoz |
|------|---------------------|
| `off` | soha nem fut |
| `final_only` | `last_successful_sync_at` > 30 perce |
| `adaptive` | van live meccs (DB query) → > 2 perce; nincs live → > 15 perce |
| `full_live` | > 1 perce (gyakorlatilag minden tick) |

A "van-e live meccs" ellenőrzés olcsó lokális DB query (`SELECT 1 FROM matches WHERE status = 'live' LIMIT 1`), nem API hívás.

## Sync mode tárolás

- **DB-ben** (`sync_state` tábla) — restart után is megmarad
- Ha nincs sor a táblában → default `off` (biztonságos)
- Admin UI a DB-t olvassa/írja (`GET/PUT /api/admin/sync/settings`)
- `FOOTBALL_SYNC_MODE` env változó megszűnik — nem kell többé
- `sync-config.ts` in-memory logika kiváltásra kerül DB read-del

## Elfogadási kritériumok

- [ ] `sync_state` tábla DB-ben: `mode` (default: `off`), `last_successful_sync_at`, `api_calls_today`, `sync_in_progress`
- [ ] Ha nincs sor a `sync_state` táblában → mode = `off`
- [ ] Admin `GET/PUT /api/admin/sync/settings` a DB-ből olvassa/írja a mode-ot (nem in-memory)
- [ ] `sync-config.ts` in-memory logika eltávolítása, `FOOTBALL_SYNC_MODE` env nem kell többé
- [ ] Új endpoint: `POST /api/internal/sync/tick` — service token auth (env: `SYNC_SERVICE_TOKEN`)
- [ ] Gate logika: `shouldRunSync(mode, lastRunAt, hasLiveMatch): boolean` — pure, tesztelhető
- [ ] `last_successful_sync_at` oszlop a `sync_state` táblában (vagy létező tábla bővítés)
- [ ] `sync_in_progress` flag (advisory lock) — idempotencia védelem dupla trigger ellen
- [ ] API rate limit védelem: napi hívásszám tracking DB-ben, hard stop 80%-nál
- [ ] A gate "skip" esetén 200-as választ ad (nincs retry), body: `{ skipped: true, reason: "..." }`
- [ ] A gate "run" esetén lefuttatja `runSync`-et, frissíti `last_successful_sync_at`-ot
- [ ] Admin manuális trigger (`POST /api/admin/sync/run`) továbbra is működik — megkerüli a gate-et
- [ ] Minden futás audit log-ba kerül (már létező logika a `runSync`-ben)
- [ ] Unit tesztek: `shouldRunSync` pure függvény összes ága lefedve
- [ ] Integration teszt: tick endpoint skip/run viselkedés mockolt DB-vel

## Render konfiguráció

- **Service type:** Cron Job
- **Schedule:** `* * * * *` (minden perc, UTC)
- **Command:** `curl -s -X POST -H "Authorization: Bearer $SYNC_SERVICE_TOKEN" https://<backend-url>/api/internal/sync/tick`
- **Költség:** $1/hó minimum (per-second billing, max néhány másodperc/futás)

## Technikai részletek

**Új fájlok:**
- `src/routes/sync-tick.routes.ts` — internal endpoint + service token middleware
- `src/services/sync-gate.service.ts` — `shouldRunSync` logika + `last_successful_sync_at` kezelés
- `tests/sync-gate.test.ts` — pure gate tesztek

**Módosítások:**
- `src/db/schema/index.ts` — `sync_state` tábla hozzáadása
- `src/services/sync-config.ts` — átírás DB-read-re (in-memory logika törlése)
- `src/routes/sync.routes.ts` — settings endpoint DB-ből olvas/ír
- `src/app.ts` — tick route regisztráció
- Render Dashboard — új Cron Job service létrehozás
- `.env` — `FOOTBALL_SYNC_MODE` eltávolítása, `SYNC_SERVICE_TOKEN` hozzáadása

**Env változók:**
- `SYNC_SERVICE_TOKEN` — shared secret a Render cron és a backend között
- `FOOTBALL_SYNC_MODE` — eltávolítandó (DB veszi át a szerepét)

## Biztonsági megfontolások

- A `/api/internal/sync/tick` endpoint NEM admin auth-ot használ, hanem fix service tokent (a cron job-nak nincs user session-je)
- A token csak ezen az egy endpointon érvényes
- Rate limit a tick endpoint-on is (max 2 req/min) — védekezés brute force ellen
