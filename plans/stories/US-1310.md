# US-1310: Match Detail – AI-generált összefoglaló a legeredményesebb játékosokról

> Epic: E5 – Insights & Match Detail

## Háttér

A `player_stats` tábla (`goals`, `assists`, `appearances`, `conceded`) szinkronizálva van API-Football-ból, de a Match Detail oldalon csak indirekt módon (insights prompt) van felhasználva. **Önálló „top játékosok" összefoglaló nincs**, pedig a felhasználónak hasznos kontextus lenne a meccs előtt: ki a két csapat legjobb gólszerzője, assist-adója, legtöbbet játszó embere.

Az AI-insights infrastruktúra él (`packages/backend/src/services/insights/llm.client.ts`, `prompt.builder.ts`, `generator.service.ts`), és a reveal-gate (`useInsightsRevealStore`) is használt.

## Story

Mint **bejelentkezett felhasználó a Match Detail oldalon**, szeretném **a Polymarket / Transfermarkt blokkok közelében egy rövid (2–3 mondatos), magyar nyelvű AI-szöveget látni mindkét csapat legeredményesebb játékosairól** (gólszerző, assist-adó, legtöbbet játszó), hogy **gyors objektív kontextusom legyen a meccs kulcsemberein, az insights elemzés olvasása előtt vagy mellett**.

## Elfogadási kritériumok

### Backend — service

- [ ] Új service: `packages/backend/src/services/insights/top-players.service.ts`.
- [ ] Forrás: csapatonként a `players` rekordokhoz tartozó `player_stats` (legutóbbi `season`, prioritásban a kvalifikációs / klubbajnokság — pontos sorrendet a service dokumentálja). Csapatonként kiválasztja:
  - **Top gólszerző** (`MAX(goals)`).
  - **Top assist-adó** (`MAX(assists)`).
  - **Legtöbbet játszó** (`MAX(appearances)`).
  - **Opcionálisan kapus** (`MAX(appearances)` ahol `position = 'Goalkeeper'`, + `conceded`).
- [ ] Egy játékos több kategóriában is szerepelhet — duplikációt a service kezel (egy név, több címke).
- [ ] **Esetkezelés:**
  - Egyik csapathoz sincs `player_stats` → `null` válasz.
  - Csak az egyik csapathoz van adat → a szöveg csak arról szól („A vendég csapatról nincs elérhető statisztika.").

### Backend — LLM és cache

- [ ] Új prompt builder: `packages/backend/src/services/insights/top-players.prompt.builder.ts` — strukturált JSON inputot ad át (csapatnevek + 3-4 játékos + statok). A meglévő `prompt.builder.ts` mintáját követi.
- [ ] LLM client: a meglévő `llm.client.ts`. Modell és provider a meglévő config flow-ból, **hardcoded modell-név tilos**.
- [ ] **Új DB tábla:** `match_top_players_summary`:
  - `matchId UUID PK` (FK → `matches.id`).
  - `summaryHu TEXT NOT NULL` — a 2–3 mondatos AI-szöveg.
  - `playersJson JSONB NOT NULL` — strukturált lista: `[{ name, teamId, category, value }]`.
  - `inputHash TEXT NOT NULL` — a két csapat `player_stats.updated_at` + `player_id` halmazából képzett hash.
  - `generatedAt TIMESTAMPTZ NOT NULL DEFAULT NOW()`.
  - Index a `matchId`-ra.
- [ ] **Cache invalidálás:** ha az aktuálisan számolt `inputHash` eltér a tárolttól → újragenerálás. Egyébként cache-hit.
- [ ] Drizzle migráció + schema kiegészítés.

### Backend — endpoint

- [ ] `GET /api/matches/:id/top-players` →
  - 200: `{ summaryHu: string, players: Array<{ name, teamId, category: 'top_scorer' | 'top_assist' | 'most_appearances' | 'goalkeeper', value: number }> }`.
  - 404: ha a match nem létezik.
  - 204 (vagy `{ summaryHu: null, players: [] }`): ha mindkét csapathoz nincs `player_stats`.
- [ ] **Trigger:** elsősorban a meglévő insights cron job (US-1303 mintájára) generálja előre. Ha nincs cache hit a kérésnél, lazy-init **engedélyezett** (egy felhasználói kérés sem maradhat a háttér-job miatt szöveg nélkül, ha generálható).
- [ ] **Rate limit / hibakezelés:** LLM hiba vagy rate limit → ha van cache-elt rekord, azt adjuk vissza; ha nincs, 503 + a frontend elrejti a blokkot.

### Backend — tesztek

- [ ] **Unit teszt** a `top-players.service.ts`-re mockolt LLM client + DB fixture-rel:
  - Top játékosok kiválasztása csapatonként.
  - Duplikációk kezelése (egy játékos több kategóriában).
  - Üres állapotok (mindkét / egyik csapat).
  - `inputHash` stabilitása ugyanazon input mellett.
- [ ] **Integrációs teszt** az endpoint-ra: cache-hit, cache-miss → generálás, 4xx, 503 LLM-hibára.

### Frontend

- [ ] Új komponens: `packages/frontend/src/components/matches/TopPlayersInsight.vue`.
- [ ] A `MatchDetailView.vue`-ban a `MarketValuesBar.vue` UTÁN illesztve, **ugyanazon reveal-gate alatt** (`useInsightsRevealStore` flag, ugyanaz a blur class — ahogy az UX-030-ban van).
- [ ] **Vizuális minta** (a meglévő insights-blokkok stílusát követi):
  - Fejléc: „Top játékosok" (uppercase, `text-xs font-medium text-gray-500 tracking-wide`).
  - Wrapper: `bg-gray-50 border border-gray-200 rounded-lg p-4` (azonos a `MatchOddsBar.vue` / `MarketValuesBar.vue`-val).
  - Tartalom: a 2–3 mondatos AI-szöveg + alatta kompakt lista: kis ikon + név + címke (pl. „⚽ 8 gól", „🎯 5 assist", „🏃 24 meccs"). A lista a `playersJson` strukturált adatból renderel.
  - **Loading skeleton**, ha a backend még generálja (lazy-init eset).
- [ ] **Empty state:** ha a backend `null` / 204 / `summaryHu === null` választ ad, a komponens **nem renderel** (semmi üres dobozka).
- [ ] **`data-testid`-k:** `top-players-section`, `top-players-summary`, `top-players-list`, `top-player-row`.
- [ ] **Feature flag:** `VITE_FEATURE_TOP_PLAYERS_INSIGHT` (default `true` éles bevezetésig). A flag `false` értéke esetén a komponens nem renderel és a backend kérés sem indul.

### Frontend — tesztek

- [ ] Komponens teszt (Vitest):
  - Rendereli a szöveget és a listát adatra.
  - Empty state-ben (null válasz) elrejti.
  - Loading skeleton helyesen.
  - Feature flag false → nem renderel.
- [ ] **E2E happy-path** (Playwright): match detail oldal → reveal → top players blokk látható és tartalmazza a két csapat legalább egy kiemelt játékosát.

### Implementációs sorrend (javasolt)

1. DB migráció + schema.
2. Service skeleton (LLM nélkül, fix string) + endpoint + frontend komponens reveal-gate-tel és feature flag-gel.
3. LLM prompt + cache integráció.
4. Cron job integráció a meglévő insights generátorba.

## Mit ne csináljunk ebben a story-ban

- Ne használjuk a `player_stats.passAccuracy / keyPasses / duelsTotal / duelsWon` mezőket — külön nice-to-have, ha a prompt finomítása kéri.
- Ne duplikáljuk a Polymarket / Transfermarkt blokkok információit — fókusz **csak** a játékos-szintű statokon.
- Ne a Match Detail oldalon kívül jelenjen meg (lista, dashboard, leaderboard nem érintett).

## Általános elvek

- **TDD kötelező** a backend service-re (CLAUDE.md).
- Domain logika nem mockolva — csak az LLM client és a DB IO.
- Hardcoded modell-név tilos, minden a meglévő config flow-ból.

- **UX-030** (Transfermarkt market values blokk) — kész, a reveal-gate stílusát ettől örökli.
