# US-1403: Automatikus játékos-szinkronizáció futball API-ból

## Leírás

Mint **adminisztrátor**, szeretném, hogy a `players` tábla automatikusan feltöltődjön a futball API-ból a szinkronizált ligák csapatainak játékosaival, hogy a `player_select` típusú speciális tippek kérdéseinél valós adatokból tudjon a felhasználó választani.

## Jelenlegi helyzet

- `players` tábla létezik a sémában (`id`, `name`, `teamId`, `position`, `shirtNumber`, `externalId`)
- A `players` táblát jelenleg csak admin CRUD-dal lehet feltölteni (manuálisan)
- `football-api.service.ts` (`FootballApiClient`) csak `fetchFixtures` és `fetchTeams` metódust tartalmaz — `fetchPlayers` nincs
- `sync.service.ts` szinkronizálja a meccseket és csapatokat, de játékosokat nem
- `sync-runner.ts` futtatja a szinkront, a Render Cron job triggereli
- A `teams` táblában van `externalId` (API-sports team ID) — ez alapján lekérhetők a csapat játékosai
- A csapatok ligához rendelése a `matches` táblából vezethető le (`league_id + home_team_id / away_team_id`)

## Elfogadási kritériumok

### Backend — football-api.service.ts bővítés

- [ ] Új interface a játékos API-válaszhoz: `ApiFootballPlayer` (tartalmaz: `player.id`, `player.name`, `player.firstname`, `player.lastname`, `statistics[0].games.position`, `statistics[0].games.number`, `statistics[0].team.id`)
- [ ] Új interface: `PlayerParams { readonly teamId: number; readonly season: number }`
- [ ] `FootballApiClient` interfészbe kerül: `fetchPlayers(params: PlayerParams): Promise<ApiFootballResponse<ApiFootballPlayer>>`
- [ ] `createFootballApiClient` implementálja a `fetchPlayers` metódust — endpoint: `GET /players?team={teamId}&season={season}`

### Backend — sync.service.ts bővítés

- [ ] Új pure helper: `mapApiPlayerToDb(apiPlayer: ApiFootballPlayer, teamInternalId: string): PlayerInsert`
  - `name`: `player.name` (fallback: `${player.firstname} ${player.lastname}`)
  - `position`: `statistics[0].games.position ?? null`
  - `shirtNumber`: `statistics[0].games.number ?? null`
  - `externalId`: `player.id`
  - `teamId`: `teamInternalId`
- [ ] Új függvény: `syncPlayersForLeague(leagueExternalId: number, season: number, client: FootballApiClient): Promise<SyncRunResult>`
  - Meghatározza a ligához tartozó csapatokat a `matches` táblából (`DISTINCT home_team_id UNION away_team_id WHERE league_id = ?`), ahol a `teams.externalId` nem null
  - Minden csapatra hívja a `fetchPlayers` API-t (rate limit kezelés: 429 esetén exponenciális backoff, már megvan a service-ben)
  - Minden játékost upsert a `players` táblába `externalId` alapján:
    - Ha létezik `externalId` → `UPDATE` (`name`, `position`, `shirtNumber`, `teamId`, `updatedAt`)
    - Ha nem létezik → `INSERT`
  - Visszatér: `{ inserted: number; updated: number; errors: string[] }`
- [ ] Ha a liga nem tartalmaz meccseket (nem deríthetők le csapatok) → üres result, nem dob hibát
- [ ] Ha egy csapat `externalId`-ja null → kihagyja (log figyelmeztetés), nem dob hibát
- [ ] Ha az API rate limitet dob → az adott csapat átugrásra kerül, az error kerül a `result.errors` tömbbe

### Backend — sync-runner.ts bővítés

- [ ] A meglévő sync-runner meghívja a `syncPlayersForLeague`-t minden konfigurált liga után
- [ ] A futás eredménye (inserted/updated/errors) logolásra kerül (strukturált log, nem `console.log`)
- [ ] Ha a players sync meghiúsul (nem rate limit, hanem általános hiba) → a teljes sync nem szakad meg, az error logolódik

### Tesztelés

- [ ] Unit teszt: `mapApiPlayerToDb` — helyes mapping, null mezők kezelése, name fallback logika
- [ ] Unit teszt: `syncPlayersForLeague` — mock client, mock db: upsert logika (insert vs update), rate limit skip, null externalId skip
- [ ] Unit teszt: `fetchPlayers` — helyes URL felépítés (`/players?team=X&season=Y`)

## Technikai megjegyzések

### API végpont

Az `api-sports.io` (v3) `/players` endpoint:
```
GET https://v3.football.api-sports.io/players?team={teamId}&season={season}
```
Válasz struktúra: `response[].player.id`, `response[].player.name`, `response[].statistics[0].games.position`, `response[].statistics[0].games.number`, `response[].statistics[0].team.id`

### Upsert pattern (Drizzle)

```typescript
await db.insert(players)
  .values(playerData)
  .onConflictDoUpdate({
    target: players.externalId,
    set: { name, position, shirtNumber, teamId, updatedAt: sql`now()` },
  })
```

### Liga → csapat kapcsolat (nincs direkt FK)

```sql
SELECT DISTINCT home_team_id AS team_id FROM matches WHERE league_id = $1 AND deleted_at IS NULL
UNION
SELECT DISTINCT away_team_id FROM matches WHERE league_id = $1 AND deleted_at IS NULL
```

Majd a belső UUID-khoz az `externalId` kiolvasása a `teams` táblából.

### Rate limit kezelés

Az `api-sports.io` ingyenes tier limitje: 100 kérés/nap. Egy ligában átlagosan 20-32 csapat → 20-32 API hívás. Több liga esetén a napi limitet figyelembe kell venni — a `syncPlayersForLeague` ne hívjon párhuzamosan, hanem sorban (szekvenciálisan).

### `externalId` egyediség

A `players.externalId` az `api-sports.io` globális játékos ID — egyedi constraint van rá a sémában, az upsert erre támaszkodik.

## Nem tartalmazza

- Frontend változás (a meglévő `PlayerSelectCombobox.vue` és `GET /api/players` endpoint változatlan marad)
- Ligaszűrés a players API-ban (az US-1402 feladata)
- Admin felületi változás a players kezeléshez
- Játékos fotó / avatar szinkronizáció
- Játékos statisztikák (gól, assziszt) szinkronizáció
- Kézzel felvett (externalId nélküli) játékosok törlése / felülírása a sync során

## Komplexitás

M — meglévő sync pattern bővítése új API endpoint + upsert logikával, sorban futó csapat-iterációval
