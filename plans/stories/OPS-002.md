# OPS-002: Render cron service-ek konfigurálása az automatikus sync-hez

## Probléma

Production-ben a 3 internal sync endpoint **soha nem fut** automatikusan. A backend kód (gate logika, optimistic lock, scoring trigger, polymarket / transfermarkt integrációk) működik, de **nincs külső scheduler ami meghívja** ezeket az endpoint-okat.

**Tünet:** meccs eredmények nem frissülnek, pontok nem számolódnak, odds elavult, piaci értékek statikusak — csak akkor, ha admin manuálisan futtatja a `/admin/sync/*` parancsokat.

**Root cause:** hiányzó cron service-ek Render-en + hiányzó `SYNC_SERVICE_TOKEN` env var.

---

## Story

Mint **adminisztrátor**, szeretném, hogy a **3 fő sync endpoint automatikusan fusson production-ben** (meccs eredmény + élő odds + piaci értékek), hogy **ne kelljen manuálisan triggerelnem** a sync-eket.

---

## Hatókör – 3 cron service

A backend `src/routes/internal-sync.routes.ts` 3 publikus internal endpointot tesz közzé, mind `serviceTokenMiddleware` mögött:

| Endpoint | Cél | Belső gate | Cron schedule |
|----------|-----|------------|---------------|
| `POST /api/internal/sync/tick` | Meccs eredmény + gólszerzők + (24h-ként) player sync | `sync-gate.ts` (mode-tól függő interval, optimistic lock) | `* * * * *` (percenként) |
| `POST /api/internal/sync/polymarket-tick` | Polymarket odds frissítés | Csak az `polymarketSyncEnabled` flag | `*/15 * * * *` (15 percenként) |
| `POST /api/internal/sync/transfermarkt-tick` | Csapat-piaci értékek | Belső 24h interval gate az endpoint-on | `0 4 * * *` (napi 1×, 04:00 UTC) |

**Megjegyzés:** A `/tick` futás végén a player sync automatikusan lefut, ha 24h+ telt el — külön cron nem kell rá.

---

## Elfogadási kritériumok

### 1. `SYNC_SERVICE_TOKEN` setup

- [ ] Generálva: kriptográfiailag random, legalább 32 byte (`openssl rand -hex 32` vagy `node -e "console.log(require('crypto').randomBytes(32).toString('hex'))"`)
- [ ] Beállítva a backend `tipp-game-backend` web service env vars közé Render dashboard-on
- [ ] `.env.example`-be hozzáadva placeholder-rel: `SYNC_SERVICE_TOKEN=replace-me-with-random-secret`
- [ ] Lokális `.env`-be is bevezetve (fejlesztéshez), hogy a service-token middleware ne 401-ezzen

### 2. `render.yaml` bővítés – 3 cron job service

Mindegyik cron service:

- `type: cron`
- `runtime: docker` vagy `runtime: image: alpine` (curl-höz minimális)
- `region: frankfurt` (azonos a backenddel → internal hálózat)
- `plan: starter` ($1/hó/job × 3 = ~$3/hó)
- `envVars`: `SYNC_SERVICE_TOKEN` (ugyanaz az érték mint a backend service-en) + `BACKEND_URL` (az internal Render URL: `http://tipp-game-backend:3000`)
- `command`: `curl --fail-with-body -sS -X POST -H "Authorization: Bearer $SYNC_SERVICE_TOKEN" $BACKEND_URL/api/internal/sync/<endpoint>`

Schedule (UTC):

- [ ] `tipp-game-cron-tick` — `* * * * *` → `/tick`
- [ ] `tipp-game-cron-polymarket` — `*/15 * * * *` → `/polymarket-tick`
- [ ] `tipp-game-cron-transfermarkt` — `0 4 * * *` → `/transfermarkt-tick`

### 3. Verifikáció

- [ ] Render dashboard, mind a 3 cron job logjában látszik a sikeres curl exit code (0) és a backend response body (`{"synced": true, ...}` vagy `{"skipped": true, "reason": "..."}`)
- [ ] Backend logok (production console, OPS-001 után strukturált log): `sync.tick.received`, `polymarket.sync.completed`, `transfermarkt.sync.completed` event-ek a megfelelő ütemben
- [ ] Élő meccs alatt: `/tick` 2 percenként ténylegesen futtat (gate `adaptive` mode + `hasLive=true`), eredmény frissül
- [ ] `/polymarket-tick` 15 percenként új odds írást eredményez (DB-ben `polymarket_odds.updated_at` mozog)
- [ ] `/transfermarkt-tick` napi 1× lefut, második napi futás `skipped: transfermarkt sync ran less than 24h ago`-val visszaadja
- [ ] **Negatív teszt:** rossz vagy hiányzó token → 401, cron job exit code ≠ 0, Render dashboard piros státusz

### 4. Dokumentáció

- [ ] `.env.example`: `SYNC_SERVICE_TOKEN` placeholder + 1-2 soros komment
- [ ] `README.md` (root): rövid szekció a 3 cron job-ról + arról, hogy lokálisan node-cron / kézi curl helyettesíti őket fejlesztés közben
- [ ] `render.yaml` kommentekkel: melyik cron melyik endpointot hívja és milyen sűrűn, miért

## Technikai megjegyzések

- A `service-token.middleware.ts` `crypto.timingSafeEqual` alapú összehasonlítást használ — a két env var pontosan ugyanaz kell legyen (azonos hossz + bytes). Render dashboard-on copy-paste hibát kerülni kell, ezért inkább "Add from another service" / "Sync" funkció ha lehet.
- A `sync-gate.ts` a `/tick`-en már kezeli az interval gate-et (`final_only: 30m`, `adaptive: live=2m / idle=15m`, `full_live: 1m`), tehát a percenkénti curl nem hív feleslegesen API-t — a backend skip-eli ha még nem telt el az interval.
- A `/polymarket-tick`-nek **nincs belső interval gate**, így a cron schedule (15 perc) határozza meg a hívási sűrűséget — élő meccs alatt elég gyakori, közben pedig nem terheli az API-t.
- A `/transfermarkt-tick` belső 24h-os gate-tel rendelkezik az endpoint-on belül, így a cron schedule csak biztosítja, hogy naponta legalább egyszer megpróbálja. Ha a Transfermarkt scrape lassú vagy fail-el, a következő napi hívás újra próbálja.
- **Költség:** 3 starter cron job × $1/hó = ~$3/hó plusz a meglévő starter web service ($7/hó) felett.
- **Régió:** `frankfurt` — fontos, hogy a cron service és a web service azonos régióban legyen, hogy az internal hálózati URL működjön (privát hálózati hívás, nem publikus internet).
