# OPS-001: Strukturált logolás és frontend hibakezelés (MVP)

## Probléma

Az alkalmazás jelenleg `console.log` / `console.error` hívásokkal logol, adhoc módon, request-korreláció nélkül. Egy user-panaszra ("nem ment a tippem") jelenleg nem lehet gyorsan kideríteni, mi történt:

- Backend: nincs request ID, nincs JSON-formátumú log, nincs strukturált event-katalógus.
- Frontend: nincs `app.config.errorHandler`, nincs `unhandledrejection` listener, a hibák silent szivárognak el.
- Nincs központi log-aggregátor (Better Stack / Sentry / hasonló).

**Trigger:** egy production bug, ahol az adaptive sync nem futott le. A diagnosztika manuális volt — fejlesztő SSH-zott a Render logokat. Ez nem skálázható.

> **Megjegyzés:** A bug *konkrét* root cause-a (hiányzó Render cron job) az **OPS-002** story scope-ja, nem ennek. Itt a reprodukálhatóság infrastruktúráját rakjuk le.

---

## Story

Mint **fejlesztő**, szeretnék **strukturált logolást request-korrelációval és centralizált log-aggregátort**, hogy **egy user panasza esetén percek alatt megtaláljam a vonatkozó backend és frontend log-okat user_id + timestamp alapján**.

---

## Elfogadási kritériumok

### A — Backend strukturált logolás

1. **Pino logger wrapper**
   - [ ] `packages/backend/src/utils/logger.ts` — pino-alapú wrapper, `LOG_LEVEL` env var olvas, ISO 8601 timestamp, `service: 'backend'`, `env`, `release` (`RENDER_GIT_COMMIT` vagy `dev`)
   - [ ] `redact` paths: `req.headers.authorization`, `req.headers.cookie`, `*.password`, `*.token`, `*.refreshToken`, `*.email`, `*.code`, `*.state`
   - [ ] `requestLogger(ctx)` child logger: `request_id`, `user_id` injektálva ctx.state-ből

2. **Request ID middleware**
   - [ ] `packages/backend/src/middleware/request-id.middleware.ts` — UUID generálás, `ctx.state.requestId`-ba teszi, `X-Request-Id` response header-ben visszaadja
   - [ ] Bekötve `error.middleware` előtt

3. **Error middleware fejlesztés**
   - [ ] 5xx esetén strukturált log: `event: 'http.request.failed'`, `data: { method, route, status, duration_ms, error: { name, message, stack } }`
   - [ ] 4xx logolás csak `401`, `403`, `429` esetén warn szinten
   - [ ] Stack trace teljes mértékben (production source map-pel — későbbi follow-up)

4. **Discriminated union event-típus**
   - [ ] `packages/backend/src/utils/log-events.ts` — TS discriminated union, **nem** `data: object`
   - [ ] Példa:
     ```typescript
     export type LogEvent =
       | { event: 'http.request.completed'; data: HttpCompletedData }
       | { event: 'http.request.failed'; data: HttpFailedData }
       | { event: 'sync.tick.completed'; data: SyncTickCompletedData }
       | { event: 'sync.tick.skipped'; data: SyncTickSkippedData }
       | { event: 'sync.tick.failed'; data: SyncTickFailedData }
       | { event: 'scoring.calculation.completed'; data: ScoringCompletedData }
       | { event: 'scoring.calculation.failed'; data: ScoringFailedData }
       | { event: 'auth.login.success'; data: AuthLoginSuccessData }
       | { event: 'auth.login.failed'; data: AuthLoginFailedData }
       | { event: 'frontend.error.unhandled'; data: FrontendErrorData };
     ```
   - [ ] `log(event: LogEvent, msg: string)` wrapper — fordítási idejű ellenőrzés event ↔ payload párokra
   - [ ] **Csak ~10 eseménnyel induljunk**, többit add hozzá amikor a call site igényli
   - [ ] `outcome: 'success' | 'failed' | 'skipped'` opcionális envelope mező a könnyű cross-subsystem szűréshez

5. **Service-level logolás (kritikus pontok)**
   - [ ] `auth.middleware`: JWT decode hiba → `auth.login.failed` warn
   - [ ] `sync.service.runAllLeagues`: tick start/end/skip → `sync.tick.*` info
   - [ ] `scoring.service`: hiba → `scoring.calculation.failed` error
   - [ ] `predictions.service`: rejected esetek → `prediction.rejected` warn (ha az event-katalógusba bekerül)

### B — Frontend hibakezelés

1. **Globális Vue error handler**
   - [ ] `packages/frontend/src/main.ts` — `app.config.errorHandler` beállítva
   - [ ] User-nak toast: "Hiba történt. (id: <user_id> / <ts>)" — **user_id + timestamp a recovery key**, nem request_id
   - [ ] Backend-re POST: `frontend.error.unhandled` payload (auth header-rel, ha bejelentkezve)

2. **Unhandled promise rejection listener**
   - [ ] `window.addEventListener('unhandledrejection')` — analóg flow

3. **Fetch wrapper request-id capture**
   - [ ] `packages/frontend/src/api/fetch.ts` — `X-Request-Id` response header begyűjtve, megjelenik a network hibák toast-jában
   - [ ] 5xx hibánál user toast: "Szerver hiba. (request: <id>)"

4. **Frontend log endpoint**
   - [ ] `POST /api/internal/logs` — auth nélkül elérhető, **csak** `frontend.error.*` event prefix-szel
   - [ ] Egyéb prefix: 400 Bad Request
   - [ ] Egyszerű rate limit: 60 req / perc / IP (létező rateLimit middleware újrahasználva)

### C — Log-aggregátor választás és integráció

1. **Vendor választás**
   - [ ] Better Stack (Logtail) ajánlás — pino transport (`@logtail/pino`), Render-rel jó integráció
   - [ ] Free tier: 1 GB/hó (idle napokra elég); fizetős: ~$10/hó 5 GB
   - [ ] Alternatíva: Sentry (frontend error capture + backend) — externe is jó, de pino → Better Stack egyszerűbb

2. **Setup**
   - [ ] `LOGTAIL_SOURCE_TOKEN` env var Render-en
   - [ ] Pino transport bekapcsolva production env-ben
   - [ ] Egy minta query dokumentálva: `event:"frontend.error.unhandled" AND user_id:"<uuid>"` — user-panasz reprodukció

### D — Tesztek

1. **Unit tesztek**
   - [ ] `logger.test.ts` — JSON output, level, discriminated union compile-time check (egy `// @ts-expect-error` annotált rossz event/data páros)
   - [ ] `log-scrubber.test.ts` — email, JWT, OAuth code, cookie redact-elve egy stub log object-en

2. **Integration teszt**
   - [ ] Request ID middleware: GET request → response `X-Request-Id` header, log-ban azonos érték

## Technikai megjegyzések

### Boríték (envelope) — minden log

```json
{
  "ts": "2026-05-25T18:42:11.123Z",
  "level": "info",
  "service": "backend",
  "env": "production",
  "release": "abc1234",
  "event": "sync.tick.completed",
  "outcome": "success",
  "request_id": "req_01HX9K8M3F7P2N",
  "user_id": null,
  "msg": "Adaptive sync tick completed",
  "data": { }
}
```

| Mező | Mindig / Opcionális | Magyarázat |
|------|------|------|
| `ts` | Mindig | ISO 8601 UTC |
| `level` | Mindig | pino szabvány: debug/info/warn/error/fatal |
| `service` | Mindig | `backend` vagy `frontend` |
| `env` | Mindig | `production`/`staging`/`development` |
| `release` | Mindig | `RENDER_GIT_COMMIT` rövidített hash |
| `event` | Mindig | discriminated union, fordítási időben ellenőrzött |
| `outcome` | Opcionális | `success`/`failed`/`skipped` — cross-subsystem szűrésre |
| `request_id` | HTTP context-ben | UUID, response header-ben is |
| `user_id` | Auth context-ben | UUID — recovery key (user_id + ts query) |
| `msg` | Mindig | Statikus szöveg (max 80 char), nincs interpoláció |
| `data` | Mindig | Event-specifikus payload, discriminated union-ban típusos |

**Mellőzött mezők (council javaslat):**
- `trace_id` — kihagyva, jelenleg `request_id`-val redundáns. Ha cron tick → nested HTTP flow valóban kialakul, akkor adjuk hozzá.
- `severity_number`, `schema_version`, `data_schema`, `span_id`, `parent_span_id` — OTel-prep, túl korai.

### User panasz → log lookup

A felhasználónál nincs request_id a kezében. A működő flow:

1. User megadja: "kb 14:30-kor tippeltem és nem ment"
2. Admin: `user_id:"<uuid>" AND ts:["14:25" TO "14:35"]` query a Better Stack-ben
3. Frontend `frontend.error.unhandled` és backend `http.request.failed` event-ek visszakeresve
4. Ha frontend hiba van: `request_id` mező → backend lookup ugyanazon a request_id-n

## Kizárások (későbbi story-k)

- **Render cron service konfiguráció** → **OPS-002** (külön story, 1 órás munka)
- **Source map upload pipeline** (Vercel build → Better Stack)
- **Sentry SDK integrálása** (alternatíva ha Better Stack nem jön be)
- **OpenTelemetry migráció** (legalább 6+ hónap múlva, ~50-soros mapping)
- **GDPR HMAC user_id hash, IP salt rotation** — gate-elve DPA-trigger eseményhez vagy >1k DAU-hoz
- **Scheduler heartbeat / missed_tick detector** — gate-elve egy valódi missed-tick incidenshez (jelenleg cron sincs amit detektálnánk)
- **`/health/logger` endpoint, stack sanitizer** — gate-elve egy valódi pino transport queue dugulás incidenshez
- **Breadcrumb trail, sendBeacon batch** — gate-elve a frontend error volume valódi szükséglethez
