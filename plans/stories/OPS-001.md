# OPS-001: Logolás és hibakezelés átnézése

## Probléma — Audit státusza (2026-05-25)

Az alkalmazás jelenleg nem rendelkezik strukturált megoldással a user-szintű hiba detektálásra és reprodukálhatóságra:

### Backend

- `packages/backend/src/middleware/error.middleware.ts:23–25` — 5xx hibák alapvetően logolva (`console.error` JSON-ként), de az error log struktura adhoc (nem garantált a kontextus)
- `packages/backend/src/index.ts:12–19` — `console.log` az entrypoint-ban, nem strukturált
- `packages/backend/src/middleware/auth.middleware.ts:71` — JWT validation hiba logolva `console.error`, de nem tartalmaz request context (user ID, IP, endpoint)
- `packages/backend/src/services/user.service.ts:37–54` — adatbázis constraint error `catch` blokk nincs logolva, szilent nyeljük
- Számos route-ban (`packages/backend/src/routes/sync.routes.ts:40–50` és hasonlók) inline validation hibák `ctx.body`-ba kerülnek, de nem logolódnak Backend-re
- **Hiányzik:** request ID / correlation ID — nem lehet összekapcsolni egy frontend hibajelentést a backend loggal
- **Hiányzik:** strukturált logger (pino / winston / custom wrapper) — log szintezés, metadata formatálás konzisztens

### Frontend

- `packages/frontend/src/main.ts:1–21` — nincsen globális Vue error handler (`app.config.errorHandler` beállítva)
- `packages/frontend/src/stores/matches.store.ts:66–67` — fetch hibák az `error` ref-be kerülnek, de a user-nak nem jelzünk toast-ot vagy error page-et
- **Hiányzik:** `window.onerror` és `unhandledrejection` listener
- **Hiányzik:** Sentry / monitoring integrácio — a produkción terjedő hibák nem detektálódnak proaktívan
- **Hiányzik:** szándékos user error telemetria — fejlesztő nem tudja, hogy egy konkrét usernél mi a probléma

### Reprodukálhatóság

- Egy user azt jelzi: "nem működik a tippem rögzítése"
- Backend log nincs estructura, a user ID, endpoint, payload, timestamp nem könnyen elkülöníthető
- Frontend hiba stack trace nincs source map-ezve
- Request tracking: hiányzik — nem lehet nyomon követni egy adott HTTP call-t frontend → backend között

### Konfigurációs hiányok

- Production error monitoring eszköz **nincs** (nem Sentry, nem Better Stack, nem Datadog)
- Sensitive data scrubbing nincs az adatok a logban (pl. email-ek, JWT-k részlegesen logolódhatnak)

---

## Story

Mint **fejlesztő / adminisztrátor**, szeretnék **strukturált logolást, request tracking-et és centralizált error monitoring-ot**, hogy **tudomást szerzek róla, ha valahol valami rosszul működik egy usernél, és reprodukálni tudjam a hibát mélyebb diagnosztika nélkül**.

## Elfogadási kritériumok

### A — Backend strukturált logolás (kötelező)

1. **Logolási infrastruktúra**
   - [ ] Egy centrális logger / logger wrapper (`packages/backend/src/utils/logger.ts`) implementálva
   - [ ] Logger támogat különböző szinteket: `debug`, `info`, `warn`, `error`
   - [ ] Logger JSON kimenet ad (`{timestamp, level, requestId, userId, endpoint, message, stack, ...}`)
   - [ ] Logger wrapper elég ahhoz, hogy console.log helyett logolni lehessen bármely service-ben
   - [ ] Source: `packages/backend/src/middleware/error.middleware.ts` és `packages/backend/src/index.ts` is ezt használja

2. **Request ID middleware**
   - [ ] Új middleware: `packages/backend/src/middleware/request-id.middleware.ts`
   - [ ] Minden beérkező HTTP request kap egyedi ID-t (UUID vagy timestamp-based)
   - [ ] Request ID a response header-ben vissza (`X-Request-Id`)
   - [ ] Request ID a `ctx.state` vagy context elérésben tárva, hogy a service-ek felhasználhassák
   - [ ] Request ID minden log entry-ben megjelenik

3. **Error middleware fejlesztés**
   - [ ] 5xx hibák mellett a részletesebb kontextust logoljuk: `method`, `url`, `status`, `requestId`, `userId` (anonimizáltan vagy hashed), `userIp` (opcionális)
   - [ ] 4xx hibák naplózása szűrőzve (csak kritikus, pl. 401, 403, vagy payload validation 400)
   - [ ] Stack trace logolva 5xx-nél, de source map-ezve a production build-ben
   - [ ] Sensitive data scrubbing: JWT, e-mail, jelszó **soha nem** kerül a logba

4. **Service-level logolás**
   - [ ] Kritikus service-ek (auth, prediction, user, sync) legalább `info` szinten naplóznak érdekes műveleteknél
   - [ ] Pl. `[auth.middleware] user login attempt: userId=xyz, success=true` (logolva, de email nélkül)
   - [ ] Pl. `[predictions.service] upsertPrediction failed: matchId=abc, error=Match not found`
   - [ ] Auth hiba (JWT decode, token expired) logolva `info` szinten — nem panic, de nyomvonal kell

5. **Admin sync logolás**
   - [ ] `packages/backend/src/routes/sync.routes.ts` — végzett sync műveletek logolva: start, lépések (polymarket, player, transfermarkt, raw stats), végeredmény
   - [ ] Pl. `[sync] Polymarket sync started: apiCalls=45, duration=2500ms`

### B — Frontend error handling (kötelező)

1. **Globális Vue error handler**
   - [ ] `packages/frontend/src/main.ts` — `app.config.errorHandler` beállítva, ami:
     - [ ] Logol a konzolon / dev environment-ben (stack trace)
     - [ ] Küldi a request ID-t (ha elérhető) a backend-re (hibajelentés endpoint vagy monitoring service)
     - [ ] User-nak mutat egy toast: "Valami hiba történt. Kérjük, próbálkozzon később." + request ID (ha prod)

2. **Unhandled promise rejection listener**
   - [ ] `window.addEventListener('unhandledrejection')` — tetszőleges uncaught promise hibák kezelve (hasonló logolás)

3. **Fetch wrapper / API error handling**
   - [ ] Frontend API wrapper (`packages/frontend/src/api/*`) — fetch hibák (network, 5xx, timeout):
     - [ ] Request ID-t magukkal vittek (backend header-ből, pl. `X-Request-Id`)
     - [ ] Hiba logolódik: `[api] GET /api/matches failed: requestId=xyz, status=500, message=Internal Server Error`
     - [ ] User-nak toast jelzés: "A mérkőzések betöltése sikertelen volt. (Request ID: xyz)"

4. **Breadcrumb trail (opcionális, de ajánlott)**
   - [ ] Router navigation és kritikus felhasználói akciók (tipp rögzítés, csoport módosítás) "breadcrumb"-ként rögzítve (lokális tárolóban vagy memory-ben)
   - [ ] Ha hiba történik, a breadcrumb-öket a backend-re küldöd: `user navigated to /matches → predicted match ABC → error`

### C — Production error monitoring (szokványesítés, nem implementálás)

### Logger implementáció vázlat (puro TypeScript, extra package nélkül)

```typescript
// packages/backend/src/utils/logger.ts
interface LogEntry {
  timestamp: string
  level: 'debug' | 'info' | 'warn' | 'error'
  message: string
  requestId?: string
  userId?: string
  endpoint?: string
  status?: number
  stack?: string
  [key: string]: unknown
}

const scrubSensitiveData = (obj: Record<string, unknown>): Record<string, unknown> => {
  const copy = { ...obj }
  if (copy.email) copy.email = `${String(copy.email).substring(0, 2)}***@example.com`
  if (copy.token) copy.token = '***'
  if (copy.authorization) copy.authorization = 'Bearer ***'
  return copy
}

export const logger = {
  info: (message: string, meta?: Record<string, unknown>) => {
    const entry: LogEntry = {
      timestamp: new Date().toISOString(),
      level: 'info',
      message,
      ...scrubSensitiveData(meta ?? {}),
    }
    console.log(JSON.stringify(entry))
  },
  error: (message: string, err?: Error | unknown, meta?: Record<string, unknown>) => {
    const entry: LogEntry = {
      timestamp: new Date().toISOString(),
      level: 'error',
      message,
      ...(err instanceof Error && { stack: err.stack }),
      ...scrubSensitiveData(meta ?? {}),
    }
    console.error(JSON.stringify(entry))
  },
}
```

### Request ID middleware vázlat

```typescript
// packages/backend/src/middleware/request-id.middleware.ts
import type { Context, Next } from 'koa'
import { randomUUID } from 'crypto'

export async function requestIdMiddleware(ctx: Context, next: Next): Promise<void> {
  const requestId = randomUUID()
  ctx.state.requestId = requestId
  ctx.set('X-Request-Id', requestId)
  await next()
}
```

### Frontend error handler vázlat

```typescript
// packages/frontend/src/utils/error-handler.ts
export function setupGlobalErrorHandler(app: App) {
  app.config.errorHandler = (err: unknown, instance, info: string) => {
    const message = err instanceof Error ? err.message : String(err)
    const stack = err instanceof Error ? err.stack : undefined
    
    console.error(`[Vue Error] ${info}`, { message, stack })
    
    // Toast mutatása
    // showErrorToast(`Hiba történt: ${message}`)
    
    // Request ID-val szerver felé küldés (opcionális, monitoring eszközhöz)
  }
  
  window.addEventListener('unhandledrejection', (event) => {
    const message = event.reason instanceof Error ? event.reason.message : String(event.reason)
    console.error(`[Unhandled Promise] ${message}`, event.reason)
    // showErrorToast(`Promise rejection: ${message}`)
  })
}
```
