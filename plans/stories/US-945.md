# US-945: Csoport végeredmény tipp – 12 csoport sorrendje + 4 továbbjutó 3. helyezett

> Forrás: design draft `docs/superpowers/specs/2026-05-29-group-standings-prediction-design.md` (v1) és `…-v2.md` (v2 — pozíció-first dropdown)
>
> Megjegyzés: a design specekben szereplő "US-938" hivatkozás ID-ütközés miatt erre a storyra (US-945) cserélve.

## User Story

Mint VB tippelő, szeretnék a torna **mind a 12 csoportja (A–L)** végső sorrendjét megtippelni — minden csoportban az 1–4. helyet — és emellett kiválasztani a **4 továbbjutó 3. helyezett csapatot**, hogy a kieséses ágra (US-946) is lefutó tippet tudjak adni, és minden részmódosításom **automatikusan mentődjön** anélkül, hogy explicit "Mentés" gombot kellene nyomnom.

## Cél, kontextus

Új special prediction típus, amely a felhasználótól a torna **12 csoportjának sorrendjét** és a **4 továbbjutó 3. helyezett csapatot** kéri **egyetlen összefüggő tippben**. A tipp **egyetlen `special_predictions` rekord**, JSON-encoded answerrel, **egyetlen egységes deadline** (a torna első csoportmeccsének kickoffja), **auto-save** a részleges állapotokhoz.

| Story | Funkció | Megosztott infrastruktúra |
|-------|---------|---------------------------|
| US-935 | Globális statisztikai bonus tippek | Egységes deadline, "Torna tippek" tab |
| US-936 | Upset Special (multi-csapatos, súlyozott) | `multi_team_weighted` enum, JSON answer |
| US-937 | Progresszió tippek (Last 32 / 16 / QF / SF / döntő) | `multi_team_select`, `group_standing` enum, `points_per_correct` |
| **Ez a story (US-945)** | **Csoport végeredmény (12× sorrend + 4 best 3rd)** | Az US-937-ben bevezetett `group_standing` inputType **kibővítése** `all_groups_standing` értékkel. |
| US-946 | **Bracket-progresszió** (kieséses ág párokkal) | **Hard dependency erre a storyra** — slot resolver innen olvas. |

A **pontozási logika** (poszícióhelyes / all-or-nothing / hibrid) **nem ennek a story-nak a hatóköre**; külön story dönti el. A tipp tárolható és módosítható deadline-ig akkor is, ha a pontozási konfig még nincs megadva.

> **Implementációs sorrend:** US-937 → US-945 (ez) → US-946. Ha az US-937 még nincs implementálva, ez a story self-contained: bevezeti a szükséges enum értékeket önállóan.

## Adatmodell

### `special_prediction_types` rekord (seed)

```ts
{
  name: 'Csoport végeredmény',
  description: 'Tippeld meg a 12 csoport (A–L) végső sorrendjét és válassz 4 továbbjutó 3. helyezett csapatot.',
  inputType: 'all_groups_standing',     // új enum érték
  isGlobal: true,
  isActive: true,
  deadline: '<első csoportmeccs kickoff>',
  options: {
    groups: ['A','B','C','D','E','F','G','H','I','J','K','L'],
    teamsPerGroup: 4,
    best3rdPicks: 4,
  },
  // pointsPerCorrect / points: 0 / null  (külön story tölti ki)
}
```

Csak **egy** rekord; `groupId: NULL` (globális). A `options.groups` lista a torna konfig alapján bővíthető, de a default a 12 csoport.

### `special_predictions.answer` (felhasználó tipp)

JSON-string a `text` mezőben (US-936 mintája szerint, backward compatible):

```json
{
  "groups": {
    "A": ["uuid-mex", "uuid-kor", "uuid-cze", "uuid-rsa"],
    "B": ["uuid-can", "uuid-sui", null, null],
    "C": [null, null, null, null],
    "...": "..."
  },
  "best3rds": ["uuid-cze", "uuid-bih", "uuid-civ", "uuid-jpn"]
}
```

- `groups[X]` minden csoportra **mindig 4 elemű tömb**, ahol az index a helyezés (0=1.hely); hiányzó pozíció `null`.
- `best3rds` 0–4 elemű `teamId` tömb.
- A backend a részleges (még nem teljes) tippet is **érvényesnek** tekinti tárolásra — csak a deadline pillanatában (vagy submit final esetén) számít teljességnek a kiértékelés.

### `correct_answer` (admin reveal után)

Ugyanaz a séma, mint a user answer, de minden mező kitöltve.

## Elfogadási kritériumok

### Backend – Schema

1. **Új `inputType` enum érték**
   - `'all_groups_standing'` — 12 csoport sorrendje + 4 best 3rd egy rekordban.
   - Drizzle migration: `ALTER TYPE special_prediction_input_type ADD VALUE 'all_groups_standing';`
   - Idempotens (`IF NOT EXISTS`).

2. **`options` JSONB séma**
   - `groups: string[]` — csoport-kódok listája (default: `['A'..'L']`).
   - `teamsPerGroup: number` — default `4`.
   - `best3rdPicks: number` — default `4`.

### Backend – Validáció

3. **PATCH/POST `/api/special-predictions/:typeId`** elfogadja a részleges answer JSON-t és validálja:
   1. `answer.groups` kulcsai mind az `options.groups` listájában vannak.
   2. Minden csoport tömbje pontosan 4 elemű (null megengedett részleges állapotban).
   3. Egy csoporton belül nincs ismétlődő (nem-null) teamId.
   4. A teamId-k a torna mezőnyében szerepelnek, és **pontosan annak a csoportnak a tagjai** (pl. MEX csak az A csoportba kerülhet).
   5. `best3rds` 0–4 elemű, ismétlődés nélküli, és minden eleme **valamelyik csoport `groups[X][2]` (3.hely) értéke**.
   - Hiba esetén 400 + a frontend rollback-el az utolsó sikeres állapotra.

4. **Idempotencia:** ugyanaz az answer többször küldve ugyanazt az állapotot eredményezi (upsert).

### Backend – Seed

5. **Idempotens seed** egy globális `all_groups_standing` típushoz:
   - `is_global: true`, `is_active: true`.
   - `deadline`: a torna első csoportmeccsének kickoffja.
   - `options.groups` a `groups` táblából deriválva (aktív torna csoport-kódjai szerint).
   - `ON CONFLICT DO NOTHING` (mint az US-935 mintában).

### Backend – Submission status

6. A PATCH/POST response a **teljes mentett state**-et adja vissza, plusz egy:
   ```json
   { "saved": true, "completion": { "groupsDone": 8, "best3rdsDone": 0, "totalDone": 8, "totalSteps": 13 } }
   ```
   mező — a frontend a sticky bar-on egységesen jeleníti meg.

### Frontend – Tippelési felület (mobile-first)

7. **`GroupStandingsPredictionView.vue`** — tab-szintű nézet:
   - Bevezető copy + deadline + auto-save infó.
   - 12 csoport-kártya accordion (alapból az első kitöltetlen kinyitva, többi collapsed).
   - Best 3rd szekció a 12 csoport után, **lockolt** amíg nem mind a 12 csoport `4/4` állapotú.
   - Sticky bottom bar: `X / 13 kész · ✓ Mentve` + 13-elemű progress dot-sor.

8. **`GroupStandingCard.vue`** — egy csoport accordion kártya:
   - Header: csoport neve + "X / 4" badge VAGY "✓ Kész" badge ha 4/4.
   - Collapsed nézet: chip-szerű csapat-helyezés lista (`🇲🇽 MEX 1 · 🇰🇷 KOR 2 · …`).
   - Expanded nézet: 4 sor (1.–4. helyezés), minden sor egy **`PositionTeamDropdown`** (v2 design — pozíció-first, nem csapat-first).
   - Kártya alján "↺ Csoport ürítése" gomb (törli mind a 4 pozíciót).

9. **`PositionTeamDropdown.vue`** (v2 — **a v1 `TeamPositionDropdown` helyett**):
   - Props: `position: 1 | 2 | 3 | 4`, `groupCode: string`, `groupTeams: readonly Team[]`, `currentAssignments: GroupState`.
   - Az opciók: **csak a még szabad csapatok** a `groupTeams` 4 csapatából (smart filter — már kiosztott csapatok nem jelennek meg).
   - Ha a pozícióban már van csapat, a saját csapata is választható lehetőségként látszik (a kiválasztott állapotban) — más szabad csapatra váltáskor a régi felszabadul.
   - **Nincs auto-swap és nincs inline "Pozíció ürítése" opció.** Pozíció-csere csak szabad csapaton keresztül lehetséges, vagy a "↺ Csoport ürítése" gombbal.

10. **`Best3rdPicker.vue`** — chip-szerű multi-toggle:
    - Lockolt overlay amíg !12/12 csoport (`is12of12 === false`).
    - Kinyíláskor a 12 csoport `groups[X][2]` (3.hely) értékeiből összegyűjtött csapatlista.
    - Maximum **4 chip aktiválható**; az 5. tap blokkolt + toast: "Csak 4 csapatot választhatsz."
    - **Dinamikus szinkronizáció:** ha valamelyik csoport 3.helyezettje megváltozik, a `best3rds` szűrésre kerül (csak azok maradnak, akik még az új available listában vannak), toast: "A csoport módosult — Last 16 választás frissítve."

11. **Csoportkártyák szerkesztési sorrend-mentessége (v2 hangsúly):**
    - Minden csoportkártya **egyenrangú** és független — a felhasználó tetszőleges sorrendben tölti ki.
    - Az accordion-fejléceken **nincs** "kövesd ezt a sorrendet" jelzés (számozás A→L csak abc-rendben, nem progresszióként).
    - A bevezető copy: "Bármelyik csoporttal kezdheted."

### Frontend – Auto-save

12. **`useGroupStandingsStore` (Pinia)**:
    - `state`: `groups: Record<GroupCode, (TeamId|null)[4]>`, `best3rds: TeamId[]`, `saveStatus: 'idle' | 'saving' | 'saved' | 'error'`.
    - `getters`: `completedGroupsCount`, `is12of12`, `isFullySubmitted` (12 csoport teljes + 4 best 3rd).
    - `actions`: `setPosition(group, team, position)`, `clearGroup(group)`, `toggleBest3rd(team)`, `flushSave()` (debounced).

13. **Auto-save flow:**
    - **Optimistic UI:** a változás azonnal látszik.
    - **Debounce:** 400 ms inaktivitás után indul a hívás.
    - **PATCH/POST endpoint** küldi a teljes aktuális answer JSON-t (~1–2 KB).
    - Sticky progress sávban státusz indikátor: `↻ Mentés…` / `✓ Mentve` / `⚠ Nem sikerült menteni — újrapróbálom` / `⛔ Mentés sikertelen`.

14. **Race-condition kezelés:**
    - A frontend monoton verziószámot (lokális counter, request-id) küld minden mentésen.
    - Last-write-wins; reload az authoritatív szerver-állapotra állít.
    - 4xx (validáció): nincs retry; UI hibaüzenet, mező rollback az utolsó sikeres állapotra.
    - 5xx / network error: exponenciális backoff retry (3×: 1s, 3s, 9s).

### Frontend – Read-only nézet (deadline után)

15. **Read-only nézet:**
    - Minden csoport collapsed, chip-szerű 4 csapat-helyezés.
    - Best 3rd: a 4 választott chip kiemelve.
    - Admin kiértékelés után (`correctAnswer` set):
      - helyes pozíció: zöld háttér + ✓
      - helytelen pozíció: piros háttér + a tényleges helyezett csapat halvány second sorban
      - best 3rd helyes: zöld; helytelen: piros
    - A pontszám-megjelenítés a pontozási story hatóköre; addig csak a vizuális visszajelzés látszik.

### Admin – Eredmény beállítás

16. **Admin reveal UI** a `correctAnswer` beállításához:
    - 12 csoport × 4 dropdown a tényleges sorrendhez.
    - Best 3rd kiválasztás (4 csapat).
    - Mentés után automatikus pontszámítás (külön story).

## Technikai megjegyzések

### Smart filter logika (PositionTeamDropdown — v2)

```ts
function availableTeamsForPosition(
  position: 1 | 2 | 3 | 4,
  groupTeams: readonly Team[],
  currentAssignments: (TeamId | null)[4]
): Team[] {
  const usedElsewhere = new Set<TeamId>()
  currentAssignments.forEach((teamId, idx) => {
    if (teamId && idx + 1 !== position) usedElsewhere.add(teamId)
  })
  return groupTeams.filter(t => !usedElsewhere.has(t.id))
}
```

### Best 3rd dinamikus szinkronizáció (computed)

```ts
const best3rdsAvailable = computed<TeamId[]>(() =>
  Object.values(state.groups)
    .map(g => g[2])
    .filter((t): t is TeamId => t !== null)
)

watch(best3rdsAvailable, (newList) => {
  const newSet = new Set(newList)
  const filtered = state.best3rds.filter(t => newSet.has(t))
  if (filtered.length !== state.best3rds.length) {
    state.best3rds = filtered
    showToast('A csoport módosult — Last 16 választás frissítve.')
    scheduleSave()
  }
})
```

### Schema migration vázlat

```sql
-- Új enum érték (csak akkor kell, ha az US-937 még nem vezette be)
ALTER TYPE special_prediction_input_type
  ADD VALUE IF NOT EXISTS 'all_groups_standing';
```

A `points_per_correct` és `max_selections` mezők **nem kötelezően** szükségesek ehhez a típushoz (a pontozást másik story dönti el); ha az US-937 már bevezette, az `options` JSONB-ben tárolt `groups` / `teamsPerGroup` / `best3rdPicks` átveszi ezt a szerepet.

## Edge case-ek

| Helyzet | Elvárt viselkedés |
|---------|-------------------|
| User egy csoportban csak 2 helyezést tölt ki | Tárolódik (auto-save), progress 2/4; submit-final még nem lehetséges. |
| User a B csoportba MEX-et tippeli (de MEX A-csoportos) | Soha nem fordulhat elő — a B csoport dropdownja csak a B csoport 4 csapatát listázza. (Csak rosszindulatú API hívásra: backend 400 + frontend rollback.) |
| User az 1. helyre szeretne tenni egy olyan csapatot, aki már 2. | A 2.-ben lévő csapat **nem jelenik meg** az 1. dropdown opciói között (smart filter). Ha cserélni akar, először a 2. dropdownjából kell másik csapatot választania, vagy a "Csoport ürítése" gombbal újrakezdeni. |
| User egy már kitöltött pozíció dropdownját megnyitja | A saját csapata kiválasztott állapotban látszik; mellette a többi szabad csapat választható (váltáshoz). Külön "Pozíció ürítése" inline opció **nincs**. |
| User mind a 4 pozíciót kitöltötte, és egyet cserélni szeretne, de nincs szabad csapat | Csak a "↺ Csoport ürítése" gomb tudja felszabadítani a pozíciókat — a teljes csoport újrakezdődik. |
| User a 3.helyezettet megváltoztatja, miközben best3rds-ben szerepel | A best3rds automatikusan szűrődik, toast jelez. |
| Network drop közben | Optimistic UI marad; reconnect után a memóriában lévő utolsó állapot egy mentésben megy ki. |
| Két tab nyitva, mindkettő szerkesztett | Last-write-wins; reload az authoritatív szerver-állapotra állít. |
| Deadline mentés közben telik le | Backend 400 Deadline elapsed; UI lockolt read-only nézetre vált. |
| Felhasználó újratölti az oldalt félig kész tippel | A szerverről jövő answer az autoritatív; a felület helyreállítja a részleges állapotot, kitöltött csoportok collapsed, az első kitöltetlen expanded. |

## Tesztelés

### Backend unit tesztek (`vitest`)

- Validáció: érvényes/érvénytelen részleges és teljes answer kombinációk; cross-group team-id kontroll.
- Seed idempotencia.
- PATCH endpoint: részleges answer felülírja a tárolt állapotot.

### Frontend unit tesztek (`vitest`)

- `PositionTeamDropdown` smart filter:
  - Először minden csapat szabadon választható.
  - Egy csapat 1.-re tétele után a másik 3 dropdown opciói között az adott csapat **nem szerepel**.
  - Egy már kitöltött pozíció dropdownja a saját csapatát kiválasztott állapotban mutatja, és a többi szabad csapat választható váltáshoz.
- `Best3rdPicker`:
  - Maximum 4 chip aktiválható; az 5. tap blokkolt + toast.
  - Csoport 3.helyezett változása szűri a best3rds-t.
- Auto-save:
  - Debounce: 5 gyors változtatás → 1 PATCH.
  - Sikertelen mentés visszaállít az utolsó sikeres állapotra.
  - Reconnect után pending változtatások mennek ki.

### E2E (`playwright`)

- Boldog út v2: csoportok kitöltése **nem-A→L sorrendben** (pl. F → C → A → ...) → 12/12 után Best 3rd unlockol → 4 chip kipipálás → reload → mentett állapot helyreáll.
- Smart filter regressziója: B csoport 1. = Canada beállítása után a 2./3./4. dropdown opciói között Canada nincs.
- "Csoport ürítése" workflow: 4/4 kitöltött csoport újrakezdéséhez a kártya alján lévő gomb használata.
- Lock megmarad: a Best 3rd szekció `disabled`/lakat marad, amíg !12/12.
- Validációs hiba: B csoportba MEX bekevergetése (rosszindulatú API hívással) → backend 400 → UI rollback.

## Kizárások

- **Pontozási logika** — külön story.
- **Drag-and-drop sorrendezés** — az MVP-ben dropdown smart filter; drag későbbi UX iteráció, ha igény van rá.
- **Auto-swap dropdown viselkedés** — explicit kizárva v2-ben (smart filter, `nincs auto-swap`).
- **Localstorage offline cache** — out of scope; csak in-memory + szerver az SSOT.
- **Eredmény diff-vizualizáció** (Sankey-szerű ábrák) — out of scope.
- **Aggregált csoportbontás** ("hányan tippelték MEX-et 1.helyre") — a US-934 mintát külön story követi.
- **Per-csoport granuláris `group_standing` típus** — az US-937 lefedi; ez a story a **kombinált, 12 csoport + best 3rd** változat.

## Nyitott kérdések

1. **Új `inputType` neve.** `all_groups_standing` vs. `tournament_standings`. Javaslat: `all_groups_standing` (pontosabb).
2. **Auto-save endpoint.** Új PATCH vs. meglévő POST upsert. Javaslat: meglévő POST upsert kiterjesztése (kisebb blast radius).
3. **4/4-es csoport pozíció-cseréjének UX-e.** A jelenlegi modellben mind a 4 pozíció kitöltése után egy konkrét pozíció módosításához a teljes csoportot újra kell kezdeni. Elfogadható-e ez UAT-ban, vagy szükséges egy "csere" alternatíva (pl. egy konkrét pozíció ürítését lehetővé tevő külön gomb a sor mellett)? **Javaslat:** MVP-ben az aktuális modell (csak teljes csoport-ürítés); ha a UAT zavarónak találja, hozzáadunk egy diszkrét "× ürítés" ikont a kitöltött pozíció-sorok jobb oldalán.
4. **Pontozási story sürgőssége.** A típus tárolható és kiértékelhető read-only, pontszám nélkül. A pontozási story bekerülhet ugyanabba a sprintbe vagy később. Javaslat: külön story, hogy ez gyorsan releaselhető legyen.
