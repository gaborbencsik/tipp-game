# OPS-003: Auto-generált DB schema referencia agentnek és fejlesztőnek

## Probléma

A DB schema "igazság forrása" jelenleg `packages/backend/src/db/schema/index.ts` (584 sor, ~16k token Drizzle DSL-ben). Két baj van vele:

1. **Az agent költsége nagy.** Bármely query- vagy endpoint-tervezéshez tipikusan az egész schema fájlt be kell olvasni — ez egy session-ben drága, és a legtöbb feladathoz a struktúra (táblák, oszlopok, FK-k, indexek, soft-delete) elég lenne, a Drizzle szintaxis nélkül.
2. **A meglévő kézi doc (`plans/02-database-schema.md`) outdated.** A 26 valós táblából csak 16-ot említ (62%), és néhány meglévő mezőnév is hibás (pl. `homeScore/awayScore` helyett valójában `home_goals/away_goals`, `group_members.role` helyett `is_admin`). Hiányzó táblák: `player_stats`, `leagues`, `live_match_states`, `match_market_data`, `match_insights`, `insight_reveals`, `group_leagues`, `user_league_favorites`, `sync_state`, `llm_usage_log`. Hiányzó enumok: `match_stage`. Tehát a kézi doc egyszerre **félrevezető** és **karbantarthatatlan** — minden schema változtatás után emlékezni kéne, hogy frissítsük, és nem így történik.

**Mit szeretnénk:** egy **mindig pontos, tömör (~1.3k token), agent-baráti markdown referencia**, ami **automatikusan generálódik** a Drizzle schema-ból, így soha nem divergálhat.

---

## Story

Mint **agent / fejlesztő**, szeretnék **egy mindig friss, tömör DB schema referenciát markdown-ban**, hogy **a schema-érintő feladatokhoz ne kelljen a teljes 584 soros Drizzle TS forrást beolvasnom, és biztos lehessek benne, hogy a doc nem hazudik**.

---

## Elfogadási kritériumok

### A — Generátor script

1. **Új fájl:** `packages/backend/scripts/gen-schema-doc.ts`
   - [ ] Importálja `db/schema/index.ts`-t
   - [ ] Drizzle `getTableConfig()` API-val végigmegy minden `pgTable` exporton
   - [ ] Minden táblához összegyűjti: oszlopnevek, típusok, `notNull`, default, FK reference, soft-delete flag (`deleted_at` jelenléte), indexek (uniqueIndex és index-ek nevei + oszlopai)
   - [ ] Enumokat is kigyűjti (`pgEnum` exportok)
   - [ ] Megírja `docs/db-schema.md`-t

2. **Output formátum** (kötött, ne térjen el):
   ```markdown
   # DB Schema (auto-generated — do not edit)

   > Source: packages/backend/src/db/schema/index.ts
   > Generator: packages/backend/scripts/gen-schema-doc.ts
   > Run: `npm -w backend run db:doc`

   ## Tables

   | Table | Cols | FKs | Soft-del | Indexes |
   |---|---|---|---|---|
   | users | id, supabase_id, email, display_name, role, ... | – | ✓ | email |
   | predictions | id, user_id, match_id, home_goals, away_goals, ... | users, matches | – | (user_id, match_id) UNIQUE |
   | ... | | | | |

   ## Enums

   - `user_role`: user, admin
   - `match_status`: scheduled, live, finished, cancelled
   - `match_stage`: group, round_of_16, ...
   ```

3. **Méret-cél:** a generált `docs/db-schema.md` **maradjon ~1.5k token alatt** a teljes schema-ra. Ha tovább nő (jövőbeli táblák), a tábla-detail mehet összevontabb formátumra (pl. csak a top-5 oszlop + "..."). A kötelező mező mindig: PK, FK-k, soft-delete oszlop, indexek által érintett oszlopok.

4. **Idempotens:** kétszer egymás után futtatva azonos byte-os outputot ad (sortolt táblanevek, sortolt oszlopok az index-listán belül).

### B — npm script

1. [ ] `packages/backend/package.json` — `"db:doc": "tsx scripts/gen-schema-doc.ts"`
2. [ ] Root `package.json` — `"db:doc": "npm -w backend run db:doc"` (kényelmi alias)
3. [ ] README vagy CLAUDE.md megemlíti: "schema módosítás után futtasd: `npm run db:doc`"

### C — `plans/02-database-schema.md` leváltása

1. [ ] **Tartalom kettéválasztva:**
   - **Auto-generált rész** → `docs/db-schema.md` (táblák, oszlopok, FK-k, indexek, enumok)
   - **Konvenciók és döntések** → `plans/02-database-schema.md` átírva: csak a "miért UUID PK", "miért TIMESTAMPTZ", "soft-delete politika", "audit_logs jelentősége", scoring-konfig politika marad — a tábla-lista kikerül.
2. [ ] A `plans/02-database-schema.md` tetején: `> Tábla-szintű referencia: docs/db-schema.md`

### D — CLAUDE.md frissítés

1. [ ] **Tervdokumentumok táblázat** kiegészítve:
   ```
   | docs/db-schema.md | Auto-generált DB schema referencia (agent-baráti) |
   ```
2. [ ] **Kontextus optimalizálás** szekció kiegészítve egy mondattal:
   "DB schema kérdéshez először `docs/db-schema.md`-t olvasd, ne a Drizzle TS forrást."
3. [ ] **Mit NE csinálj** szekció kiegészítve:
   "Ne szerkeszd kézzel `docs/db-schema.md`-t — auto-generált."

### E — Schema változás → doc szinkron biztosíték

**Két opció — válassz egyet:**

**E1 (preferált): CI check**
- [ ] GitHub Action workflow (vagy meglévő CI workflow-ba step): `npm run db:doc && git diff --exit-code docs/db-schema.md`
- [ ] Ha schema változott és a doc nincs regenerálva → CI bukik világos hibaüzenettel: "Run `npm run db:doc` and commit the result."

**E2 (alternatíva, ha nincs CI workflow erre): pre-commit hook**
- [ ] Husky (vagy egyszerű git hook) — ha `packages/backend/src/db/schema/**` változott a staged fájlok között, futtassa `npm run db:doc`-ot, és adja hozzá a generált fájlt a commit-hoz.
- [ ] Ha husky még nincs telepítve, **ne ezt válaszd** — túl sok overhead egyetlen story-ért. Inkább E1.

> **Megjegyzés:** Ha sem CI sem hook nem reális most, a doc kézi futtatásra marad bízva — ekkor a CLAUDE.md figyelmeztetés legalább emlékezteti az agent-et és a fejlesztőt. Ez elfogadható átmenet, de E1 a cél.

### F — Tesztek

1. [ ] **Smoke teszt:** `gen-schema-doc.test.ts` — futtatja a generátort egy tmp fájlba, és ellenőrzi:
   - A tabla-lista tartalmazza mind a 26 (jelenlegi) táblát
   - `predictions` sorban szerepel `users, matches` mint FK
   - `users` sorban szerepel ✓ a soft-delete oszlopban
   - Az enum lista tartalmazza `user_role`-t és `match_status`-t
2. [ ] Token-budget asszerció: a generált fájl `< 8000` karakter (durva proxy ~1.5k tokenre)

## Technikai megjegyzések

### Drizzle introspect API

```typescript
import { getTableConfig } from 'drizzle-orm/pg-core'
import * as schema from '../src/db/schema'

for (const [name, exported] of Object.entries(schema)) {
  if (!isPgTable(exported)) continue
  const cfg = getTableConfig(exported)
  // cfg.columns, cfg.indexes, cfg.foreignKeys, cfg.name
}
```

`getTableConfig` minden szükséges metaadatot kiad — nincs szükség külön DB introspect-re (ami egyébként csak Supabase-en futna, lokálisan nem mindig elérhető).

### Mit NE tegyünk a generált doc-ba

- **Mermaid ER diagram** — token-pazarló (~600 token plusz), és az agent-nek a táblázat többet mond. Ha emberi szemnek kell, az külön fájl és külön story.
- **Per-tábla detail fájlok** (`docs/schema/<table>.md`) — most overkill. Csak akkor vezessük be, ha a szimpla táblázat nem elég.
- **DDL-szintű részletek** (constraint név, default expression nyersen) — a Drizzle TS-ben elérhető, oda menjen, aki migrációt ír.

### A `plans/02-database-schema.md` átalakítás vázlata

Ami **megmarad** (decisions, "miért"):
- PK politika (UUID)
- Soft-delete politika és érintett táblák listája — *de tábla-szintű részlet nélkül*
- TIMESTAMPTZ konvenció
- Scoring-konfig: DB-tárolt, csoportonkénti override
- Audit log politika: melyik művelet mehet bele

Ami **kikerül** (tábla-lista):
- A jelenlegi `## Táblák` szakasz teljes egészében — ezt `docs/db-schema.md` adja friss formában.

### Érintett fájlok

**Új fájlok:**
- `packages/backend/scripts/gen-schema-doc.ts`
- `packages/backend/scripts/gen-schema-doc.test.ts`
- `docs/db-schema.md` (auto-generált, első futtatás után commit)
- `.github/workflows/db-schema-check.yml` (ha E1) — vagy meglévő CI-be step

**Módosított fájlok:**
- `packages/backend/package.json` — `db:doc` script
- `package.json` (root) — `db:doc` alias
- `CLAUDE.md` — Tervdokumentumok tábla, Kontextus optimalizálás, Mit NE csinálj
- `plans/02-database-schema.md` — táblalista kivéve, csak konvenciók/döntések maradnak

## Kizárások (későbbi story-k)

- **Mermaid ER diagram generálás** humán review-ra — külön story, ha valós igény jön rá.
- **Per-tábla deep-dive markdown fájlok** (`docs/schema/<table>.md`) — csak ha a táblázatos referencia bizonyíthatóan kevés.
- **DB introspekció Supabase-ből** (drift-detection a Drizzle schema vs. valós DB között) — külön ops story; nem ennek a scope-ja.
- **Migráció-szintű changelog generátor** — szintén külön téma.
