# OPS-008: SSE-alapú live update – polling kiváltása élő meccs adatokra

## Probléma

[[OPS-004]] gate-elte a polling-ot live meccs jelenlétére + 60s intervallumra, de **élő meccs alatt továbbra is** percenként hívja minden online user a backendet a virtualPoints és a meccs-státusz endpointokon. 100 user × 60 hívás/h × 30 KB = ~180 MB / óra élő meccs alatt **csak ezen a két endpointon**.

Egy mérkőzés gólja kb. **5-10 percenként** van. Az 50-60 polling hívás ezért 50-58 felesleges (304 vagy ugyanaz). [[OPS-006]] ETag-gel ez 200 byte-ra zsugorodik, de **még mindig** 100 × 60 hívás / óra a backendre.

Az SSE (Server-Sent Events) **push** model: a backend csak akkor küld eseményt a kliensnek, **amikor tényleg történik valami** (gól, státusz váltás). Egy kliens egy nyitott connection-t tart, és kapja a delta-t.

A backend-ben **már van** SSE infrastruktúra (`/api/events`, `events.routes.ts`). Ki kell terjeszteni live meccs eseményekre.

**Becsült megtakarítás: 40-60%** a teljes egress-en élő meccs napokon.

---

## Story

Mint **felhasználó**, szeretném, hogy **az élő meccs során az eredmény és a virtuális pontszám automatikusan frissüljön push-on keresztül**, hogy **ne kelljen a kliensemnek percenként lekérdeznie ugyanazt, és a UI azonnal reagáljon a gólváltozásra**.

## Elfogadási kritériumok

### A — Backend SSE bővítés

1. [ ] A `/api/events` SSE endpoint új csatornát ad ki: `match.update`
   - Payload: `{ matchId: string, status: MatchStatus, homeScore: number | null, awayScore: number | null, updatedAt: string }`
2. [ ] Új csatorna: `virtualPoints.update`
   - Payload: `{ groupId: string, userId: string, points: number, updatedAt: string }`
   - **Csak** az érintett user-nek küldve (per-user SSE szűrés)
3. [ ] Eseményküldés trigger pont:
   - Match status / score változás (admin manual update VAGY football API sync) → `match.update` broadcast minden subscriber-nek
   - Virtual points újraszámolás (gólváltozás után automatikusan) → `virtualPoints.update` az érintett users-nek

### B — Backend eseménybusz

1. [ ] Egy in-memory event bus (`EventEmitter` Node-ból, nincs külső dep)
2. [ ] A `matches.service.ts` és a virtualPoints recompute kód `eventBus.emit('match.update', payload)`-t hív, az SSE endpoint subscribálja
3. [ ] Multi-instance edge case: a Render-en jelenleg 1 instance van, ezért nem szükséges Redis pub/sub. **Ha skálázunk:** komment a kódban: `// MULTI-INSTANCE TODO: switch to Redis pub/sub`

### C — Frontend SSE consumer

1. [ ] Új composable: `packages/frontend/src/composables/useMatchEvents.ts`
   - Open EventSource a `/api/events`-re a JWT-vel
   - Listen `match.update` és `virtualPoints.update` event-ekre
   - Match update → `useMatchesStore().applyMatchUpdate(payload)`
   - Virtual points update → a GroupDetailView frissíti a saját state-jét
2. [ ] Reconnect logika: ha az SSE leszakad, exponenciális backoff (1s, 2s, 4s, max 30s)

### D — Polling kivonása

1. [ ] Ha az SSE connection **aktív és healthy**, a [[OPS-004]] virtualPoints polling **leáll** (a `useMatchEvents` flag-eli)
2. [ ] Ha az SSE leszakad / nem támogatott (régi mobile böngésző): fallback a polling-ra (graceful degradation)

### E — Tesztek

1. [ ] **Unit teszt** az `eventBus`-ra (emit + on + off)
2. [ ] **Integration teszt** a `/api/events` route-ra:
   - Subscribe → `eventBus.emit('match.update')` → SSE kliens megkapja
   - JWT alapján filter (más usernek nem megy ki a privát virtualPoints update)
3. [ ] **Frontend unit teszt** a `useMatchEvents`-re mock `EventSource`-szal:
   - Reconnect backoff
   - Cleanup `onUnmounted`-ben
   - Polling fallback ha az SSE 3 reconnect után sem áll fel

### F — Megfigyelhetőség

1. [ ] Backend log: SSE kapcsolatok száma, broadcast count, error rate
2. [ ] Admin dashboard mini-tile: "Aktív SSE klienskezdetek: N"

## Technikai megjegyzések

### Architektúra

```
[match update]
    ↓
[matches.service.ts] → [eventBus.emit('match.update', payload)]
    ↓
[/api/events SSE handler subscribed]
    ↓ (per-user filter)
[res.write(`event: match.update\ndata: ${JSON}\n\n`)]
    ↓
[Frontend EventSource onmessage]
    ↓
[useMatchesStore.applyMatchUpdate(payload)]
```

### Frontend SSE token

A `EventSource` API **nem támogatja** a custom headert (Authorization: Bearer). Két opció:
1. Cookie-alapú auth (komplex, biztonsági review kell)
2. JWT token query paraméterben: `/api/events?token=<jwt>` — **ez a default**, de figyelni hogy ne logoljuk a token-t URL-ekben (Render access log)
3. Vagy `EventSourcePolyfill` (npm), ami támogat headert — preferred, kis dep

**Decision:** `EventSourcePolyfill` használata, ha még nincs a deps-ben (~5 KB, minified). Egyértelműbb biztonsági posztúra.

### Backend SSE keep-alive

1. [ ] A backend 30 mp-enként küld egy `: ping` comment frame-et, hogy a Render proxy / mobile network ne zárja le idle connection miatt
2. [ ] Heartbeat interval konstansként: `SSE_HEARTBEAT_INTERVAL_MS = 30_000`

### Error / fallback

- Ha a frontend 3-szor egymás után nem tud SSE-t nyitni (404, 401, network error), a [[OPS-004]] polling **automatikusan visszaáll**, log az error reporting-ban
- A user UI-jában nincs explicit "SSE connected" indicator (overkill MVP-re)

### Érintett fájlok

**Új fájlok:**
- `packages/backend/src/services/event-bus.service.ts` (in-memory EventEmitter wrapper)
- `packages/frontend/src/composables/useMatchEvents.ts`
- Tesztek a fenti fájlokhoz

**Módosított:**

## Mit NE csinálj

- Ne építs Redis pub/sub-ot (egy instance van, YAGNI)
- Ne dobd ki a polling-ot teljesen — fallback kell
- Ne küldj minden user-nek minden virtualPoints update-et (privacy + egress)
- Ne loggolj JWT-t a Render access log-ba (URL query param miatt résen lenni — akár az `EventSourcePolyfill` headerrel jobb)

## Várt egress-megtakarítás

| Időablak | Polling (jelenleg, [[OPS-004]] után) | SSE (új) | Megtakarítás |
|----------|--------------------------------------|----------|--------------|
| Élő meccs alatt | 60 hívás/h × 30 KB = 1.8 MB / user / h | 1 nyitott TCP + ~5 push event/h × 500 byte = 2.5 KB / user / h | **~99%** |
| Nincs élő meccs | 0 (gate) | 0 (gate) | — |

100 aktív user × 3 óra élő meccs / nap → **~540 MB / nap → ~7.5 KB / nap**. Hatalmas csökkenés, de a teljes egress-en (sok minden más is van) realisztikus becslés **40-60%**.
