# UX-042: Admin torna-tipp kiértékelés körönkénti csapatlistával

## Háttér

Az US-1311 bevezette az admin-vezérelt kiértékelési infrastruktúrát: szeletenként (pl. Last 16 kör, A csoport) futtatható kalkuláció, PATCH/POST API-k. A `bracket_progression` típusnál jelenleg a `BracketProgressionPicker` komponens a teljes 32 mérkőzés fa-szerű inputját követeli meg egyetlen megjelenítésben — túl bonyolult és nem felhasználó-barát az adminisztráció során, amikor körönként, fokozatosan érkez meg az információ.

Az új megoldás: körönkénti csapatlistás selector, amely az admin felületen (AdminTournamentEvaluationView) helyettesíti a régi picker-t. Az admin egyszerűen kiválassza az adott körben továbbjutó csapatokat egy csapatlistából (chipek, multi-select), körönként mentse, majd körönként kiértékelje.

## Hatókör — LE FED

### 1. Új admin frontend komponens

Új komponens: `packages/frontend/src/components/admin/BracketRoundTeamList.vue`

Megjelenítés:

- **6 kör lap** (tab vagy accordion): last_32, last_16, qf, sf, final, bronze.
- Minden körre: az admin egy csapatlistából választ ki az adott körben **továbbjutó csapatok számával** egyenlő darabszámú csapatot.
  - last_32: 32 csapat pool-ből, admin **16 csapatot** választ (a Last 32 továbbjutói).
  - last_16: az előző kör winnereit + az ott nem kiválasztott csapatokat, admin **8-at** választ.
  - qf: Last 16 winnerseiből, admin **4-et** választ.
  - sf: QF winnerseiből, admin **2-t** választ.
  - final: SF winnerseiből, admin **1-et** választ (a bajnok).
  - bronze: SF vesztesei közül (auto-derive), admin **0 kiválasztás szükséges** — read-only megjelenítés, a 2 SF vesztes automatikusan a bronze pool-ként mutatódik, az admin közülük választ 1-et.

- **Csapatlista UI**: chip-lista (mint a GroupResultPicker vagy más multi-select komponens a projekten belül). Chip: csapatnév + logó, kattintható toggle.

- **Limit enforce**: az admin csak az előírt számú csapatot választhatja ki. Ha túllépi, a gomb disable-ként marad. Toast ha túllépi: „{körönév}: max N csapat választható".

- **Cascade logika**: ha egy korábbi kör még nem mentett, a következő kör read-only / disabled, üzenet: „Előbb a {előző kör} körét értékelje ki."

- **Mentés gomb körönként**: „Mentés és kiértékelés: {körönév}". Klikk → frontend `mapRoundWinnersToBracketAnswer(round, selectedTeamIds, currentAnswer, template, correctGroupStandings)` hívása → új `BracketProgressionAnswer` szerializálása → `PATCH /api/admin/special-types/:typeId/correct-answer` body: `{ correctAnswer: serialized_answer_json }` → `POST /api/admin/special-types/:typeId/evaluate` body: `{ slice: 'last_16' }` → toast: „{körönév} kiértékelve: N felhasználó · X pont".

### 2. Mappolás logika helper

Új fájl: `packages/frontend/src/lib/bracketRoundMapping.ts`

```typescript
export function mapRoundWinnersToBracketAnswer(
  round: BracketRound,
  selectedTeamIds: string[],
  currentAnswer: BracketProgressionAnswer | null,
  template: BracketProgressionTemplate,
  correctGroupStandings: AllGroupsStandingAnswer
): BracketProgressionAnswer

// Logika:
// 1. Előző körök winners map-jét megtartja az `currentAnswer`-ből.
// 2. Az aktuális kör match-eit a `deriveBracket(correctGroupStandings, template, round)` alapján resolveol.
// 3. Minden match-hez hozzárendeli a selectedTeamIds közül azokat, amelyik a match résztvevői.
// 4. Ha egy csapat egyik match-be se tartozik → error throw: `BracketTeamNotInRoundError`.
// 5. Ha egy csapat több match-be tartozik (nem előfordul normál kör-struktúrában) → warning.
// 6. Válasz: kibővített `{ winners: { ...prev, matchId1: teamId1, ... } }`.
```

Szenáriók:

- (1) **Minden csapat findable**: az admin kiválasztott 8 csapat a last_16 kör 8 match-ében megtalálható → winners map sikeresen felépül.
- (2) **Csapat nem része egyetlen match-nek sem**: az admin egy csapatot választott, amely a last_16 levezetésű csapatai között nincs (pl. elírás, rossz gyűjteményes import) → error message a felhasználónak (toast vagy inline).
- (3) **Partial mapping ok**: a winners map-ben csak az adott kör csapatai szerepelnek, előző körök érintetlenek.

### 3. UI integráció — AdminTournamentEvaluationView

Meglévő nézet: `packages/frontend/src/views/AdminTournamentEvaluationView.vue`

Módosítás:

- A `bracket_progression` típusnál az **eddigi `BracketProgressionPicker` + "Mind kiértékelése" gomb blokkja teljes egészében helyettesítődik** az új `BracketRoundTeamList` komponenssel.
- Az `all_groups_standing` típusnál a régi UI marad (12 csoport, csoportonkénti gombsor).
- Egyéb típusok: marad az eredeti input + „Kiértékelés futtatása" gomb.

### 4. Validáció / Előfeltételek

- **Last 32 kör szerkeszthetősége**: csak akkor, ha az `all_groups_standing` típus `correctAnswer`-je rögzítve van (a slot kódok resolveolásához szükséges). Ha hiányzik: toast „Hiányzik a Csoport végeredmény helyes válasza."
- **Kör-kaszkád**: ha az N-edik kör még nem értékelt, az (N+1)-edik kör disabled. Üzenet: „Előbb a {N. kör} körét értékelje ki."
- **Bronze kör**: az SF vesztesei (az SF winnerek inverze) automatikusan a pool-ként kerülnek fel, read-only. Az admin közülük választ 1-et.

## Technikai megjegyzések

- **Adatstruktúra (`BracketProgressionAnswer`)**: `{ winners: { matchId: teamId } }`. A mappolás logika körönként felépíti az új kör match-eit a `deriveBracket()` alapján, majd mergelja az előzővel.
- **Csapatlistás pool**: a `correctGroupStandings` a Last 32 csapatait már definiálja (a derived bracket slotjai). A `mapRoundWinnersToBracketAnswer` ezen a derivált lista alapján reconcilál.
- **Bronze kör specifikum**: SF vesztesei = SF match winner inverze. A komponens ezt automatikusan kalkulálja az SF winners-ből.
- **API-k (már van az US-1311-ben)**:
  - PATCH `/api/admin/special-types/:typeId/correct-answer`
  - POST `/api/admin/special-types/:typeId/evaluate?slice=<round>`
- **Chip-feliratok csapatnév-feloldás**: a chipek nem UUID-t mutatnak, hanem
  emberi nevet. A komponens kétféleképp tölti fel az `id → név` mapet:
  1. opcionális `teamNameById` prop a szülő `AdminTournamentEvaluationView`
     `loadNameCache()`-éből (teams + players együtt),
  2. saját `GET /api/teams` fetch mountkor, ami safety net ha a parent
     cache késve érkezik vagy üres.
  Mindkét helyen a `VITE_DEV_AUTH_BYPASS=true` esetén a `'dev-bypass-token'`
  literált használjuk (az admin-store mintájára), különben Supabase
  session access token-t — különben helyi dev környezetben a fetch nem
  küldene tokent és a chipek UUID-ra esnének vissza.
