# BUG-003: Drizzle migráció csendben kihagyja az új migrációkat ha a journal `when` érték nem monoton

**Prioritás:** Should Have
**Függőség:** --
**Becsült méret:** S (~2-3 óra)
**Státusz:** Done (2026-05-21)

---

## Implementációs jegyzetek

**Megoldás:** A `drizzle-orm/migrator` és a régi `ensureMigrationJournal` bootstrap kicserélve egy saját idx-alapú runnerre, ami hash szerint dönt skip vs run-ról — a `when` mezőre egyáltalán nem támaszkodik.

- `scripts/migrate.ts` — teljes átírás:
  - `runMigrationsOnClient(client, loader)` — DI-vel tesztelhető, alap loader fájlrendszerből olvas
  - `_journal.json` entries idx ASC sorba rendezve, hash-elve (sha256 a teljes SQL fájl tartalmán — kompatibilis a drizzle eredeti hash-ével)
  - Skip ha hash már a `__drizzle_migrations`-ben van, futtatás BEGIN/COMMIT tranzakcióban + INSERT `(hash, when)`
  - Statement-breakpoint splittelés (kompatibilis a drizzle generálta SQL-ekkel)
  - Verify-step: ha bármelyik journal-hash hiányzik a táblából a futás végén → throw + exit 1
  - `console.log` minden ténylegesen lefuttatott migráció `tag`-jával

- `scripts/migrate.test.ts` — 14 új unit teszt:
  - `splitStatements` (3 eset), `hashSql` stabilitás (2)
  - Üres tábla → összes alkalmazva
  - Részleges → csak hiányzók
  - Idempotens (minden már alkalmazva → "Up to date")
  - **Non-monoton `when` → idx sorrendben fut** (a fő bug)
  - Shuffled journal array → idx-szerinti rendezés
  - BEGIN/COMMIT sorrend
  - Failure → ROLLBACK + nincs INSERT
  - Statement-breakpoint helyes szétválasztás
  - Verify-step throw ha hiányzik

**Élő DB drift reconcilation** (egyszeri, a deploy része):
- `0015_group_leagues` + `0016_group_leagues_rls_and_seed` SQL-ek lefutottak régen, de a tracking táblába sosem kerültek be (a régi drizzle silent-skip áldozatai). Hash-eik beillesztve.
- `0028_transfermarkt_market_value` és `0032_pre_wc_friendly_league_split` SQL fájl utólag módosult — a tracking táblában lévő hash-t a fájl jelenlegi tartalmából számoltra UPDATE-eltük (a séma maga helyes, csak a könyvelést hoztuk szinkronba).

**Verifikáció:** backend tsc zöld, 497 teszt zöld (41 fájl), `npm run db:migrate` host DB ellen → "Up to date — 37 migration(s) already applied." Idempotens.

**Kihagyott** (scope szűkítés):
- `_journal.json` `when` értékek visszamenőleges normalizálása — nem kell, az új runner idx szerint megy.
- Migration lock (parallel deploy) — Render single-instance, jelenleg nem probléma.
- Down-migrations — külön story ha kell.

## Eredeti specifikáció

## Kontextus

A 2026-05-21-i US-1401B deployment során a host DB-n a `0036_foamy_marvel_boy.sql` migráció **csendben kimaradt**: `npm run db:migrate` "Migrations complete." üzenettel sikeresen lefutott, de a `scoring_configs.frozen_at`, `match_results.points_calculated_at`, `sync_state.recalc_in_progress` és `sync_state.last_recalc_result` oszlopok nem jöttek létre. Manuális `psql -f` futtatással + `__drizzle_migrations` táblába hash insert-tel kellett pótolni.

A vizsgálat során kiderült, hogy a 0034, 0035, 0036 migrációk mind kimaradtak, és a `_journal.json`-ban a `when` mezők **nem monotonok**:

| idx | tag | when |
|----:|-----|-----:|
| 32 | `0032_pre_wc_friendly_league_split` | 1779393372284 |
| 33 | `0033_subscription_deadline_override` | 1779479772284 |
| 34 | `0034_typical_prodigy` | 1779286673981 *(régebbi mint 33)* |
| 35 | `0035_smart_pet_avengers` | 1779288662754 *(régebbi mint 33)* |
| 36 | `0036_foamy_marvel_boy` | 1779354708113 *(régebbi mint 33)* |

A drizzle-orm `migrate()` `created_at > MAX(applied.created_at)` összehasonlítást használ, ezért minden olyan journal bejegyzést **kihagy**, aminek a `when` értéke kisebb mint a már alkalmazott migrációk maximuma. Ez egy lokálisan generált migráció után CI-ben/prod-on órák/napok múlva csendben elromolhat.

## Gyökérok

**Két egymást erősítő hiba:**

1. **`scripts/migrate.ts` bootstrap túl agresszív** (sor 41-48): ha a `drizzle.__drizzle_migrations` tábla nem létezik, az ÖSSZES journal-bejegyzést applied-ként jelöli — még a soha-nem-futtatott újakat is. Ha valaki egy üres DB ellen futtatná egy frissen klónozott repón, az schema sosem épülne fel. (Ez a deploy során nem aktiválódott, mert a tábla már létezett — de potenciális landmine.)

2. **`drizzle-orm/migrator` `when`-alapú filterezés**: a `migrate()` függvény a journal `when` értékét hasonlítja a `__drizzle_migrations.created_at` MAX-jához, és csak az ennél `>` értékeket futtatja. Ha egy fejlesztő `git rebase` / `git pull --rebase` / merge után olyan migrációt commitol, aminek a `when`-je a már alkalmazott migrációk előtti, a migrator csendben átugorja.

A jelen incidens (0034/0035/0036 mind régebbi mint 0033) valószínűleg konkurens branch-eken párhuzamosan generált migrációkból áll össze, amik merge után rossz sorrendben lettek commit-olva.

## Elfogadási kritériumok

1. **Bootstrap viselkedés javítása**: ha a `__drizzle_migrations` tábla nem létezik, a script ne jelölje minden journal-bejegyzést applied-ként. Vagy üres tábla létrehozása (a normál migrator futtatja az összeset), vagy explicit "baseline" mechanizmus environment változó alapján (pl. `DRIZZLE_BASELINE_IDX=33`).

2. **`when`-független migrációs sorrend**: a script ne támaszkodjon a `_journal.json` `when` értékeire. Ehelyett:
   - Saját runner: minden journal-bejegyzést iterál `idx` szerint sorrendben, és lefuttatja azokat amiknek a `hash`-e nincs a `__drizzle_migrations` táblában.
   - VAGY: pre-migrate ellenőrzés ami `_journal.json` `when` értékeit monotonra rendezi (és ha nem az, hibával fail-el).

3. **Verify-step**: a migráció végén a script ellenőrzi, hogy minden journal-bejegyzés `hash`-e benne van a `__drizzle_migrations` táblában. Ha nem, **hangosan failel** (`process.exit(1)`) — nem "Migrations complete."-tel zárul.

4. **Logolás**: minden ténylegesen lefuttatott migráció `tag`-ja kiíródik, így a deploy log-ból egyértelmű mi futott le. Jelenleg csak "Running migrations..." + "Migrations complete." látszik.

5. **Idempotencia**: ha minden migráció már alkalmazva van, a script "Up to date — no migrations to run." üzenettel zárul (success).

6. **Backward compatible**: a meglévő `__drizzle_migrations` tábla tartalma érvényben marad, nincs törlés / újraépítés.

## Tesztelés

- **Unit teszt** a saját migration runner-re (ha az 2-es opciót választjuk):
  - Üres DB + üres tracking tábla → összes journal-bejegyzés alkalmazva
  - Részlegesen alkalmazott DB → csak a hiányzó hash-ek futnak le, sorrendben
  - Nem-monoton `when` értékek → akkor is sorrendben fut idx szerint
  - Verify-step megbukik ha egy migráció kimaradt

- **Integrációs teszt**: dockeres Postgres ellen lefuttatni egy mesterségesen rontott `_journal.json`-nal (0034 `when`-je < 0033) és ellenőrizni hogy a 0034 lefut.

- **Manual smoke**: friss DB ellen `npm run db:migrate` → minden tábla létrejön. Már migrated DB ellen → no-op.

## Technikai megjegyzések

### Javasolt megközelítés: saját idx-alapú runner

A `drizzle-orm/migrator` lecserélése egy minimális saját implementációra (~50 sor), ami:

```ts
// 1. Olvassa a _journal.json-t
// 2. Iterál entries[] sorrendben (a tömb sorrendje = idx sorrend)
// 3. Minden entry-re kiszámolja a sql fájl hash-ét
// 4. SELECT 1 FROM drizzle.__drizzle_migrations WHERE hash = $1
// 5. Ha nincs → futtatja az SQL-t tranzakcióban, INSERT-eli a hash-t
// 6. Ha van → skip
// 7. Végén: ellenőrzi hogy minden journal-hash megvan a táblában
```

Ez kiküszöböli a `when`-függőséget és átláthatóbb mint a drizzle migrator black-box-a.

### Alternatíva: pre-migrate journal-validáció

```ts
// Ellenőrzés _journal.json beolvasásakor:
let prevWhen = 0
for (const entry of journal.entries) {
  if (entry.when <= prevWhen) {
    throw new Error(
      `Non-monotonic journal: ${entry.tag} (when=${entry.when}) <= previous (when=${prevWhen}). ` +
      `Run a fix-script to renumber, or rebase your branch.`
    )
  }
  prevWhen = entry.when
}
```

Ez gyorsabb javítás, de a fejlesztőknek manuálisan kell rendezniük a `when` értékeket merge után. A saját runner robustabb.

### Bootstrap eltávolítása

A `ensureMigrationJournal()` jelenlegi logikája csak akkor lenne helyes, ha a kliensek között lett volna olyan, aki már produkcióban használta a sémát mielőtt drizzle-t bevezettünk volna. Erre nincs bizonyíték — a projekt drizzle-zal indult. A bootstrap teljes egészében törölhető. Ha valaha kell baseline, az legyen explicit env változó (`DRIZZLE_BASELINE_IDX`) vagy külön CLI command.
