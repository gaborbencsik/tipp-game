# US-1501: Polymarket meccs odds megjelenítése a meccs részletes nézetben

> **Státusz:** Draft  
> **Prioritás:** Should Have  
> **Effort:** M (6-8 óra)  
> **Függőség:** Nincs (a Polymarket API publikus, nincs auth)

---

## Összefoglaló

A MatchDetailView-ban megjelenítjük az adott meccs Polymarket moneyline odds-ait (win probability mindkét csapatra + döntetlen). Ez közvetlen kontextust ad a tippeléshez: a user látja, kit tart a piac esélyesebbnek az adott meccsen.

Emellett az odds adatokat DB-ben tároljuk, hogy később historikus odds-változás is megjeleníthető legyen.

### User Value

- **Közvetlen relevancia** — a meccs moneyline odds DIREKT a tippelési döntéshez kapcsolódik (ellentétben a tournament-winner odds-szal)

- **Korlátozás:** Nem minden meccshez lesz Polymarket piac (valószínűleg csoportkör nagymeccsek + egyenes ági meccsek)

### Pozícionálás

- **Megnevezés:** "Piaci esélyek" / "Market odds"
- **Alcím/tooltip:** "A Polymarket piaci becslése a meccs kimeneteléről"
- **Branding:** Diszkrét Polymarket hivatkozás (adatforrás megjelölés)

### Megjelenítés meccs státuszonként

| Státusz | Megjelenít? | Indoklás |
|---------|-------------|----------|
| scheduled | Igen | Fő use case — tippelés előtt hasznos kontextus |
| live | Igen | Élő meccs közben is érdekes (odds változik!) |
| finished | Igen | Retrospektív — "mit mondott a piac a meccs előtt" |

### MVP vs Later

**MVP (US-1501):**
- Backend: Polymarket Sports API proxy + DB storage
- Frontend: Odds bar a MatchDetailView-ban (home win / draw / away win)
- Matching: Polymarket event ↔ belső match összekapcsolás (csapatnevek + dátum)
- Fallback: ha nincs adat, nem jelenik meg (graceful degradation)

**Later (opcionális, külön story-k):**
- Historikus odds chart (hogyan változott az esély a meccs előtt)
- Odds megjelenítés a match card-on is (lista nézetben)
- Spread és totals megjelenítése
- Live odds frissítés (WebSocket/polling)
- "Odds momentum" — változás jelzése (felfelé/lefelé nyíl)

## System Architect elemzés

### Polymarket Sports API

**Endpoint:** `GET https://gamma-api.polymarket.com/v2/leagues/fifa-world-cup/events`

**Paraméterek:**
- `type=sport` — meccs-szintű piaci adatok (nem futures)
- `limit=50`, `offset=0` — pagination

**Response struktúra (releváns mezők):**
```typescript
interface PolymarketSportsEvent {
  id: string
  title: string           // pl. "France vs Germany"
  subtitle: string        // dátum/idő
  startDate: string       // ISO timestamp
  markets: PolymarketMarket[]
  // teams info nested
}

interface PolymarketMarket {
  sportsMarketType: 'SPORTS_MARKET_TYPE_MONEYLINE' | 'SPORTS_MARKET_TYPE_SPREAD' | 'SPORTS_MARKET_TYPE_TOTAL'
  marketSides: MarketSide[]
  bestBid: number
  bestAsk: number
  line?: number           // spread/total esetén
}

interface MarketSide {
  label: string           // "France", "Germany", "Draw"
  price: number           // 0-1 (probability)
}
```

### Match ↔ Polymarket event matching

**Stratégia:** A `matches` táblán egy `polymarket_slug` mező tárolja a Polymarket event slug-ot (pl. `fifwc-fra-sen-2026-06-16`). Ez egyértelműen összeköti a belső match-et a Polymarket event-tel.

**Miért nem generáljuk dinamikusan:**
- A Polymarket saját nem-standard team kódokat használ (pl. Curaçao = `kor`, South Korea = `kr`, Congo DR = `cdr`)
- A slug dátuma US Eastern timezone-ban van, ami esti UTC meccseknél eltér az UTC dátumtól
- Statikus mapping megbízhatóbb mint runtime heurisztika

**API hívás:** `GET https://gamma-api.polymarket.com/events/slug/{polymarket_slug}`

### DB séma

**Módosítás a `matches` táblán:**

```sql
ALTER TABLE matches ADD COLUMN polymarket_slug TEXT;
CREATE INDEX idx_matches_polymarket_slug ON matches(polymarket_slug);
```

**Új tábla: `match_market_data`** — time-series, minden 5 percben INSERT

```sql
CREATE TABLE match_market_data (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  match_id UUID NOT NULL REFERENCES matches(id),
  source TEXT NOT NULL DEFAULT 'polymarket',

  -- Core probabilities (0-1)
  home_win DOUBLE PRECISION NOT NULL,
  draw DOUBLE PRECISION,
  away_win DOUBLE PRECISION NOT NULL,

  -- Per-outcome price movement
  one_day_change_home DOUBLE PRECISION,
  one_day_change_draw DOUBLE PRECISION,
  one_day_change_away DOUBLE PRECISION,
  one_week_change_home DOUBLE PRECISION,
  one_week_change_draw DOUBLE PRECISION,
  one_week_change_away DOUBLE PRECISION,

  -- Market metrics
  market_volume DOUBLE PRECISION,
  market_liquidity DOUBLE PRECISION,
  best_bid_home DOUBLE PRECISION,
  best_ask_home DOUBLE PRECISION,
  last_trade_price_home DOUBLE PRECISION,

  -- Event-level (changes rarely but stored with each snapshot for simplicity)
  competitive DOUBLE PRECISION,
  context_description TEXT,

  -- Forward compatibility
  raw_payload JSONB,

  fetched_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_match_market_data_match_fetched ON match_market_data(match_id, fetched_at DESC);
```

**Miért egy tábla:**
- Minden adat ugyanabban a sync-ben jön, együtt változik
- Egyszerűbb query: legutolsó sor = teljes snapshot
- A `context_description` ismétlődése sor/meccs/5 perc kevés overhead ~72 meccsnél

**Miért DB:**
- Historikus adatok: hogyan változott az odds a meccs előtt
- Későbbi feature: odds chart, trend kijelzés
- Nem kell újra lekérni ha a Polymarket API elérhetetlen
- Audit trail: mikor milyen odds volt elérhető

### Cache + Sync stratégia

- **Scheduled job (cron):** 5 percenként lekéri az összes aktív FIFA WC meccs odds-ot (~288 req/nap, nincs dokumentált rate limit)
- **In-memory cache:** 5 perc TTL a response-on (ne terhelje a DB-t minden page load)
- **Stale-while-revalidate:** Ha a Polymarket API nem elérhető, az utolsó DB-beli rekordot szolgálja

### Endpoint design

```
GET /api/matches/:matchId/odds
Response: {
  homeTeam: { name: string, odds: number | null },
  draw: number | null,
  awayTeam: { name: string, odds: number | null },
  source: "polymarket",
  sourceUrl: string | null,
  updatedAt: string
}
```

### Ha nincs Polymarket piac a meccshez

- Kis csapatok meccseinél (pl. Szaúd-Arábia vs Új-Zéland) valószínűleg nincs moneyline piac
- Response: 404 vagy `{ homeTeam: { odds: null }, draw: null, awayTeam: { odds: null } }`
- Frontend: a szekció nem jelenik meg

### Architektúra összefoglaló

```
Sync Job (5 percenként)
  → GET Polymarket Sports API (fifa-world-cup events, type=sport)
  → Parse moneyline markets
  → Match belső match-hez (csapatnév + dátum)
  → INSERT match_market_data rekord

Frontend (MatchDetailView)
  → GET /api/matches/:matchId/odds
     → Backend: DB lookup (legutóbbi match_market_data rekord)
     → + in-memory cache (10 min)
     → Response
```

## Elfogadási kritériumok

### Funkcionális

1. **GIVEN** egy meccs részletei megtekintve, **WHEN** van Polymarket moneyline odds az adott meccshez, **THEN** megjelenik a "Piaci esélyek" szekció 3 sávval (home/draw/away) a tipp szekció alatt
2. **GIVEN** a Polymarket-en elérhető egy meccs piaca, **WHEN** a sync job fut, **THEN** az odds adatok bekerülnek a `match_market_data` táblába
3. **GIVEN** nincs Polymarket piac az adott meccshez, **WHEN** a user megnyitja a meccset, **THEN** az odds szekció nem jelenik meg
4. **GIVEN** a Polymarket API nem elérhető, **WHEN** a sync job fut, **THEN** az utolsó ismert odds marad a DB-ben (nincs adatvesztés)
5. **GIVEN** az odds megjelenik, **THEN** van egy info sor ami jelzi: ez Polymarket piaci becslés
6. **GIVEN** az odds megjelenik, **THEN** van egy "Polymarket" link az adatforrás oldalra
7. **GIVEN** egy 2-way market (nincs draw opció), **THEN** csak 2 sáv jelenik meg

### Nem-funkcionális

8. A sync job 5 percenként fut automatikusan
9. Az API response 5 percig cache-elődik in-memory
10. A frontend odds lekérés NEM blokkolja a fő meccs adat betöltését (párhuzamos fetch)
11. Mobil nézetben a layout nem törik el (320px-ig működik)

## Post-launch fixes (2026-06-04 / 06-05)

### Bug 1: home/away csapat odds felcserélődött, ha a vendég volt a favorit

**Tünet:** Polymarket-en pl. Qatar–Switzerland: Switzerland 80%, Qatar 6.7%; az UI-on viszont "Qatar 80%". Mindig a bal oldali (hazai) csapathoz került a magasabb érték, függetlenül attól, melyik csapaté valójában.

**Root cause:** `parseMoneylineOdds` pozíció szerint osztotta szét a piacokat — első nem-draw market = home, második = away. A Polymarket viszont a piacokat **valószínűség szerint csökkenő sorrendben** adja vissza, így vendég-favorit esetén az ő odds-a került a `homeWin` oszlopba. A bug az első verzióban (US-1501 commit, 2026-05-13) is jelen volt; csak hazai-favorit meccseknél működött helyesen véletlenül.

**Fix:** Pozíció helyett **csapatnév szerint** illesztjük a piacot a home/away oszlophoz. `parseMoneylineOdds(event, homeTeamName, awayTeamName)` — a `syncAllMatchOdds` JOIN-olja a `teams` táblát, és átadja a neveket. Új teszt: vendég-favorit eset.

### Bug 2: 23 meccs sync-je elhasalt csapatnév-eltérés miatt

**Tünet:** Az 1-es bug fixe után a 72-ből ~20 meccs `failed`-re esett. Strukturált log nem volt, ezért az ok rejtve maradt.

**Mitigáció előbb:** `createLogger('polymarket')` integráció — minden sync per-match `info`/`warn` logot ír (slug, csapatok, ok, market címek). Innen látszott:

| DB név | Polymarket `groupItemTitle` |
|---|---|
| USA | United States |
| South Korea | Korea Republic |
| Czech Republic | Czechia |
| Ivory Coast | Côte d'Ivoire |
| Cape Verde Islands | Cabo Verde |
| Congo DR | DR Congo |
| Bosnia & Herzegovina | Bosnia and Herzegovina |
| Iran | IR Iran |

**Root cause:** A DB-beli csapatnév és a Polymarket `groupItemTitle` szöveg eltér (rövidítések, diakritikák, "&" vs "and", szórend).

**Fix:** `normalizeTeamName()` — NFD-normalizáció + diakritika-strip + `&`→`and` + non-alnum-strip. `TEAM_NAME_ALIASES` tábla a felfedezett párokra. A market akkor illeszkedik, ha a normalizált forma egyezik. Új parametrikus teszt minden alias-párra.

### Bug 3: rate-limit / observability hiány

**Mitigáció:** `User-Agent` header a `fetch`-ben + 150 ms throttle a kérések között. Sync-eleje és -vége strukturált log (`sync started` / `sync finished` total/synced/failed).

### Tanulság

A `parseMoneylineOdds` eredeti tesztjei mind hazai-favorit eseteket fixáltak (France-Senegal 0.55/0.20), így a pozíció és a név alapú hozzárendelés egyformán átment volna. Ez a vakfolt rejtette el az 1-es bug-ot 3 hétig. A "Risks" tábla `Team name mismatch (matching)` sora helyesen azonosította a kockázatot, de a mitigáció (`Alias mapping`) csak post-launch valósult meg.
