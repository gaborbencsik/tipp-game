# US-948: Admin liga-oldal (frontend) – liga-lista státusz badge-dzsel + archiválás/visszaállítás

> Epic: E-Admin – Liga- és tornakezelés

## Háttér

A backend liga-CRUD **teljes egészében létezik** (`GET/POST/PUT/DELETE /api/admin/leagues`,
`leagues.service.ts`), és a US-940 hozzáadja az archiválás/visszaállítás backend rétegét
(`archiveLeague`/`restoreLeague`, `POST /api/admin/leagues/:id/archive`
és `/restore`, `getLeagues({ includeArchived })`, `GET /api/admin/leagues?includeArchived=true`).
A liga láthatósági **állapotát** a US-949 `status` ENUM-ja (`'active' | 'archived'`) hordozza,
ami felváltotta a korábbi `archived_at` timestamp-ot.

**Frontend-en viszont NINCS admin liga-oldal:** nincs `/admin/leagues` route
(`packages/frontend/src/router/index.ts`), nincs liga-nézet a `views/`-ban, és az
`AdminNav.vue` link-listájában sincs liga belépési pont. Az `api/index.ts`-ben van egy
`api.admin.leagues` wrapper (`list`/`create`/`update`/`delete`), de a US-940-hez szükséges
`archive`/`restore`/`includeArchived` hívások hiányoznak.

US-940 elfogadási kritériumai közt szerepel egy "admin liga-lista" badge-dzsel és
archive/restore gombokkal — **de az a felület nem létezik**. EZ a story hozza létre a
frontend admin liga-oldalt, amire a US-940 backend rétege ráépül.

## Story

Mint **admin**, szeretnék **egy dedikált admin liga-oldalt**, ahol látom az összes ligát
(aktívat és archiváltat egyaránt) státusz badge-dzsel, és **egy kattintással archiválhatok
vagy visszaállíthatok** egy ligát, hogy **a lezárt tornát elrejthessem az aktív felületekről,
majd szükség esetén újra aktiválhassam — adatvesztés nélkül**.

## Scope – MI tartozik EHHEZ a story-hoz (frontend)

- Új admin nézet: `AdminLeaguesView.vue`.
- Új route: `/admin/leagues` (`requiresAuth: true, requiresAdmin: true`).
- Új nav belépési pont az `AdminNav.vue` link-listájában.
- Új Pinia store: `admin-leagues.store.ts` (list + archive + restore, optimista/refetch,
  toast + error a meglévő minta szerint).
- API wrapper bővítés (`api/index.ts` `api.admin.leagues`): `archive`, `restore`, és a
  `list` `includeArchived` paraméter.
- `League.status` mező hozzáadása a **frontend** típushoz (`packages/frontend/src/types`),
  hogy a badge és a toggle a mezőre tudjon támaszkodni (a backend már visszaadja US-949 után).
- Magyar i18n kulcsok.
- `data-testid`-k + Vitest komponens teszt(ek).

## Scope – MI tartozik US-940 / US-949-hez (előfeltétel, NE duplikáld)

- **US-949:** DB `leagues.status` ENUM (`leagueStatusEnum`, `'active' | 'archived'`) + index +
  `archived_at` → `status` data-migráció. A **státusz mező és migráció US-949-é.**
- **US-940:** Service `getLeagues({ includeArchived })`, `archiveLeague`, `restoreLeague`
  (`status = 'archived'` / `status = 'active'`-re állít) + audit log. Az **archive/restore
  akció + audit US-940-é.**
- `matches.service.ts` archivált-kizáró szűrés + `MatchesFilters.includeArchivedLeagues`.
- Backend endpointok: `POST /api/admin/leagues/:id/archive`, `POST /api/admin/leagues/:id/restore`,
  `GET /api/admin/leagues?includeArchived=true`, `?includeArchivedLeagues` a `/api/matches`-en.
- Backend `League.status` a válasz-DTO-ban.
- Audience liga-választók archivált-kizárása (`matches.store.ts` liga-picker).

> **Előfeltétel:** a US-948 azt feltételezi, hogy a US-949 `status` mezője és a US-940 backendje
> (endpointok + `status` mező a válaszban) kész és deployolt. A frontend a kész szerződésre épül,
> backend munkát nem végez.

## Elfogadási kritériumok

### Route + navigáció

- [ ] Új route `/admin/leagues` néven (`admin-leagues`), lazy-loadolt `AdminLeaguesView.vue`,
      `meta: { requiresAuth: true, requiresAdmin: true }` — a meglévő admin route-ok mintájára.
- [ ] `AdminNav.vue` link-listájába új elem: `{ to: '/admin/leagues', label: 'Ligák' }`
      (a többi admin link stílusával konzisztensen).

### Lista + státusz

- [ ] Az oldal betöltéskor lekéri a ligákat **`includeArchived=true`**-val, hogy az archiváltak is
      megjelenjenek (`api.admin.leagues.list(token, { includeArchived: true })`).
- [ ] A lista táblázatban jelenik meg (a meglévő admin nézetek táblázat-mintája szerint):
      oszlopok legalább Név, Rövid név, Státusz, Műveletek.
- [ ] Minden sornál státusz badge:
  - **Aktív** — semleges/zöld stílus, ha `league.status === 'active'` (testid: `league-active-badge`).
  - **Archivált** — semleges szürke stílus, ha `league.status === 'archived'`
    (testid: `league-archived-badge`).
- [ ] Betöltés alatt spinner (`data-testid="spinner"`); hiba esetén hibabanner
      (`data-testid="error-banner"`).
- [ ] Üres lista esetén értelmes üres állapot szöveg (nem tört UI).

### Archiválás / visszaállítás

- [ ] Aktív ligánál (`status === 'active'`) **Archiválás** gomb (testid: `league-archive-btn`),
      ami `POST /api/admin/leagues/:id/archive`-ot hív a store-on keresztül.
- [ ] Archivált ligánál (`status === 'archived'`) **Visszaállítás** gomb (testid:
      `league-restore-btn`), ami `POST /api/admin/leagues/:id/restore`-t hív. A két gomb a
      `status` alapján egymást kizárva jelenik meg (toggle, sosem mindkettő egyszerre).
- [ ] Archiválás előtt megerősítő lépés (`window.confirm` a meglévő admin minta szerint, szöveg:
      `admin.leagues.archiveConfirm`); visszaállítás **nem** igényel megerősítést.
- [ ] Sikeres művelet után a sor státusza és a badge azonnal frissül (a store frissíti az adott
      liga `status` mezőjét a válaszból, vagy refetchel), és sikeres toast jelenik meg.
- [ ] Hiba esetén hibabanner + hiba-toast, a lista változatlan marad (nincs félrevezető UI-állapot).

### API wrapper (`api/index.ts` → `api.admin.leagues`)

- [ ] `list(token, opts?: { includeArchived?: boolean })` — `includeArchived: true` esetén
      `?includeArchived=true` query paramétert fűz; egyébként a meglévő viselkedés.
- [ ] `archive(token, id)` → `POST /admin/leagues/:id/archive`, `Promise<League>`.
- [ ] `restore(token, id)` → `POST /admin/leagues/:id/restore`, `Promise<League>`.
- [ ] Minden hívás `Authorization: Bearer <token>` fejléccel, a meglévő `request` wrapperrel.

### Frontend típus

- [ ] A frontend `League` interfészhez `readonly status: 'active' | 'archived'` mező hozzáadva
      (bővíthető union — később új állapotok jöhetnek, most csak `active`/`archived`).
      (Az `api.admin.leagues` visszatérési típusa `League` marad.)

  - `admin.leagues.title` = „Ligák kezelése"

  - `admin.leagues.empty` = „Nincs megjeleníthető liga."

### Tesztek (Vitest komponens teszt)

- [ ] Aktív ligán (`status === 'active'`) megjelenik az `league-active-badge` és az
      `league-archive-btn`, **nem** jelenik meg `league-restore-btn`.
- [ ] Archivált ligán (`status === 'archived'`) megjelenik az `league-archived-badge` és az
      `league-restore-btn`, **nem** jelenik meg `league-archive-btn`.
- [ ] Archiválás gomb kattintás → confirm elfogadva → a store `archive` metódusa a helyes
      liga-id-val hívódik (a `window.confirm` mockolva `true`-ra).
- [ ] Archiválás gomb kattintás → confirm elutasítva (`false`) → a store `archive` **nem** hívódik.
- [ ] Visszaállítás gomb kattintás → a store `restore` metódusa a helyes id-val hívódik,
      confirm nélkül.
- [ ] Betöltéskor a store `fetchLeagues` `includeArchived: true`-val hívódik.
- [ ] (Opcionális) store unit teszt: sikeres `archive`/`restore` frissíti az adott liga
      `status` mezőjét (`'archived'` / `'active'`); hiba esetén `error` beáll.

## Edge case-ek és megjegyzések

- **Idempotencia a UI-on:** a backend `archive`/`restore` idempotens (US-940). A UI sosem kínál
  fel „archiválás" egy már archivált ligára (a gomb toggle), így dupla-archiválás normál úton
  nem fordulhat elő; de ha mégis (pl. két tab), a válasz-`status` alapján a UI konvergál.
- **404 nem létező liga:** ha a liga időközben törlődött (`DELETE`), a backend 404-et ad; a UI
  hibabannert + toastot mutat és refetchel.
- **Concurrent állapot:** ha a szerver-válasz `status`-a eltér a lokális várttól, mindig a
  szerver-válasz a forrás (store a válaszból írja a mezőt).
- **Nincs create/edit/delete UI ebben a story-ban:** a liga létrehozás/szerkesztés/törlés nem
  scope — csak lista + státusz + archive/restore. (Ha később kell, külön story — US-951.)
- **Auth token:** a store a meglévő `getAccessToken()` mintát követi (Supabase session vagy
  `DEV_AUTH_BYPASS`), lásd `admin-teams.store.ts`.
