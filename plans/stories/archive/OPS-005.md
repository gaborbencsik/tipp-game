# OPS-005: HTTP cache headers a quasi-statikus GET endpointokon

## Probléma

A backend jelenleg **semmilyen** cache headert nem ad vissza. Minden megnyitott oldal teljes payloadot tölt le újra: csapatlista (60 sor), ligák, scoring config, scoring explainer, globális speciális tipp típusok — pedig ezek óránként legfeljebb egyszer változnak (vagy soha a session során).

Egy `Cache-Control: max-age=300` + ETag páros:
- A böngésző 5 percig **nem is** hív vissza (RAM cache hit)
- Lejárat után `If-None-Match` → ha nem változott, **304 Not Modified** üres body (~200 byte) helyett ~10-50 KB

**Becsült megtakarítás: 15-25%** a teljes egress-en (a [[OPS-004]] utáni második legnagyobb egress-szivattyú a quasi-statikus GET-ek halmaza).

---

## Story

Mint **platform-üzemeltető**, szeretném, hogy **a ritkán változó GET endpointok HTTP cache headerrel és ETag-gel térjenek vissza**, hogy **a böngésző és a CDN ne kérdezze újra le ugyanazt 5 percen belül, és ha mégis, csak diff-et töltsön le**.

## Elfogadási kritériumok

### A — Generikus cache middleware

1. [ ] Új fájl: `packages/backend/src/middleware/http-cache.middleware.ts`
2. [ ] Exportál egy `withHttpCache(options)` factory-t Koa middleware-rel:
   - `options.maxAge: number` — másodperc
   - `options.swr?: number` — `stale-while-revalidate` szekundum (opcionális)
   - `options.scope?: 'private' | 'public'` — default `'private'` (per-user válaszok!)
3. [ ] A middleware:
   - Lefuttatja a `next()`-et (a route handler beállítja `ctx.body`-t)
   - Ha `ctx.status === 200` és `ctx.body !== null`:
     - Kiszámol egy ETag-et a serializált body sha256 hash első 16 hex karakterén (`W/"<hash>"` weak ETag)
     - Beállítja: `Cache-Control: <scope>, max-age=<maxAge>, stale-while-revalidate=<swr>`
     - Beállítja: `ETag: <hash>`
     - Ha a kérés `If-None-Match` headere egyezik az új ETag-gel: `ctx.status = 304`, `ctx.body = null`
4. [ ] Csak `GET` kérésekre alkalmazódik (`HEAD` is mehet, egyéb metódusra no-op)

### B — Alkalmazás az endpointokon

A következő route-okra rárakni a middleware-t (mind GET, mind cache-elhető):

| Endpoint | maxAge | swr | scope | Indok |
|----------|--------|-----|-------|-------|
| `/api/teams` | 1800 | 3600 | private | csapatlista, VB előtt már fix |
| `/api/players` | 1800 | 3600 | private | játékoslista, ritka admin/sync változás |
| `/api/leagues` | 3600 | 7200 | private | ligák, alig változnak |
| `/api/leagues/:leagueId/teams` | 1800 | 3600 | private | liga-szűkített csapatok |
| `/api/scoring-config/default` | 600 | 1200 | private | globális pontkonfig |
| `/api/groups/:groupId/scoring-config` | 60 | 300 | private | csoport pontkonfig (admin írja) |
| `/api/scoring/explainer` | 600 | 1200 | private | szabályzat, semmilyen user-state |
| `/api/stat-prediction-templates` | 600 | 1200 | private | statikus konstans lista |
| globális `/api/special-prediction-types` | 60 | 120 | private | torna szintű spec tippek (rendezvény alatt változhat) |

Megjegyzés: a `private` scope kötelező mindenhol, ahol a JWT alapján más usernek más a válasz (pl. csoport-szűrés). Ahol a válasz teljesen user-független (`/api/scoring/explainer`), `public` is mehet.

### C — Tesztek

1. [ ] **Unit teszt** a `http-cache.middleware.ts`-re (`http-cache.middleware.test.ts`):
   - Stabil ETag: ugyanaz a body kétszer ugyanazt az ETag-et adja
   - Eltérő body → eltérő ETag
   - `If-None-Match` egyezés → 304, `ctx.body === null`, headerek megmaradnak
   - `If-None-Match` nem egyezik → 200, body benne marad
   - Nem-GET kérés → no-op (nincs ETag, nincs Cache-Control)
   - `ctx.status !== 200` → no-op
2. [ ] **Integration teszt** (1-2 endpointra, pl. `/api/teams`, `/api/scoring/explainer`):
   - Első GET: 200 + ETag header
   - Második GET ugyanazzal az `If-None-Match`-el: 304, üres body
3. [ ] Minden meglévő teszt zöld marad

### D — Frontend felülvizsgálat (kötelező mini-fix)

1. [ ] Ahol a frontend `Cache-Control: no-cache`-t küld a fetch wrapper-ben, **vegyük le** ezekről az endpointokról (vagy hagyjuk hogy a böngésző döntsön — a `private, max-age=N` válasz felülírja)
2. [ ] **Ne adjunk** `If-None-Match` headert kézzel a frontend kódban — a böngésző automatikusan kezeli a `fetch()` esetén

## Technikai megjegyzések

### Middleware vázlat

```typescript
// packages/backend/src/middleware/http-cache.middleware.ts
import type { Context, Next } from 'koa'
import { createHash } from 'node:crypto'

export interface HttpCacheOptions {
  readonly maxAge: number
  readonly swr?: number
  readonly scope?: 'private' | 'public'
}

export function withHttpCache(options: HttpCacheOptions) {
  const scope = options.scope ?? 'private'
  const directives = [
    scope,
    `max-age=${options.maxAge}`,
    options.swr !== undefined ? `stale-while-revalidate=${options.swr}` : null,
  ].filter(Boolean).join(', ')

  return async function httpCacheMiddleware(ctx: Context, next: Next): Promise<void> {
    await next()
    if (ctx.method !== 'GET' && ctx.method !== 'HEAD') return
    if (ctx.status !== 200 || ctx.body === null || ctx.body === undefined) return

    const serialized = typeof ctx.body === 'string' ? ctx.body : JSON.stringify(ctx.body)
    const hash = createHash('sha256').update(serialized).digest('hex').slice(0, 16)
    const etag = `W/"${hash}"`

    ctx.set('Cache-Control', directives)
    ctx.set('ETag', etag)

    if (ctx.request.headers['if-none-match'] === etag) {
      ctx.status = 304
      ctx.body = null
    }
  }
}
```

### Route alkalmazás példa

```typescript
// matches.routes.ts
import { withHttpCache } from '../middleware/http-cache.middleware.js'

router.get(
```

## Mit NE csinálj

- Ne tegyél `Cache-Control`-t POST/PUT/PATCH/DELETE-re
- Ne tegyél `public` scope-ot user-függő válaszokra (a CDN cache-elné és más usernek tálalná)
- Ne erőltesd a max-age-et 1 óránál hosszabbra a `/api/teams`-en — admin CRUD esemény után a usernek 5 perc várás max OK, 1 óra már zavaró
- Ne számolj ETag-et nagyon nagy (> 1 MB) válaszokra (CPU pazarlás) — eleve nincs ilyen ezeken az endpointokon, de ha lenne, kerülni
- Ne használj `strong` ETag-et (`"<hash>"` `W/` nélkül) — Koa default body serialization nem byte-ra ekvivalens a két kérés között garantáltan

## Várt egress-megtakarítás

- Frontend session indításkor jelenleg ~5-7 quasi-statikus GET fut le (~50-100 KB összesen)
- 5 perces cache hit: SPA navigációk **nem** hívják újra (0 byte)
- Cache miss + 304: ~200 byte fejléc (vs. ~10-50 KB body)

**Becslés: 15-25% teljes egress-csökkenés.**
