# OPS-011: Tick handler – push és player sync függetlenítése a futball sync gate-től

## Probléma

A Render `tipp-game-cron-tick` percenként POST-ol a `/api/internal/sync/tick` endpointra, ahol jelenleg minden alábbi munka a futball sync gate (`shouldRunSync` + napi API kvóta + `markSyncStarted` claim) **mögé** van zárva:

- `runMatchKickoffReminderJob` — meccs előtti push (45 percen belüli kickoff)
- `runDailyMatchReviewJob` — napi push hiányzó tippekről
- `runPlayerSyncIfDue` — napi 24h-s player sync

**Következmény:** ha a `mode` `final_only` / `adaptive` és nincs aktív / közelgő meccs, a sync gate `false`-t ad → a tick visszatér → a fenti munkák **sosem futnak le**. Ezért nem érkezett ma 12:00 körül a `daily_match_review` push, holott a job kód helyes.

A push joboknak nincs futball API hívása, a player syncnek saját 24h gate-je van. Egyiknek sem szabadna függnie a `shouldRunSync`-től vagy a kvótától.

## Cél

A `/tick` handler szervezze át a futási sorrendet úgy, hogy a push job-ok és a player sync **minden tickben lefussanak**, függetlenül attól, hogy a futball sync gate engedélyezi-e a sync-et vagy sem. Egyik munka hibája ne akadályozza a többit (try/catch izoláció, ahogy a player syncnél már most is).

## Hatókör

### `/tick` handler (`packages/backend/src/routes/internal-sync.routes.ts`)

Az új sorrend minden tickben:

1. **`runMatchKickoffReminderJob`** — közvetlenül futtatva, try/catch izoláció
2. **`runDailyReviewIfDueWindow(now)`** — új vékony wrapper a `runDailyMatchReviewJob` köré (lásd lent), try/catch izoláció
3. **`runPlayerSyncIfDue`** — kihúzzuk a sync ágból, try/catch izoláció (változatlan logika)
4. **Futball sync** — kvóta + `shouldRunSync` + `markSyncStarted` + `runAllLeagues` (változatlan)

A válasz body (`ctx.body`) **minden ágon** tartalmazza a `kickoffReminder`, `dailyReview`, `playerSync` mezőket. Ha a sync skippel: `{ skipped: true, reason: ..., kickoffReminder, dailyReview, playerSync }`. Ha fut: `{ synced: true, results, kickoffReminder, dailyReview, playerSync }`.

### Új függvény: `runDailyReviewIfDueWindow`

Helye: `packages/backend/src/jobs/daily-match-review.job.ts` (a meglévő job mellé).

Felelőssége: **napi 1× engedélyezni a daily review push-t, dedikált trigger időablakkal**.

```ts
const DAILY_REVIEW_TRIGGER_HOUR_BUDAPEST = 12  // 12:00 Budapest
const DAILY_REVIEW_WINDOW_MINUTES = 5          // 12:00 .. 12:04 inclusive

export async function runDailyReviewIfDueWindow(now: Date = new Date()): Promise<
  DailyMatchReviewJobResult | { skipped: true; reason: string }
> {
  const { hour, minute } = budapestHourAndMinute(now)
  if (hour !== DAILY_REVIEW_TRIGGER_HOUR_BUDAPEST || minute >= DAILY_REVIEW_WINDOW_MINUTES) {
    return { skipped: true, reason: 'not in trigger window' }
  }
  return runDailyMatchReviewJob(now)
}
```

**Idempotencia:** a `pushNotificationLog` `(user_id, type, scope_key)` unique constraint védi a duplikációt — ugyanazon a napon 5 perces ablakban max 1 push megy ki userenként, akkor is, ha a wrapper esetleg többször hívná.

### Tesztek

`packages/backend/tests/internal-sync.routes.test.ts` bővítés:

- **Push és player sync minden ágon fut:**
  - `mode = 'off'` → sync skippel, de `kickoffReminder`, `dailyReview`, `playerSync` válaszban benne, és a job-ok hívva.
  - `apiCallsToday >= limit` → sync skippel kvóta miatt, push + player futott.
  - `interval not elapsed` → sync skippel gate miatt, push + player futott.
  - `sync already in progress` → sync skippel claim miatt, push + player futott.
  - "minden gate átment" → sync fut, push + player ugyanúgy futott.
- **Hibaszigetelés:** ha `runMatchKickoffReminderJob` dob, a többi munka és a sync ettől még fut; a tick végül 200-zal tér vissza, és a body-ban `kickoffReminder: { error: '...' }` szerepel.
- **`runDailyReviewIfDueWindow` (új unit test):** időablakon belül delegál `runDailyMatchReviewJob`-ra; időablakon kívül `{ skipped: true, reason: 'not in trigger window' }`.

`packages/backend/tests/daily-match-review.job.test.ts` változatlan — a job belső viselkedését nem érintjük.

## Elfogadási kritériumok

1. A `/tick` handler push és player sync hívásai minden tickben lefutnak, függetlenül a `mode`-tól, a kvótától, a `shouldRunSync` döntésétől és a `markSyncStarted` claim eredményétől.
2. Egy push job hibája nem buktatja meg a többit és nem akadályozza meg a futball sync futását.
3. `runDailyReviewIfDueWindow` napi 1× tűz Budapest 12:00 ± 5 percen belül; bármikor máskor `{ skipped: true, reason: 'not in trigger window' }`.
4. A `daily_match_review` push ténylegesen kimegy egy olyan napon, amikor van mai (vagy 17:00 → 07:00 ablakban) meccs és a usernek van hiányzó tippje, akkor is, ha a futball sync mode `final_only` és gate-elve van.
5. Minden új és meglévő backend teszt zöld; `tsc --noEmit` tiszta.

## Mit ne csináljunk

- Ne adjunk hozzá új Render cron service-t — egyetlen percenkénti tick elég.
- Ne módosítsuk a `cron/run.sh`-t — továbbra is `/api/internal/sync/$ENDPOINT`-ra POST-ol.
- Ne módosítsuk `runMatchKickoffReminderJob` és `runDailyMatchReviewJob` belső logikáját.
- Ne piszkáljuk a `shouldRunSync` és a kvóta-kezelést — csak a sorrend változik.
- Ne adjunk hozzá új DB oszlopot / migrációt — a `pushNotificationLog` unique már elég az idempotenciához.
