# US-951: Admin liga-menedzsment UI (frontend) – liga létrehozás/szerkesztés sync-mezőkkel

> Epic: E-Admin – Liga- és tornakezelés

## Háttér

A US-948 létrehozza az admin liga-oldalt (`AdminLeaguesView.vue`, `/admin/leagues` route,
`admin-leagues.store.ts`), de az **szűk**: csak liga-**lista** + státusz badge +
archiválás/visszaállítás. Nincs benne liga **létrehozás** vagy **szerkesztés**.

A US-950 kibővíti a backend liga-CRUD-ot a sync-mezőkkel (`externalId`, `season`, `syncEnabled`,
`status`, opc. `syncFrom`/`syncTo`, `fixtureAllowlist`, `startsAt`) a **meglévő** `/leagues`
POST/PUT endpointokon keresztül. **Ahhoz, hogy az admin ténylegesen env és deploy nélkül, a
felületről tudjon új ligát bekötni vagy meglévőt átkonfigurálni, kell egy létrehozó/szerkesztő UI.**

**Cél:** a US-948 admin liga-oldal **kibővítése** liga létrehozás/szerkesztés form-mal, a
US-950 CRUD endpointjaira kötve — a lista/archive/restore rész **változatlan** marad.

## Story

Mint **admin**, szeretnék a liga-oldalon **új ligát létrehozni és meglévőt szerkeszteni**
(név, rövid név, külső API-azonosító, szezon, `status` láthatóság, `sync_enabled` kapcsoló),
hogy **egy tornát pusztán a felületről, adatból tudjak bekötni a szinkronba — env-változó és
deploy nélkül**.

## Scope – MI tartozik EHHEZ a story-hoz (frontend)

- Liga **létrehozás** form (modal vagy inline, a meglévő admin form-minta szerint) az
  `AdminLeaguesView.vue`-ban: mezők `name`, `shortName`, `externalId`, `season`,
  `status` SELECT (Aktív/Archivált) **és külön** `sync_enabled` toggle (opcionálisan
  `syncFrom`/`syncTo` ha egyszerű).
- Liga **szerkesztés** form ugyanezen mezőkkel, egy meglévő liga soráról indítva.
- Az `admin-leagues.store.ts` bővítése `create` + `update` akciókkal (a US-950 endpointokra).
- A frontend `League` típus bővítése az új mezőkkel (`externalId`, `season`, `syncEnabled`, stb.).
- Az `api.admin.leagues` wrapper `create`/`update` bővítése az új mezőkkel (ha a US-948 már
  hozzáadta a `create`/`update`-et, csak a mezőkkel egészítsd ki).
- Magyar i18n kulcsok + `data-testid`-k + Vitest komponens teszt(ek).

## Scope – MI NEM tartozik ide (előfeltételek / más story)

- A liga-**lista**, státusz badge, archiválás/visszaállítás UI → **US-948** (NE duplikáld).
- A backend CRUD (endpointok, service, validáció, DB) → **US-950** + **US-949** (NE végezz
  backend munkát).

> **Előfeltétel:** a US-948 oldala (`AdminLeaguesView.vue`, route, nav, store) és a US-950
> backend CRUD-ja (bővített `/leagues` POST/PUT + `League` DTO mezők) kész és deployolt.
> Ez a story a kész szerződésekre épül.

## Elfogadási kritériumok

### Létrehozás

- [ ] Az `AdminLeaguesView.vue`-n **„Új liga"** gomb (testid: `league-create-btn`), ami
      megnyit egy form-ot (modal vagy inline, a meglévő admin minta szerint —
      `AdminTeamsView.vue` / `AdminPlayersView.vue`).
- [ ] A form mezői: `name` (kötelező), `shortName` (kötelező), `externalId` (opcionális, integer),
      `season` (opcionális, integer), `status` SELECT (Aktív/Archivált, default „Aktív"),
      `sync_enabled` toggle (default kikapcsolva) — a `status` és a `sync_enabled` **külön**
      vezérlő, nem egy közös kapcsoló.
- [ ] Mentés a store `create` akcióján keresztül → `POST /admin/leagues` a US-950 bővített
      inputjával; siker után a lista frissül és sikeres toast jelenik meg.
- [ ] Validáció a UI-on: `name`/`shortName` üres → gátolt mentés; `externalId`/`season` nem-szám
      → hibajelzés (a backend validációs hibát is meg kell jeleníteni error-bannerben/toastban).

### Szerkesztés

- [ ] Minden liga soránál **„Szerkesztés"** gomb (testid: `league-edit-btn`), ami a form-ot a
      liga meglévő értékeivel tölti fel (a `status`-t a SELECT, a `syncEnabled`-et a toggle veszi át).
- [ ] Mentés a store `update` akcióján keresztül → `PUT /admin/leagues/:id`; siker után a sor és
      a lista frissül + sikeres toast.
- [ ] A meg nem változtatott mezők értéke megmarad (a form az aktuális `League`-ből töltődik).

### Store (`admin-leagues.store.ts`)

- [ ] `create(input)` akció: hívja `api.admin.leagues.create`-et, siker után frissíti a listát
      (refetch vagy lokális push), hiba esetén `error` beáll + hiba-toast.
- [ ] `update(id, input)` akció: hívja `api.admin.leagues.update`-et, siker után frissíti az adott
      liga sorát, hiba esetén `error` beáll + hiba-toast.
- [ ] Az `archive`/`restore`/`list` akciók (US-948) **változatlanok**.

### API wrapper (`api/index.ts` → `api.admin.leagues`)

- [ ] `create(token, input)` és `update(token, id, input)` az új mezőket (`externalId`, `season`,
      `syncEnabled`, `status`, opc. `syncFrom`/`syncTo`, `fixtureAllowlist`, `startsAt`) is elküldi.
- [ ] Minden hívás `Authorization: Bearer <token>` fejléccel, a meglévő `request` wrapperrel.

### Frontend típus (`types/index.ts`)

- [ ] A frontend `League` interfész bővítve: `readonly externalId: number | null`,

## Edge case-ek és megjegyzések

- **Archivált + `sync_enabled = true` (HARD RULE):** a form megengedi, hogy egy `status =
  'archived'` ligán a `sync_enabled` toggle be legyen kapcsolva, de **ez a liga akkor sem
  szinkronizálódik** (US-949 runner-szűrő: `sync_enabled = true AND status = 'active'`). Érdemes
  a form-on jelezni, hogy archivált státusz mellett a sync nem fut, bármit is mutat a toggle.
- **Backend validációs hiba megjelenítése:** duplikált `externalId` vagy tartományon kívüli
  `season` (US-950 validáció) → a UI a backend hibaüzenetét error-bannerben/toastban mutatja,
  a form nyitva marad, az adat nem vész el.
- **`sync_enabled = true` + üres `externalId`:** ha a backend ezt validációs hibának tekinti
  (US-950), a UI a hibát megjeleníti; a form-on érdemes jelezni, hogy syncehez `externalId` kell.
- **Concurrent szerkesztés:** siker után mindig a szerver-válasz `League` a forrás (store abból
  frissít), nem a lokális form-állapot.
- **Nincs új route/oldal:** ugyanaz az `AdminLeaguesView.vue` és `/admin/leagues` route —
  csak form-mal bővül.
