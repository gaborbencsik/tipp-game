# US-946: Bracket-progresszió tipp – pár-alapú győztes kiválasztás a kieséses ágon

> ✅ **Státusz:** kész (2026-05-30). Lásd `## Implementáció utáni jegyzetek` szekciót a story alján a végleges döntésekért.

>
> Megjegyzés: a design specekben szereplő "US-939" hivatkozás ID-ütközés miatt erre a storyra (US-946) cserélve. A v2 a hatályos design; a v1-es chip-pool változat csak referenciaként szerepel.

## User Story

Mint VB tippelő, szeretnék a torna **kieséses ágának teljes lefutását megtippelni 6 körön át** (Last 32 → Last 16 → negyeddöntő → elődöntő → döntő + bronz), úgy hogy minden körben **a tényleges torna-bracket szerinti párokat** látom, és **páronként a győztest** jelölöm meg. Szeretném, ha a Last 32 párosítások **automatikusan az én csoport végeredmény tippemből (US-945)** jönnének, és ha a tippet bármelyik körben módosítom, a későbbi körök **automatikusan frissüljenek** (a már nem érvényes győztes-pick-ek törlődjenek toast jelzés mellett).

## Cél, kontextus

A v1-es design a 6 kört **körönként független chip-pool**-ként kezelte (válassz N csapatot egy halmazból). A v2 alapján — ami a **hatályos design** — minden kör **pár-listát** mutat (Last 32: 16 pár, Last 16: 8 pár, …), és a user **páronként a győztest** pickeli. A győztesek az **admin-rögzített bracket-template** szerint automatikusan a következő kör pár-szerkezetébe kerülnek.

| Kör | Párok száma | Pool / forrás | User pick |
|-----|-------------|---------------|-----------|
| **Last 32** | 16 | Csoport végeredmény tipp (US-945) → bracket slot mapping | 16 győztes |
| **Last 16** | 8 | Last 32 győztesek + bracket-template | 8 győztes |
| **Negyeddöntő (8)** | 4 | Last 16 győztesek + bracket-template | 4 győztes |
| **Elődöntő (4)** | 2 | Negyeddöntő győztesek + bracket-template | 2 győztes |
| **Döntő (2)** | 1 | Elődöntő győztesek | 1 győztes (= bajnok) |
| **Bronz (1)** | 1 | Elődöntő vesztesek (auto) | 1 győztes (= bronz) |

**Kulcs előny a v1-hez képest:** a Last 16-tól kezdve a felhasználó **látja, milyen csapatok versenyezhetnek egyáltalán** egy adott pozícióért — a bracket-szerkezet kizárja, hogy két ugyanazon ágon lévő csapat találkozzon. (Ezt a v1 chip-pool nem tudta visszaadni.)

A teljes tipp **egyetlen `special_predictions` rekord**, JSON-encoded answerrel. Egységes deadline (a torna első csoportmeccsének kickoffja), auto-save (400 ms debounce).

| Story | Funkció | Mi a különbség ehhez képest? |
|-------|---------|-------------------------------|
| US-935 | Globális statisztikai bonus tippek | Egyetlen érték; nincs kör-rendszer. |
| US-936 | Upset Special (multi-csapatos, súlyozott) | Egy körös pick; nincs cascading. |
| US-937 | Progresszió bonus tippek (stage-szintű chip-pool) | Stage-enként független pick a 48 csapatból; nincs pár-szintű tipp és nincs körök közötti megkötés. |
| US-945 | Csoport végeredmény (12 csoport sorrend + 4 best 3rd) | A csoportkört fedi le; **ez a story (US-946) hard dependency-vel kapcsolódik hozzá**. |
| **Ez a story (US-946)** | **Bracket-progresszió pár-alapú győztes-pick** | A user a saját csoport-tippjéből származó bracket-slot-okkal indul, és **mérkőzésenként** jelöli a győztest. |

> **Implementációs sorrend:** US-937 → US-945 → **US-946 (ez)**. Feltétlenül ebben a sorrendben.

## Erős kapcsolat az US-945-tel

- A bracket template szlot-jai (W A1, RU B1, 3rd C/D/E, …) a torna **fix struktúrája**.
- A slot → csapat mapping a user **US-945-ös tippjéből** olvasódik.
- Ha a user nem töltötte ki az US-945-öt: a bracket UI **lockolt**, csak az „Töltsd ki előbb a Csoport végeredmény tippet" prompt látszik, ami **deeplinkkel** vezet az US-945 nézetbe.
- US-945 részleges állapotnál (pl. csak A–F csoport kész) is megjelenik: a betöltött slot-ok látszanak, a hiányzó slot-ok placeholder-ek (pl. „G csoport — még nincs tippelve") és lockoltak.

## Adatmodell

### `special_prediction_types` rekord (seed)

```ts
{
  name: 'Bracket-progresszió',
  description: 'Tippeld meg minden mérkőzés győztesét a torna kieséses ágán.',
  inputType: 'bracket_progression_paired',  // új enum érték
  isGlobal: true,
  isActive: true,
  deadline: '<első csoportmeccs kickoff>',
  options: {
    bracketTemplate: {
      // Fix admin-konfig — a torna sorsolása szerint.
      matches: [
        { id: 'l32_m1',  round: 'last_32', slotA: 'W_A1',     slotB: 'RU_B1',    winnerTo: 'l16_m1' },
        { id: 'l32_m2',  round: 'last_32', slotA: 'W_C1',     slotB: '3rd_DEF',  winnerTo: 'l16_m1' },
        { id: 'l32_m3',  round: 'last_32', slotA: 'W_D1',     slotB: 'RU_E1',    winnerTo: 'l16_m2' },
        { id: 'l32_m4',  round: 'last_32', slotA: 'W_F1',     slotB: '3rd_ABC',  winnerTo: 'l16_m2' },
        // ... 12 további l32 match (összesen 16 db)
        { id: 'l16_m1',  round: 'last_16', slotA: '<l32_m1>', slotB: '<l32_m2>', winnerTo: 'qf_m1' },
        // ... 7 további l16 match (összesen 8 db)
        // ... 4 qf, 2 sf
        { id: 'final',   round: 'final',   slotA: '<sf_m1>',         slotB: '<sf_m2>',         winnerTo: null },
        { id: 'bronze',  round: 'bronze',  slotA: '<sf_m1_loser>',   slotB: '<sf_m2_loser>',   winnerTo: null },
      ],
    },
    slotResolver: 'group_standings',  // jelzi, hogy az US-945 tippből kell deriválni a slot-okat
  },
}
```

A `bracketTemplate` torna-konfigtól függ; admin konfigurálja egyszer torna előtt (jelen story-ban SQL seed). A `slotResolver = 'group_standings'` egy referencia az US-945 típusra; a backend runtime-ban összerakja a slot → teamId mapping-et.

### `special_predictions.answer` (felhasználó tipp)

```json
{
  "winners": {
    "l32_m1":  "uuid-mex",
    "l32_m2":  "uuid-cze",
    "l32_m3":  "uuid-fra",
    "l32_m4":  "uuid-cmr",
    "l16_m1":  "uuid-mex",
    "...":     "...",
    "qf_m1":   "uuid-mex",
    "sf_m1":   "uuid-bra",
    "sf_m2":   "uuid-fra",
    "final":   "uuid-fra",
    "bronze":  "uuid-arg"
  }
}
```

- Kulcs = match ID a `bracketTemplate.matches` listából.
- Érték = `teamId` (a két párban szereplő csapat egyike).
- **Részleges állapot érvényes**: nem minden match kell ki legyen töltve. Submit-final csak akkor, ha minden match-hez van győztes.
- **Nincs külön `champion` / `bronze` mező** — `winners.final` a bajnok, `winners.bronze` a bronzérmes.

### Származtatott állapot (read-only, runtime computed)

A frontend és backend a következőket **számolja** a tárolt `winners` map-ből:

```ts
type DerivedBracket = {
  matches: Array<{
    id: string,
    round: 'last_32' | 'last_16' | 'qf' | 'sf' | 'final' | 'bronze',
    teamA: TeamId | null,    // null ha az upstream slot/match nem töltött
    teamB: TeamId | null,
    winnerId: TeamId | null, // user pick, vagy null
    isLocked: boolean,       // true ha teamA vagy teamB hiányzik
  }>,
}
```

A számítás:

1. **Last 32 match-ek `teamA` / `teamB`:** US-945 group standings tippből (vagy admin `correctAnswer`-ből, ha deadline után).
2. **Magasabb körök match-ek `teamA` / `teamB`:** a `winnerTo` mező alapján — minden match győztese az upstream `winnerTo` match egyik slot-ját adja.
3. **Bronze:** a 2 SF-vesztes (a két SF match teamA/teamB - winnerId).

### `correct_answer` (admin reveal után)

Ugyanaz a séma mint a user answer. A torna előrehaladtával fokozatosan tölthető (Last 32 lezárása után csak a `l32_*` mezők beállíthatók).

## Elfogadási kritériumok

### Backend – Schema

1. **Új `inputType` enum érték**
   - `'bracket_progression_paired'` — pár-alapú győztes-pick.
   - Drizzle migration: `ALTER TYPE special_prediction_input_type ADD VALUE IF NOT EXISTS 'bracket_progression_paired';`

2. **`options` JSONB séma**
   - `bracketTemplate.matches: Match[]` — 32 match definíció (16 + 8 + 4 + 2 + 1 + 1).
   - `slotResolver: 'group_standings'` — referencia az US-945 típusra.

### Backend – Validáció

3. **PATCH/POST `/api/special-predictions/:typeId`** elfogadja a részleges answer JSON-t és validálja:
   1. `winners` kulcsai mind a `bracketTemplate.matches[].id` halmazából.
   2. Minden `winners[matchId]` érték = `teamA` vagy `teamB` az adott match-ben (a derivált `teamA` / `teamB` ellen ellenőrizve).
   3. **Lánc-konzisztencia:** ha `winners[downstreamId]` létezik és az upstream győztes `winners[upstreamId]` nem szerepel a downstream párban → 400.
   4. Az US-945 group_standings tipp **olvasásra szükséges** a slot resolver-hez. Ha az US-945 nem létezik / üres → a backend a Last 32-es match-eket „locked" állapotként adja vissza (nem 400, csak a UI lockolt).

4. **Cascading mentés** — a backend **nem** módosít autonóm más rekordot. A frontend felelős az invalidálásért és a teljes szűrt `winners` map mentéséért. A backend csak a 3. pont validációt futtatja.

### Backend – Slot resolver

5. **Slot → teamId mapping**:
   - A backend (vagy a frontend store) a következő slot-kódokat értelmezi:
     - `W_<groupCode>1` → adott csoport 1.helyezett (`groups[X][0]`).
     - `RU_<groupCode>1` → adott csoport 2.helyezett (`groups[X][1]`).
     - `3rd_<groupCombo>` → a 4 továbbjutó 3.helyezett közül a `bracketTemplate` által megadott egy konkrét csoportkombinációhoz tartozó csapat (a torna sorsolási szabálya szerint).
   - A `correct_answer` (admin reveal) US-945-ből resolver-elt slot-okat használja.

### Backend – Seed

6. **Idempotens seed** egy globális `bracket_progression_paired` típushoz:
   - `is_global: true`, `is_active: true`.
   - `deadline`: a torna első csoportmeccsének kickoffja.
   - `options.bracketTemplate.matches`: a 2026-os FIFA VB sorsolása alapján (32 match definíció SQL seed-ben).
   - `ON CONFLICT DO NOTHING` (mint az US-935 / US-945 mintában).

   - Ha az US-945 nincs kitöltve: lockolt prompt + deeplink (lásd 9. pont).
   - 6 kör accordion (Last 32 → Last 16 → QF → SF → Final → Bronze).
   - Sticky bottom bar: `Last 32: X/16 · ✓ Mentve` + 32-elemű progress dot-sor (összes match).

8. **`RoundCard.vue`** (×6) — egy kör accordion kártya:

   - Expanded nézet: a kör match-jei `MatchCard`-ban listázva.
   - Lockolt állapot: ha a kör egyetlen match-je sem tölthető (mert minden upstream match üres) → kártya disabled, prompt: „Töltsd ki előbb az előző kört."

### Frontend – Pinia store és cascading reaktivitás

13. **`useBracketProgressionStore`**:
    - `state.winners`: `Record<MatchId, TeamId>` (sparse).
    - `state.saveStatus`: `'idle' | 'saving' | 'saved' | 'error'`.
    - **Cross-store reaktivitás:** `useGroupStandingsStore`-ból olvas (US-945).
    - `getters`:
      - `derivedBracket`: a fenti 3.3-as számított struktúra.
      - `isComplete(round)`: minden round match-jén van pick.
      - `overallProgress`: pick-elt match-ek / összes match.
    - `actions`:
      - `pickWinner(matchId, teamId)`: pick set + cascading invalidate (downstream match-ek, ha a győztes csapat eltűnt).
      - `clearWinner(matchId)`: pick törlés + cascade.
      - `flushSave()` (debounced).

14. **Pick-cascading** — ha a user módosít egy győztest, a downstream pick automatikusan invalidálódik:
    ```ts
    function pickWinner(matchId: MatchId, teamId: TeamId) {
      const oldWinner = state.winners[matchId]
      state.winners[matchId] = teamId
      if (oldWinner && oldWinner !== teamId) {
        const downstream = derivedDownstreamMatches(matchId, bracketTemplate)
        for (const dsId of downstream) {
          if (state.winners[dsId] === oldWinner) {
            delete state.winners[dsId]
          }
        }
      }
      scheduleSave()
    }
    ```

15. **Group-standings change reactivity** (cross-store) — ha a user módosítja az US-945-ös tippjét:
    ```ts
    watch(useGroupStandingsStore().answer, (newAnswer) => {
      const newSlotMap = computeSlotMap(newAnswer)
      const orphaned = []
      for (const [matchId, teamId] of Object.entries(state.winners)) {
        const match = derivedMatches[matchId]
        if (match.teamA !== teamId && match.teamB !== teamId) {
          orphaned.push(matchId)
          delete state.winners[matchId]
        }
      }
      if (orphaned.length > 0) {
        showToast(`A csoport-tipped módosult — ${orphaned.length} győztes-pick törlésre került.`)
        scheduleSave()
      }
    })
    ```

### Frontend – Auto-save

16. **Auto-save flow** (azonos az US-945-tel):
    - **Optimistic UI** + **400 ms debounce** + **teljes answer JSON PATCH**.
    - Sticky progress sávban státusz indikátor: `↻ Mentés…` / `✓ Mentve` / `⚠ Retry` / `⛔ Sikertelen`.
    - Race-condition: monoton request-id, last-write-wins.
    - Validációs hiba (4xx): nincs retry; rollback az utolsó sikeres állapotra.
    - 5xx / network error: exponenciális backoff (3×: 1s, 3s, 9s).

### Frontend – Read-only nézet (deadline után)

17. **Read-only nézet**:
    - Minden match-kártya: a user pickje **kiemelve**, a tényleges győztes (admin reveal) zöld outline-nal jelölve.
      - Ha egyezik: dupla zöld pipa, ✓✓.
      - Ha eltér: piros háttér a user pickjén (`team-slot--wrong`), zöld dashed outline a tényleges győztesön (`team-slot--actual`).
    - Minden kör fejlécében: „N / M helyes pick".
    - A pontszám-megjelenítés a pontozási story hatóköre.

### Admin – Eredmény beállítás

18. **Admin reveal UI** a `correctAnswer.winners` beállításához:
    - Match-ID alapú dropdown UI: minden mérkőzéshez a `teamA` / `teamB` közül választás.
    - Fokozatos kitöltés: Last 32 lezárása után csak a `l32_*` mezők szerkeszthetők; a többi a torna előrehaladtával nyílik.
    - Mentés után automatikus pontszámítás (külön story).

## Technikai megjegyzések

### Schema migration vázlat

```sql
ALTER TYPE special_prediction_input_type
  ADD VALUE IF NOT EXISTS 'bracket_progression_paired';
```

### Slot resolver példa

```ts
// W A1 = A csoport 1.helyezett (US-945 groups['A'][0])
// RU B1 = B csoport 2.helyezett (US-945 groups['B'][1])
// 3rd DEF = a 4 továbbjutó 3.hely közül a "DEF kombo" által beazonosított csapat
function resolveSlot(slotCode: string, us945Answer: GroupStandingsAnswer): TeamId | null {
  if (slotCode.startsWith('W_')) return us945Answer.groups[slotCode.slice(2, 3)]?.[0] ?? null
  if (slotCode.startsWith('RU_')) return us945Answer.groups[slotCode.slice(3, 4)]?.[0] ?? null
  if (slotCode.startsWith('3rd_')) {
    const combo = slotCode.slice(4)               // pl. 'DEF'
    return resolveBest3rdForCombo(combo, us945Answer.best3rds, us945Answer.groups)
  }
  return null
}
```

### Cross-store utility

A két Pinia store közötti reaktivitás új pattern a kódbázisban — érdemes egy kis `subscribeAcrossStores(sourceStoreGetter, callback)` utility-be refaktorálni az implementáció során, hogy más cross-store dependent feature-ök (pl. jövőbeli statisztikák) is használhassák.

## Edge case-ek

| Helyzet | Elvárt viselkedés |
|---------|-------------------|
| User még nem töltötte ki az US-945-öt | A teljes bracket lockolt; deeplink az US-945 nézetbe (`GroupStandingsGate`). |
| US-945 részlegesen kitöltött (pl. csak A–F kész) | A betöltött slot-okhoz tartozó match-ek megjelennek, a hiányzó slot-ok placeholder-rel; ezek a match-ek lockoltak, inline „[Csoport G tipp →]" link. |
| User módosítja az US-945-öt (pl. átírja A csoport 1.helyét MEX → KOR) | A bracket store reaktívan újraszámol; orphaned `winners` pick-ek törlődnek; toast jelez. |
| User pickel egy győztest, majd módosítja az upstream győztest | A downstream pick automatikusan invalidálódik (ha a régi győztes már nincs az új downstream párban). |
| User a Final-ben pickel, majd módosítja az SF-et | A Final pár csapata megváltozhat → ha a régi finalistát már nem nyerte meg az SF, a Final pick törlődik. Ugyanez a Bronze-ra. |
| Bronze auto-szinkron | A 2 SF-vesztes a `derivedBracket` számolja. SF-pick változás → Bronze pár automatikusan frissül; régi Bronze-pick orphan, toast. |
| Network drop mentés közben | Optimistic UI marad; reconnect után utolsó állapot egy mentésben megy ki. |
| Két tab nyitva | Last-write-wins; reload az authoritatív szerver-állapotra állít. |
| Deadline mentés közben telik le | Backend 400 Deadline elapsed; UI lockolt read-only. |
| Admin a csoportkör után beállítja a `last_32` correct_answer-t, a többi még üres | Read-only nézet a Last 32-höz színkódolt visszajelzést mutat; a többi kör vár. |
| User reload félig kész tippel | Szerverről jövő answer az autoritatív; az első kitöltetlen kör expanded. |

## Tesztelés

### Backend unit tesztek (`vitest`)

- Validáció: érvényes/érvénytelen részleges és teljes `winners` map kombinációk.
- Lánc-konzisztencia: orphaned winner pick → 400.
- Cross-store olvasás: US-945 részleges állapotnál a bracket helyesen jelez locked match-eket.
- Slot resolver: minden slot-kód típus (`W_X1`, `RU_X1`, `3rd_XYZ`) helyes csapatra fordul.
- Seed idempotencia.

### Frontend unit tesztek (`vitest`)

- `MatchCard` pick: tap valamelyik csapatra → state update; tap másikra → swap.
- `MatchCard` lockolt állapot: `teamA` vagy `teamB` null → kattintás disabled, toast.
- Cascading pick invalidate: pickelek MEX-et `l32_m1`-ben, majd `l16_m1`-ben (ahol MEX szerepel mint `l32_m1` győztese). Visszaállítom `l32_m1`-et KOR-ra → `l16_m1` MEX-pick törlődik.
- Group-standings change cascade: módosítok a US-945 store-ban → bracket store reaktívan invalidál pick-eket; toast megjelenik.
- Final + Bronze sync: SF1 győztest változtatok → Bronze SF1-vesztes oldala változik; régi Bronze-pick orphan kezelése.
- Auto-save:
  - Debounce: 5 gyors pick → 1 PATCH.
  - Sikertelen mentés visszaállít az utolsó sikeres állapotra.

### E2E (`playwright`)

- Boldog út: US-945 kitöltése → bracket nézet betöltődik → Last 32 16 match pick → Last 16 → QF → SF → Final + Bronze pick → reload → mentett állapot megjelenik.
- Cross-tipp módosítás: US-945 átírom egy csoport sorrendjét → bracket toast → érintett pick-ek törölve.
- US-945 nélkül: bracket nézet mutatja a lockolt prompt-ot, deeplink működik.
- Részleges US-945: csak A–F csoport kész → érintett match-ek pickelhetők, többi `⏳ vár` állapotban.
- Deadline átlépés szimulációja → lockolt read-only nézet.

## Implementáció utáni jegyzetek

A story implementációja során az alábbi finomítások történtek a tervhez képest:

### 1. US-945 fix scope-ban — `best3rdPicks: 4 → 8`

A FIFA WC2026 valójában **8 továbbjutó 3. helyezettet** ad (12 csoportból a 8 legjobb), nem 4-et. A US-945 seedjében elírás volt. A `0052_fix_best3rd_picks_to_8.sql` migration `jsonb_set`-tel javítja a `special_prediction_types` rekordot. Az i18n fordítások (`groupStandings.intro`) is frissültek.

### 2. FIFA Annex C teljes 495-kombinációs táblázat

A 8 továbbjutó 3. helyezett kombinatorikailag 12-ből 8 = **495 különböző kvalifikáló halmaz**. Minden halmazhoz a FIFA Annex C megadja, hogy melyik csoport 3. helyezettje melyik W_X1 ellen kerül. A teljes táblázat a `services/wc2026-annex-c.ts` (backend) és `lib/wc2026-annex-c.ts` (frontend) konstansba került, közvetlenül a hivatalos FIFA PDF-ből parse-olva.

### 3. Új enum érték (nem `bracket_progression_paired`)

A v1 chip-pool változat sosem indult prod-ban, így egyetlen enum érték elég: **`bracket_progression`** (nem a tervezett `_paired` szuffix).

### 4. Frontend könyvtárszerkezet — `lib/`, nem `utils/`

A projekt konvenciója a `packages/frontend/src/lib/` (deadline.ts, supabase.ts mintájára), nem `utils/`. A bracket derive utility ide került: `lib/bracketDerive.ts` + `lib/wc2026-annex-c.ts`.

### 5. UI: Bronz a Döntő ELŐTT a kártyán

A `FinalsAndBronzeCard.vue` sorrendje a hivatalos FIFA torna program alapján: **Bronz → Döntő** (a bronz egy nappal a döntő előtt játszódik). Korábbi rendezés: Döntő → Bronz, javítva.

### 6. Státusz badge: "Folyamatban" új állapot

A `TournamentTipsView.vue` badge logikája bővült. A több-pickes tippek (group standings, bracket progression) addig nem váltanak `Beadva` státuszra, amíg a `completion.totalDone < totalSteps`. Új state: **`Folyamatban`** (amber színű) — részleges válasz esetén. A `Beadva` (kék) csak `totalDone === totalSteps`-nél (12 csoport × 4 + 8 best3rd = 56, ill. 32 mérkőzés). Locale: `tournamentTips.statusInProgress`.

### 7. Pontozási logika placeholder

A `scoreBracketProgression()` szignatúra a tervezett, de **0-t ad vissza** — match-szintű pontozás külön story tárgya. Az admin reveal nem hibázik, csak nem ad pontot.

### 8. Out of scope megerősítve

- Bracket template admin szerkesztő UI (csak SQL seed).
- Klasszikus bracket-ábra desktop view (mobil-szerű accordion marad).
- US-937 stage enum integráció (független storyk).
