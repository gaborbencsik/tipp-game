## Probléma

A WC2026 48-csapatos formátuma miatt az egyenes kieséses szakasz **Round of 32**-vel kezdődik. Az api-football `league.round` mezőben ezt pontosan így adja vissza:

```
$ curl ".../fixtures?league=1&season=2026" | jq -r '.response[] | .league.round' | sort -u
Group Stage - 1
Group Stage - 2
Group Stage - 3
Round of 32
```

A jelenlegi `parseRound()` (`packages/backend/src/services/sync.service.ts:37-57`) **nem ismeri** a `"Round of 32"` stringet:

- `KNOCKOUT_MAP` csak `Round of 16`, `Quarter-finals`, `Semi-finals`, `3rd Place Final`, `Final` értékeket tartalmaz.
- A `match_stage` enum (`packages/backend/src/db/schema/index.ts:21-28`) sem tartalmaz `round_of_32` értéket.
- Ezért minden Round of 32 meccs **némán a default fallback**-ra esik (`stage: 'group'`, `groupName: null`), és csoportkör meccsként kerül a DB-be.

A `predictions.match_id` UUID FK-val mutat a `matches.id`-re, és a sync `external_id`-vel konfliktál — tehát a meccs `id`-je nem változik újra-sync-kor. A tippek és a scoring rekordok érintetlenek maradnak.

A felhasználó döntése: **csoport-meccsek `groupName` mezőjével nem foglalkozunk** (a knockout szakaszban már nincs csoport).

## Story

Mint **fejlesztő/admin**, szeretném, hogy **a Round of 32 meccsek a helyes `stage` értékkel kerüljenek a DB-be**, hogy **a felhasználók ne lássák tévesen csoportkör meccsként a 32-es döntőt, és a frontend stage-szerinti szűrés és felirat helyes legyen**.

## Elfogadási kritériumok

### A — DB schema

1. [ ] `matchStageEnum`-ba beszúrva: `'round_of_32'` (`round_of_16` ELÉ — a torna sorrendje szerint)
2. [ ] Új migration fájl: `0071_match_stage_round_of_32.sql` — `ALTER TYPE match_stage ADD VALUE IF NOT EXISTS 'round_of_32' BEFORE 'round_of_16'`

### B — Backend `parseRound`

1. [ ] `KNOCKOUT_MAP` bővítve: `['Round of 32', 'round_of_32']`
2. [ ] A néma fallback (`return { stage: 'group', groupName: null }`) kapjon `log.warn('parseRound_unknown_format', { round })`-ot, hogy a jövőben azonnal kiderüljön ha új round-format érkezik
3. [ ] A `MatchStage` type union (`packages/backend/src/types/index.ts:91`) tartalmazza a `'round_of_32'` értéket

### C — Backend tesztek (`tests/sync.service.test.ts`)

1. [ ] Új teszt: `parseRound('Round of 32') === { stage: 'round_of_32', groupName: null }`
2. [ ] Új teszt: ismeretlen round string esetén a logger `warn` metódusa meghívódik (mockolt logger)
3. [ ] A meglévő `"Regular Season - 22"` fallback teszt marad — a viselkedés továbbra is `{ stage: 'group', groupName: null }`, csak warn-loggal egészül ki

### D — Frontend type + UI

1. [ ] `MatchStage` type union (`packages/frontend/src/types/index.ts:22, 91`) bővítve `'round_of_32'`-vel
2. [ ] `KNOCKOUT_STAGES` konstansok bővítve `'round_of_32'`-vel (`MatchesView.vue`, `AdminMatchesView.vue`, `MatchDetailView.vue`)
3. [ ] `stageLabel()` függvények mindhárom view-ban kezelik a `'round_of_32'` esetet (`t('stage.round32')`)
4. [ ] Admin űrlapon (`AdminMatchesView.vue` stage `<select>`) választható: `<option value="round_of_32">32-es döntő</option>`
5. [ ] Új i18n kulcs: `stage.round32` — HU: `"32-es döntő"`, EN: `"Round of 32"`

### E — Re-sync (manuális lépés a deploy után)

1. [ ] A javítás éles deploy után egy normál fixtures sync futtatás — az `upsertFixtures` `onConflictDoUpdate` automatikusan átírja az érintett meccsek `stage` mezőjét `'group'`-ról `'round_of_32'`-re. A `matches.id` UUID nem változik, így minden meglévő tipp érintetlen marad.

## Mit NE csinálj

- Ne számold újra a `prediction_points` rekordokat — a user kifejezetten kérte, hogy nem kell újraszámolás.
- Ne nyúlj a csoport-stage `groupName` parse-oláshoz — a knockout szakaszban a csoportokkal már nem foglalkozunk.
- Ne módosítsd kézzel a meglévő migration fájlokat (`0001`–`0070`).

### Migration biztonság

A PG `ALTER TYPE ... ADD VALUE` nem futtatható tranzakción belül néhány PG-verzióban (pl. PG 12 előtt), de **PG 14+ támogatja transactional módban is**. A projekt PG 18.3-on fut. A custom migrator (`db/migrator.ts`) BEGIN/COMMIT-tel csomagolja a migration fájlt — PG 18 ezt elviseli.

`IF NOT EXISTS` a defenzív idempotencia miatt szükséges, hogy a már részben futtatott migration ne dobjon hibát.

### Re-sync mechanika

A `upsertFixtures` (`sync.service.ts:253-334`) `onConflictDoUpdate({ target: matches.externalId, set: { ..., stage: sql\`excluded.stage\` } })` — vagyis minden következő sync felülírja az érintett meccsek `stage` mezőjét. Mivel az `external_id` változatlan, a `matches.id` UUID stabil — `predictions.match_id` FK ép marad.
