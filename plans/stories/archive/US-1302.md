# US-1302: Match Pulse – AI insight generálás (batch job)

## Leírás

Mint **rendszer**, szeretnék **minden soron következő mérkőzéshez 5 különböző AI-generált statisztikai insight-ot előállítani és tárolni**, hogy **a felhasználók meccs előtt releváns, adatalapú elemzést kapjanak**.

## Elfogadási kritériumok

- [ ] Létezik egy `generateInsightsForMatch(matchId: string): Promise<void>` service függvény, amely:
  - Meghívja a `collectMatchStats(matchId)` függvényt (US-1308)
  - Az összegyűjtött adatot egy strukturált promptba rendezi
  - Pontosan 5 insight-ot kér az LLM-től, minden insight más kategóriából (`insight_type_enum` értékei közül az LLM választja a legfontosabbakat)
  - Az LLM válaszát `zod`-dal validálja az elvárt struktúra szerint
  - A validált insight-okat a `match_insights` táblába írja upsert-tel: `(match_id, type)` konflikt esetén `data`, `summary`, `generated_at`, `updated_at` frissül
- [ ] Az elvárt output séma per-insight: `{ type: InsightType, title: string (max 80 char), body: string (max 400 char), dataPoints: Record<string, number> }`
- [ ] Az LLM prompt **angol nyelven** íródik és **angol nyelvű** kimenetet kér: tartalmaz csapatneveket, a `RawMatchStats` tömörített összefoglalóját, explicit elvárást a 2-4 mondatos, konkrét számokat tartalmazó szövegre, anti-hallucinációs szabályt (csak a megadott adatokból), és a JSON output formátumot. A magyar fordítást egy külön story (US-1310) végzi a megjelenítés előtt.
- [ ] A `dataPoints` kulcsai kötelezően angol `snake_case` formátumúak (pl. `home_goals`, `away_clean_sheets`, `goals_per_match`) — a prompt explicit szabályt tartalmaz erre, és példát is ad
- [ ] Ha az LLM érvénytelen JSON-t vagy Zod-hibát dob, a függvény 1 automatikus újrapróbálkozás után `InsightGenerationError`-t dob
- [ ] Ha egy `type`-hoz már létezik insight a DB-ben és `generated_at` < 24 óra régi, a generálás kihagyja azt a típust (idempotens futás)
- [ ] Az LLM provider konfigurálható environment variable-ből: `INSIGHT_LLM_PROVIDER=gemini|claude|openai` (default: `gemini`), `INSIGHT_LLM_MODEL` (default: `gemini-flash-latest`)
- [ ] **Rate limit védelem (15 RPM):** a `generateInsightsForMatch` hívások egy közös `RateLimiter`-en mennek keresztül, ami biztosítja, hogy a Gemini API felé másodpercenként legfeljebb 1 hívás (≤ 15 RPM) menjen ki. Tömeges admin trigger esetén (pl. 48 meccs) a hívások egyenletesen szétkenve, sorrendben futnak — nem párhuzamosan, nem burst-ben. Konfigurálható env-ből: `INSIGHT_LLM_MIN_INTERVAL_MS=4000` (default: 4000 = 15 RPM)
- [ ] **Napi kvóta számláló:** új tábla `llm_usage_log` (oszlopok: `id`, `provider`, `model`, `match_id` nullable, `input_tokens`, `output_tokens`, `latency_ms`, `success`, `error_code` nullable, `created_at`). Minden Gemini hívás után rögzítendő egy sor. Az LLM client maga írja a logot — a generator service nem tud róla.
- [ ] **Napi limit guard:** generálás előtt a `RateLimiter` ellenőrzi a mai (UTC midnight óta) request count-ot. Ha eléri a `INSIGHT_LLM_DAILY_LIMIT`-et (default: 450, 50 puffer az 500 RPD-ből), a hívás `LlmDailyLimitExceededError`-t dob, nem küld API-hívást.
- [ ] **Admin usage endpoint:** `GET /api/admin/insights/usage` — visszaadja az aktuális napi statisztikát: `{ date, requestsToday, inputTokensToday, outputTokensToday, dailyLimit, remaining, last7Days: [{ date, requests, tokens }] }`. Admin middleware mögött.
- [ ] Unit tesztek (mockolt LLM + DB):
  - Sikeres eset: 5 insight kerül a DB-be
  - Érvénytelen JSON eset: 1 retry után `InsightGenerationError`
  - Részleges regenerálás: ha 2 típus már friss, csak 3 kerül generálásra
  - **Rate limiter:** 3 egymás utáni hívás esetén a 2. és 3. legalább `MIN_INTERVAL_MS` várakozással indul
  - **Napi limit:** ha az `llm_usage_log` mai sorai elérik a limitet, a 4. hívás `LlmDailyLimitExceededError` (nem hív API-t)
  - **Usage logging:** sikeres és sikertelen hívás után is keletkezik `llm_usage_log` sor (a hibásnál `success=false`, `error_code` kitöltve)
- [ ] Integration teszt: teljes flow mockolt api-football.com és LLM hívásokkal — ellenőrzi, hogy pontosan 5 sor kerül a `match_insights` táblába
- [ ] Admin trigger endpoint: `POST /api/admin/insights/generate` — opcionális `matchId` body param; ha nincs → az összes közelgő meccsre generál; response: `{ generated: number, skipped: number, errors: string[] }`
- [ ] Az admin trigger endpoint admin middleware mögött van
- [ ] **Admin sync UI:** új szekció az `AdminSyncView.vue`-n (`data-testid="insights-sync-section"`), a `raw-stats-sync-section` mintájára:
  - Cím: "Match Pulse AI insightok"
  - "Generálás indítása" gomb → `POST /api/admin/insights/generate` (matchId nélkül, batch mód)
  - Eredmény szöveg: `Kész: X generálva, Y kihagyva, Z hiba`
  - Napi usage kijelző: `Ma: {requestsToday} / {dailyLimit} kérés` (a `GET /api/admin/insights/usage` endpoint-ról), `onMounted`-kor és sikeres generálás után frissül
  - Ha `LlmDailyLimitExceededError` jön vissza, piros hibaüzenet jelenik meg
  - Magyarázó szöveg: "Az összes 'scheduled' meccshez 5 AI insight-ot generál a Gemini API-val. Rate limit: 15 RPM, napi keret: {dailyLimit}."

## Technikai megjegyzések

- LLM wrapper: `packages/backend/src/services/insights/llm.client.ts` — provider-agnosztikus interface, Gemini SDK (`@google/genai`) implementációval (Anthropic / OpenAI opcionális, később)
- Prompt builder: `packages/backend/src/services/insights/prompt.builder.ts` — külön fájl, nem a generáló service-be hardcodeolva
- Rate limiter: `packages/backend/src/services/insights/rate-limiter.ts` — singleton, in-memory queue-val. Process restart után reset, de a napi limit guard a DB-ből (`llm_usage_log`) számol, így az nem reset-elődik
- Usage repository: `packages/backend/src/services/insights/usage.repository.ts` — `recordUsage()`, `getDailyStats(date)`, `getLast7Days()`
- Haiku → Gemini: a `RawMatchStats` összefoglalóját tömöríteni kell, hogy az input ~4k token alatt maradjon (POC: 553 input token volt elég 5 insight-hoz)
- A `summary` mező a `body` első mondatával töltendő (manuális truncate, nem LLM-generált)
- `generateInsightsForMatch` helye: `packages/backend/src/services/insights/generator.service.ts`
