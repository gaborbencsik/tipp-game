# US-1502: Élő meccs eredmény megjelenítés és virtuális pontszámítás

## Kontextus

Amikor egy meccs `live` státuszban van és a sync pipeline frissíti az eredményt, a felhasználó jelenleg nem látja az aktuális állást. A feature célja: élő eredmény megjelenítése vizuálisan megkülönböztetve a végeredménytől, plusz egy "virtuális pont" kijelzés, ami mutatja, hány pontot kapna a játékos a tippjéért, ha most lenne vége a meccsnek.

## Architektúrális döntés

A tanácsi deliberáció (Aristotle / Ada / Feynman) konszenzusa alapján:

1. **A `match_results` tábla szemantikája "végleges eredmény" — élő részeredményt NEM írunk bele.** A `UNIQUE(matchId)` finalize-kor ütközne, és a tábla diszkriminátora elveszne.
2. **Új `live_match_states` tábla** tárolja az élő részeredményt — explicit szétválasztás végleges és átmeneti adat között.
3. **A `virtualPoints`-ot a backend számolja** (nem a frontend) — egyetlen forrásból `calculatePoints` + `ScoringConfig`, nincs duplikáció.

## Adatbázis változás

Új tábla: `live_match_states`

```ts
export const liveMatchStates = pgTable('live_match_states', {
  matchId:     uuid('match_id').primaryKey().references(() => matches.id, { onDelete: 'cascade' }),
  homeScore:   smallint('home_score').notNull(),
  awayScore:   smallint('away_score').notNull(),
  minute:      smallint('minute'),
  apiStatus:   text('api_status'),
  updatedAt:   timestamp('updated_at', { withTimezone: true }).notNull().defaultNow(),
})
```

**Életciklus:**

| Esemény | Művelet |
|---|---|
| Meccs `live`-ra vált, sync tick | UPSERT a `live_match_states`-be |
| Sync tick élő meccsen (újabb tick) | UPDATE (home/away/minute) |
| Meccs `finished` | tranzakció: DELETE `live_match_states` + INSERT/UPDATE `match_results` |
| Meccs `cancelled` | DELETE `live_match_states` |

## Elfogadási kritériumok

### 1. Backend — sync pipeline

- [ ] A sync runner az API-Football `goals.home` / `goals.away` / `fixture.status.elapsed` mezőit `status='live'` esetén a `live_match_states` táblába írja (UPSERT a `matchId`-re)
- [ ] `status='live'` esetén a `match_results` táblát NEM érinti
- [ ] Ha az API státusz finalizáltra vált (`FT`, `AET`, `PEN`), egyetlen tranzakcióban: DELETE `live_match_states` + UPSERT `match_results`
- [ ] Ha az API státusz `cancelled`-re vált: DELETE `live_match_states`

### 2. Backend — virtual points endpoint

- [ ] Új végpont: `GET /api/matches/virtual-points?groupId=<uuid>` — visszaadja az aktuális user adott csoportbeli leadott tippjeinek élő virtuális pontjait
- [ ] Válasz formátum: `{ matchId, homeScore, awayScore, minute, virtualPoints }[]` — csak azokra a meccsekre, amiknek van `live_match_states` rekordja ÉS van leadott tipp
- [ ] A `virtualPoints` számítás: `calculatePoints(prediction, liveMatchStateAsResult, scoringConfig)` ahol `scoringConfig` a csoport effektív konfigja (group override vagy global)
- [ ] Auth: bejelentkezett user, csak saját tippek
- [ ] Teszt: pure function `calculatePoints` változatlan, új service-réteg teszteli az élő → virtualPoints leképezést

### 3. Backend — adatfrissítés

- [ ] Frontend polling-gel frissít: `GET /api/matches/virtual-points?groupId=X` 30 másodpercenként, csak amíg legalább egy élő meccs van a Tippjeim nézetben
- [ ] SSE támogatást NEM ez a story építi — külön story (jövőbeli) feladata, ha a polling terhelése indokolja

### 4. Frontend — élő eredmény megjelenítés

- [ ] Ha `match.status = 'live'` és van `liveScore`, az aktuális gólállás megjelenik a meccs kártyán
- [ ] LIVE badge + a végeredménytől eltérő vizuális stílus (pulzáló indikátor vagy szín)
- [ ] `liveScore` hiányában (live státusz, de még nincs adat): "0–0" + LIVE badge fallback
- [ ] Ha `match.status = 'finished'`, az élő jelzés eltűnik, a `match_results` veszi át

### 5. Frontend — virtuális pont megjelenítés

- [ ] A "Tippjeim" nézetben minden élő, leadott tippes meccsnél: `~N pont (élő)` jelölés
- [ ] Az érték a `GET /api/matches/virtual-points` endpointról jön
- [ ] Polling 30 másodpercenként, amíg élő meccs van a nézetben — különben kikapcsol
- [ ] Általános meccs listán NEM jelenik meg, csak a "Tippjeim" nézetben
- [ ] Ha nincs leadott tipp egy élő meccsre, a virtuális pont nem jelenik meg

### 6. Üzleti szabályok

- [ ] A virtuális pont sosem kerül a `group_prediction_points` táblába
- [ ] A ranglistát nem befolyásolja — az csak `finished` meccsek után számolódik
- [ ] Finalizáláskor: a normál `calculateAndStorePoints` flow fut, függetlenül az élő virtuális pontoktól

## Tesztelési stratégia

- **Unit:** `calculatePoints` változatlan (már lefedett)
- **Service:** új `live-match-state.service.ts` — UPSERT, finalize tranzakció, cancel
- **Integration:** sync runner — `live` API válaszra `live_match_states` upsertel, `FT` válaszra finalize
- **Endpoint:** `GET /api/matches/virtual-points` — autentikált user, csoport, tippek + élő állapotok kombinációja
- **E2E (smoke):** élő meccs a UI-ban LIVE badge-et mutat, virtuális pont megjelenik a Tippjeim nézetben

## Migráció

- Új migráció: `live_match_states` tábla létrehozása
- Adatmigráció nem kell — üres táblával indul

## Kizárások (NEM része ennek a storynak)

- Élő animáció / gól-effekt
- Push notification élő gólról
- Élő eredmény alapján ranglistaváltozás előnézet
- Hosszabbítás / tizenegyes virtuális pont (a `liveScore` csak alapidő pontszámot tartalmaz)
- Virtuális pont megjelenítés az általános meccslistán (csak "Tippjeim" nézetben)
- Multi-instance backend deploy konzisztencia (jelenleg single-dyno-n fut)
- **SSE infrastruktúra** — frontend polling, nem push (külön jövőbeli story)

## Tanácsi háttér (referencia)

- **Aristotle**: külön tábla = explicit kategória-szétválasztás (végleges vs átmeneti)
- **Ada**: backend számol `virtualPoints`-ot, nem a frontend — egyetlen logika forrás
- **Feynman**: API `goals.home/away` egyetlen mezőből származik élő ÉS végleges állás is, csak a `status` dönt — finalize-kor ezért átmozgatás kell `match_results`-ba
