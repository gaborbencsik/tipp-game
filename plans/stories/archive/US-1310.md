# US-1310: Match Pulse – AI insight magyar fordítás

## Leírás

Mint **magyar felhasználó**, szeretném **a Match Pulse insight-okat magyar nyelven olvasni**, hogy **természetes nyelvi élményt kapjak a meccs előtti elemzésnél**.

## Háttér

Az US-1302 a Gemini insight-okat **angol nyelven** generálja, mert ez stabilabb JSON struktúrát, kevesebb hallucinációt és konzisztens `dataPoints` kulcsokat ad. A magyar UX-hez ezeket le kell fordítani, viszont a fordítás külön kvótát fogyaszt és nem minden insight-ra van szükség azonnal — ezért külön story.

## Elfogadási kritériumok

### DB séma

- [ ] Új migráció: a `match_insights` tábla kibővül két nullable oszloppal:
  - `title_hu TEXT` — magyar fordítás a `title`-höz
  - `body_hu TEXT` — magyar fordítás a `body`-hoz
- [ ] Új oszlop: `translated_at TIMESTAMPTZ` — fordítás időpontja, vagy null ha még nincs
- [ ] Az `dataPoints` (JSONB) **nem fordítandó** — a kulcsok angol snake_case-ek maradnak

### Service réteg

- [ ] Új függvény: `translateInsight(insightId: string): Promise<void>` a `packages/backend/src/services/insights/translator.service.ts`-ben
- [ ] A függvény az insight `title` + `body` mezőit egyetlen Gemini hívásban fordítja le (egy hívás / insight, nem 5)
- [ ] A fordító Gemini hívás ugyanazt az `LlmClient` interface-t és `RateLimiter`-t használja mint az US-1302 (közös 15 RPM limit, közös napi kvóta, közös `llm_usage_log`)
- [ ] A `usage_log` `provider` mezője `gemini-translate` értéket kap (megkülönböztethető generálástól), `match_id` az insight `match_id`-jával töltve
- [ ] Új batch függvény: `translateInsightsForMatch(matchId: string): Promise<void>` — egy meccs összes le nem fordított insight-jára meghívja a `translateInsight`-ot, sorban
- [ ] Idempotens: ha `title_hu` és `body_hu` már nem-null és `translated_at` < 7 nap régi, kihagyja
- [ ] Hibakezelés: 1 retry után `InsightTranslationError`. A hiba nem akadályozza meg a többi insight fordítását ugyanazon meccsen.

### Fordító prompt

- [ ] A prompt rövid, fókuszált, **angol nyelvű utasításokkal**: "Translate the following football insight from English to natural, journalistic Hungarian. Keep numbers and team names. Do not add or remove information."
- [ ] Input: a `title` és `body` JSON-ben átadva
- [ ] Output: JSON `{ titleHu: string, bodyHu: string }`, Zod-validálva
- [ ] A Hungarian output stílusa: **újságírói, semleges, nem szóról-szóra fordítás** — szakmai sportzsargon engedélyezett ("együttes", "találkozó", "góltermés")
- [ ] Char limit megőrzendő: `title_hu ≤ 100 char` (kicsit lazább mint az angol 80, mert a magyar tipikusan hosszabb), `body_hu ≤ 500 char`

### Trigger pontok

- [ ] **Automatikus:** az `generateInsightsForMatch` (US-1302) befejezése után, közvetlenül meghívja a `translateInsightsForMatch`-t — opcionálisan, env flag-gel kapcsolható: `INSIGHT_AUTO_TRANSLATE=true|false` (default: `true`)
- [ ] **Manuális:** új admin endpoint `POST /api/admin/insights/translate` — opcionális `matchId` body param; ha nincs → minden még nem fordított insight; response: `{ translated: number, skipped: number, errors: string[] }`
- [ ] **Lazy fallback:** ha az API valamiért elhagyja a fordítást és a frontend kapná az insight-ot fordítás nélkül, az API response a `title_hu`/`body_hu` mezőkben az angol értéket adja vissza fallback-ként (nem null), így a UI sosem ír ki üres szöveget

### Quota tervezés

- [ ] A napi `INSIGHT_LLM_DAILY_LIMIT` (US-1302, default 450) megosztott a generálás és fordítás között. Egy meccs teljes feldolgozása: **6 hívás** (1 generálás + 5 fordítás). 48 meccs = 288 hívás → bőven a limit alatt.
- [ ] Az admin usage endpoint (`GET /api/admin/insights/usage`) megkülönbözteti a `generate` és `translate` hívásokat: `{ requestsToday: { generate: number, translate: number, total: number }, ... }`

### Tesztek

- [ ] Unit tesztek (mockolt LLM):
  - Sikeres fordítás: `title_hu` és `body_hu` kitöltve, `translated_at` set
  - Idempotens: friss fordítás esetén kihagy
  - Egyetlen insight hibája nem akadályozza a többit ugyanazon meccsen
  - Char limit túllépés esetén Zod-hiba → retry → `InsightTranslationError`
  - Auto-translate env flag: ha `false`, a generálás után nem hív translation-t
- [ ] Integration teszt: 1 meccsre teljes flow (generate → translate), ellenőrzi hogy mind az 5 insight `title_hu` és `body_hu`-val rendelkezik

## Technikai megjegyzések

- A fordítás külön szolgáltatás, hogy később könnyen lecserélhető legyen lokális modellre, DeepL-re vagy ne-AI fordítóra
- Megfontolandó: a fordítást **csak akkor** indítani amikor a meccs T-24 óra közelében van — most automatikus a generálás után, de cron/scheduling story (US-1303) finomíthatja
- A frontend (US-1304) a `title_hu` és `body_hu` mezőket olvassa, az angol eredetit nem mutatja (de el is tárolva marad audit/debug célra)

## Nem tartalmazza

- Más célnyelvek (csak magyar)
- Cron ütemezés (US-1303 hatóköre — ott eldől, hogy a translate is automatikus-e a generálás után)
- Frontend megjelenítés (US-1304)
- DataPoints kulcsok fordítása — szándékosan angolok maradnak

## Komplexitás

## Follow-up notes (implementáció)

- Auto-translate hook a `generateInsightsForMatch`-ban (nem a batch service-ben), így bármely jövőbeli hívó (cron US-1303) örökli; env flag: `INSIGHT_AUTO_TRANSLATE` (default `true`).
- LLM hívás megosztott `callJson<T>()` helperrel a `LlmClient`-en belül — `generateInsights` (`provider: gemini`) és `translate` (`provider: gemini-translate`) közös rate limiter + usage log.
- `api.formatter.ts` előretekintő helper a US-1304-hez (még nincs insight-fetching endpoint); lazy fallback `titleHu/bodyHu` → angol ha nincs fordítás.
- Admin endpoint a meglévő `/admin/sync` route-csoporton: `POST /admin/sync/insights-translate`.
