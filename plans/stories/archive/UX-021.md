# UX-021: Statisztika liga-szűrő és ranglista scope perzisztálása localStorage-ben

## Leírás

Mint **bejelentkezett felhasználó**, szeretném, hogy **a Statisztika (Tippjeim) oldalon a liga-szűrő, a Ranglista oldalon pedig a scope (Globális / csoport) választásom megmaradjon page reload és új session után is**, hogy **ne kelljen minden látogatáskor újra kiválasztanom ugyanazt a szűrést**.

## Jelenlegi helyzet

- **Mérkőzések oldal** (`MatchesView.vue`): a liga-szűrő már perzisztált localStorage kulccsal (`matches_league_filter`)
- **Statisztika oldal** (`MyTipsView.vue`): ugyanazt a `matchesStore.leagueFilter` reactive ref-et használja, de in-memory; ha a user direkt `/app/my-tips` URL-re tölt be (pl. reload), akkor a store default `null` állapotból indul, és a meccsek oldal LS értéke nem töltődik vissza
- **Ranglista oldal** (`LeaderboardView.vue`): `selectedScope` mindig `'global'` default-ra ugrik vissza minden mount-kor

## Elfogadási kritériumok

### Statisztika oldal (MyTipsView)

1. **Megosztott LS kulcs a meccsekkel**
   - A statisztika oldal a meccsek oldallal **azonos** LS kulcsot használ: `matches_league_filter`
   - A MatchesView jelenlegi LS init és write-back logikája kiemelhető közös composable-be (pl. `useLeagueFilter()` `composables/`-ben), és mind a két oldal ezt használja
   - Cél: ha a user a meccseken Premier League-et választ, a Statisztikán is azt látja, és fordítva

2. **Init logika a Statisztika oldalon**
   - `onMounted`-ben: ha az `matchesStore.leagueFilter` még `null` (nincs in-memory érték), akkor olvassa be a `matches_league_filter` LS kulcsot
   - Ha az LS érték érvényes (a user jelenlegi `userLeagues` listájában szerepel) → beállítja
   - Ha nem érvényes (törölt csoport / kilépett liga) → `null`-t hagy, és törli az LS-ből
   - Ha a usernek pontosan 1 ligája van → automatikusan ezt választja (mint a meccsek oldalon)

3. **Watch + LS write-back**
   - `matchesStore.leagueFilter` változására ugyanaz a write-back logika fut (érték → `setItem`, `null` → `removeItem`), mint a meccsek oldalon
   - Idempotens: ne írjon kétszer ha mind a két oldal mountolva van

### Ranglista oldal (LeaderboardView)

4. **Külön LS kulcs scope-hoz**
   - LS kulcs: `leaderboard_scope`
   - Tárolt érték: `'global'` vagy egy csoport `id` (UUID string)
   - **Indok:** a ranglista scope nem liga-választás, hanem csoport/globális — más koncepció, ezért külön kulcs

5. **Init logika**
   - `onMounted`-ben: `localStorage.getItem('leaderboard_scope')` beolvasása
   - Ha `'global'` → `selectedScope = 'global'`, globális leaderboard betöltése
   - Ha csoport-id, és a `groupsStore.groups`-ban szerepel → `selectedScope = id`, csoport leaderboard betöltése
   - Ha csoport-id, de már nincs a user csoportjai között (kilépett, törölve) → fallback `'global'`-ra, és törli az LS kulcsot
   - Ha LS üres → marad a jelenlegi `'global'` default

6. **Watch + LS write-back**
   - `selectedScope` változására (vagy `onScopeChange` callback-ben) `setItem('leaderboard_scope', value)`

### Edge case-ek

7. **localStorage nem elérhető (privát mód, kvóta, kikapcsolt)**
   - Mind a három oldal try/catch-csel ellátva → silent fallback default értékre (mint a jelenlegi MatchesView megvalósítás)

8. **Cross-tab szinkronizáció**
   - **Nem** követelmény: `storage` event listener kiépítése külön story (kihagyva)
   - Ha a user két tabban más-más ligát választ, az utolsó írás nyer (ahogy most is működne a meccs oldalon)

9. **SSR / hydration**
   - A projekt CSR-only Vue 3, így nincs SSR mismatch kockázat — `onMounted`-ben olvasni biztonságos

## Technikai megjegyzések

### Közös composable (javasolt refaktor)

- **Új fájl:** `packages/frontend/src/composables/useLeagueFilter.ts`
- **Felelősség:** a `matches_league_filter` LS kulcs olvasása + write-back logikája egy helyen
- **API:**
  ```ts
  function useLeagueFilter(userLeagueIds: ComputedRef<readonly string[]>): {
    initFromStorage(): void
  }
  ```
- A composable a `matchesStore.leagueFilter`-t írja közvetlenül (Pinia store reactive)
- Mind `MatchesView.vue`, mind `MyTipsView.vue` ezt használja `onMounted`-ben

### Ranglista LS

- **Fájl:** `packages/frontend/src/views/LeaderboardView.vue`
- **Új konstans:** `const LS_LEADERBOARD_SCOPE = 'leaderboard_scope'`
- **Init:** az `onMounted` jelenlegi `loadLeaderboard()` hívása előtt olvassa be az LS-t és állítsa be `selectedScope`-ot
- **Watch:** `watch(selectedScope, val => localStorage.setItem(LS_LEADERBOARD_SCOPE, val))` — vagy az `onScopeChange`-ben írja le

### Tesztelés

- **MyTipsView.test.ts** (új tesztek):
  - LS-ből betölti a liga-szűrőt mount-kor, ha érvényes
  - Érvénytelen LS érték → null + LS kulcs törölve
  - 1 liga + üres LS → auto-default az egyetlen ligára
- **LeaderboardView.test.ts** (új fájl, ha nincs):
  - LS-ből betölti a scope-ot mount-kor (`'global'` vagy csoport-id)
  - Érvénytelen csoport-id → fallback `'global'`-ra + LS törölve
  - Scope váltás → LS kulcs frissül
- **MatchesView.test.ts**: jelenlegi LS tesztek továbbra is zöldek maradnak (regresszió)

## Kizárások

- Cross-tab `storage` event szinkronizáció — külön story
- Backend változtatás nincs (csak frontend persistálás)
- Filterek további típusai (idő-tartomány, csak saját csoportok stb.) — későbbi story
- `MatchesView` jelenlegi LS implementáció működése változatlan, csak a logika kerül kiemelésre composable-be
