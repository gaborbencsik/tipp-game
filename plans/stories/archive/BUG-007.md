# BUG-007: Csoport-leaderboard nem szűri ki a soft-deleted usereket

## Leírás

A csoport-szintű ranglista (`group-leaderboard.service.ts`) listájában megjelennek azok a felhasználók is, akiknél a `users.deleted_at` nem `NULL` (soft-deleted). A globális leaderboard helyesen szűri őket, a csoport-leaderboard viszont nem.

Ennek következménye: ha egy usert a Supabase Auth-ból törölnek és a saját `public.users` táblában is `deleted_at`-tel megjelölődik (pl. a `BUG-002` triggerrel), a globális ranglistáról eltűnik, de a csoport ranglistákon továbbra is látszik.

## Reprodukció

1. Egy csoport-tag userre futtass: `UPDATE public.users SET deleted_at = now() WHERE id = '<user_id>'`
2. Nyisd meg a csoport ranglista oldalát, ahol a user tag
3. **Várt:** a user nem jelenik meg
4. **Tényleges:** a user megjelenik a ranglistán

## Elfogadási kritériumok

- [ ] A `group-leaderboard.service.ts` mindkét query-jében (favorite team double points-os és normál ág, kb. `:60-93`) a `users` join-ra `isNull(users.deletedAt)` szűrő kerül
- [ ] A globális leaderboard viselkedése változatlan
- [ ] Unit/integration teszt fedi le: soft-deleted csoport-tag NEM szerepel a `getGroupLeaderboard` válaszban
- [ ] Aktív (deleted_at IS NULL) tagok továbbra is megjelennek
- [ ] A specialPredictions / groupPredictionPoints alqueryk is konzisztensek: ne számoljanak pontokat soft-deleted userhez (vagy ha számolnak, az eredmény ne jelenjen meg sehol)
- [ ] Manuális verifikáció: a 3 korábban azonosított árva user (akik már hard delete-tel eltávolításra kerültek) helyett egy szándékosan soft-deleted user sem látszik a csoport oldalain

## Technikai jegyzetek

**Érintett fájl:**
- `packages/backend/src/services/group-leaderboard.service.ts` (`:68`, `:89` környékén — `innerJoin(users, eq(users.id, groupMembers.userId))`)

**Megoldás vázlat:**
```ts
.innerJoin(users, and(
  eq(users.id, groupMembers.userId),
  isNull(users.deletedAt),
))
```

vagy a `where` klauzulához hozzáadni: `isNull(users.deletedAt)`.

**Megfontolandó:**
- Érdemes a `groupMembers` rekord törlése helyett is csak filterelni? (jelenleg a member rekord ott marad, csak a user soft-deleted)
- Az audit log megőrzése miatt jó hogy a member rekord nem törlődik kaszkáddal — a UI szűrés a helyes megoldás
