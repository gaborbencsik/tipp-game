# OPS-009: JWKS client singleton — Supabase egress és auth latency csökkentés

## Probléma

A `packages/backend/src/middleware/auth.middleware.ts`-ben a `getSigningKey()` függvény **minden authentikált API kérésnél új `jwksClient` instance-t gyárt**:

```ts
async function getSigningKey(kid: string): Promise<string> {
  const client = makeJwksClient()   // ← ÚJ INSTANCE MINDEN HÍVÁSNÁL
  const key = await client.getSigningKey(kid)
  return key.getPublicKey()
}
```

A `jwks-rsa` könyvtár **instance-szintű** cache-t használ (`cache: true, cacheMaxAge: 600000`), így új instance-szal a cache **mindig üres**, és minden hívás friss `GET /auth/v1/.well-known/jwks.json` kérést indít a Supabase felé.

### Mért hatás (2026-06-10, Supabase Observability)

```
GET /auth/v1/.well-known/jwks.json    1561 hívás
POST /auth/v1/token?refresh_token        4 hívás
```

→ ~390× több JWKS hívás, mint amennyi tényleges auth művelet volt. Ez igazolja, hogy a JWKS minden authentikált request-nél lekérdezésre kerül.

### Hatás a Render bandwidth-re

- **Service-Initiated egress** (Render → Supabase) felesleges traffic: kb. ~2-3 KB JWKS payload × authentikált kérés count
- **Latency**: minden API request extra ~50-200 ms-os Supabase round-trip-pel terhelt
- A 2026-06-10-i Render figyelmeztetés (3.77 GB / 5 GB elhasznált bandwidth) Service-Initiated 1.71 GB komponensébe ez direktbe beleszámít

---

## Story

Mint **platform-üzemeltető**, szeretném, hogy **a backend csak egyszer hozza létre a JWKS klienst és újrahasználja a cache-elt signing key-eket**, hogy **csökkenjen a Supabase egress és minden authentikált API kérés latency-je**.

---

## Elfogadási kritériumok

### A — JWKS client singleton

1. [ ] A `makeJwksClient()` hívást **module-szintre** kell hoisztolni: a kliens egyszer jön létre az első `authMiddleware` hívásnál (lazy singleton), és minden további hívás ugyanazt használja.
2. [ ] A `getSigningKey()` függvény a singletonon hívja a `getSigningKey(kid)`-et — ne gyártson új instance-t.
3. [ ] A `cache: true, cacheMaxAge: 600000` (10 perc) opciók megmaradnak — most már ténylegesen érvényesülnek.
4. [ ] **`SUPABASE_URL` env változó hiányának kezelése**: a singleton lazy módon készüljön (nem importáláskor), hogy a hibaüzenet az első request-nél jöjjön elő, ne process startup-kor (mert akkor unit teszt is feldől).

### B — Tesztek

1. [ ] **Unit teszt** (`auth.middleware.test.ts`):
   - Két egymás utáni `authMiddleware` hívás ugyanazzal a `kid`-del **csak egyszer** hívja meg a JWKS-fetch-et (mock-olt `jwksClient.getSigningKey`-en keresztül).
   - Eltérő `kid`-del 2 fetch történik.
2. [ ] Minden meglévő auth-teszt zöld marad.

### C — Smoke verifikáció

1. [ ] Lokálisan elindítva a backendet, és 5 különböző `/api/...` kérést küldve egy valós Supabase token-nel: a backend log-ban (vagy Supabase Observability-ben rövid ablakon belül) **csak egy** JWKS request látszik.

## Technikai megjegyzések

### Implementáció (vázlat)

```ts
// auth.middleware.ts
let jwksClientInstance: jwksClient.JwksClient | null = null

function getJwksClient(): jwksClient.JwksClient {
  if (jwksClientInstance) return jwksClientInstance
  const supabaseUrl = process.env['SUPABASE_URL']
  if (!supabaseUrl) throw new Error('SUPABASE_URL is not set')
  jwksClientInstance = jwksClient({
    jwksUri: `${supabaseUrl}/auth/v1/.well-known/jwks.json`,
    cache: true,
    cacheMaxAge: 600000,
  })
  return jwksClientInstance
}

async function getSigningKey(kid: string): Promise<string> {
  const key = await getJwksClient().getSigningKey(kid)
  return key.getPublicKey()
}
```

> **Lazy singleton** — nem module-load-időben példányosítunk, mert akkor minden teszt-szerveles fail-elne, ahol nincs `SUPABASE_URL`. Az első request triggerli a létrehozást.

### Miért nem `Map<kid, publicKey>` saját cache?

A `jwks-rsa` package már TTL-es LRU cache-t implementál (`cacheMaxAge` + `rateLimit` opciókkal). Ha singleton-ként használjuk, ingyen megkapjuk. **Saját cache-t építeni YAGNI.**

### Kulcs-rotation

A 10 perces TTL után a kliens újra fetch-eli a JWKS-t — ha Supabase rotálja a kulcsot, max 10 perc késéssel látjuk. Ez elfogadható (Supabase ritkán rotál, és a régi kulcs egy ideig még él párhuzamosan).

## Mit NE csinálj

- Ne példányosítsd a klienst module-load-kor (top-level `const`) — `SUPABASE_URL` hiánya a teszteket eldönti.
- Ne építs saját Map alapú cache-t — a `jwks-rsa` belső cache-e elég.
- Ne állíts `cacheMaxAge`-t 1 óránál nagyobbra — kulcs-rotation ablak ne legyen túl hosszú.

## Várt hatás

- **JWKS fetch volume**: ~1561 → ~6-10 / nap (azaz 99%+ csökkenés a JWKS-en)
- **Auth latency**: minden authentikált request -50-200 ms (cache hit esetén)
- **Service-Initiated egress**: kis darabszám, de a teljes Supabase forgalmat is csökkenti — nyomozzon tovább, ha az 1.71 GB nem esik vissza érdemben
