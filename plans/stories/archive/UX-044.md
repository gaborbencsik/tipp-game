# UX-044: Admin bracket körönkénti participantok kiértékelése – új adatmodell

## Háttér

Az UX-042-ben bevezettünk egy körönkénti csapatlistás selector felületet az admin `bracket_progression` kiértékeléséhez. A jelenlegi implementáció minden körben a **továbbjutókat** (winners-map) tárol. Ez azonban nem modellezi helyesen a szórt igazságot: az admin számára a kör **résztvevőit** kellene kiválasztani, nem csak a winnereket.

Például a Last 32 kör során az admin 32 csapata közül választ (nem pedig „az összes közül választ 16 továbbjutót"), majd a Last 16 körben csak az előző körben kiválasztott 32-ből választ 16-ot, stb.

Ezzel párhuzamosan: a **bronze meccs** és a **champion** körök logikailag is hiányoztak az UX-042-ből. A champion kör a final 2 csapatából 1-et kijelöl bajnoknak. A bronze kör az SF vesztesei közül 1-et az elválasztott 3. helyezettnek.

Ez a story az **adatmodell átalakítása** (backend + frontend), **új körök hozzáadása**, és a **scoring logika frissítése** ezek alapján.

## Jelenlegi helyzet

- `BracketProgressionCorrectAnswer` jelenleg nem létezik formálisan — a `correctAnswer` egy `{ winners: { matchId: teamId } }` map.
- Bronze és champion körök az admin felületen nem szerkeszthetők.
- A cascade pool UX-042-ben a last_32-től kezd, az adott körben kiválasztott csapatokhoz viszonyított szűrés nélkül.
- A migráció az UX-042 után is lehetséges (backward-kompatibilitás szükséges).

## Adatmodell (backend)

### Új interface: BracketProgressionCorrectAnswer

```typescript
interface BracketProgressionCorrectAnswer {
  readonly participants: {
    readonly last_32: readonly string[]  // 32 csapat UUID
    readonly last_16: readonly string[]  // 16 csapat UUID, ⊆ last_32
    readonly qf: readonly string[]       // 8 csapat UUID, ⊆ last_16
    readonly sf: readonly string[]       // 4 csapat UUID, ⊆ qf
    readonly final: readonly string[]    // 2 csapat UUID, ⊆ sf
  }
  readonly champion: string              // 1 csapat UUID, ∈ final
  readonly bronzeWinner: string | null   // 1 csapat UUID, ∈ (sf \ final)
}
```

### Backward kompatibilitás

- A meglévő `{ winners: {...} }` alakú `correctAnswer`-okat a backend **továbbra is támogatja**.
- Az új API GET végpontok az új alakot adják vissza (migráció vagy on-the-fly konverzió).
- A `scoreBracketProgression` mindkét alakot érti (legacy winners-map + új participants-shape).

### User-facing `BracketProgressionAnswer` (szept — nem változik)

```typescript
interface BracketProgressionAnswer {
  readonly winners: {
    readonly [matchId: string]: string  // match UUID → csapat UUID
  }
}
```

A user továbbra is a teljes 32 mérkőzés fa alapján tipp le (match-szintű, winners-map). A scoring a felhasználó winners-mapjéből **derives** a csapathalmazokat az egyes körökön (last_32: az összes slot csapatai, stb.), majd metszi az admin correctAnswer participants halmazaival.

## Scoring logika (backend)

### Függvény: `scoreBracketProgression`

```typescript
export function scoreBracketProgression(
  userAnswer: BracketProgressionAnswer,
  correctAnswer: BracketProgressionCorrectAnswer | LegacyBracketProgressionCorrectAnswer,
  config: ScoringConfig
): number
```

**Logika:**

1. Mindkét `correctAnswer` alaknál: az admin participants halmazokat lekérjük.
   - Ha új alakú: közvetlenül a `participants` objektumból.
   - Ha legacy (`winners` map): a meglévő `deriveBracket()` logikával kivonjuk a participants halmazokat.

2. Mindkét oldalnál: a user `winners` mapjéből derives a körönkénti csapat halmazokat (matching against template slots).

3. Metszet számítás és pontszámítás körönként:
   - last_32: `|userTeams_32 ∩ correctTeams_32|` × 2p
   - last_16: `|userTeams_16 ∩ correctTeams_16|` × 3p
   - qf: `|userTeams_8 ∩ correctTeams_8|` × 4p
   - sf: `|userTeams_4 ∩ correctTeams_4|` × 6p
   - final: `|userTeams_2 ∩ correctTeams_2|` × 8p
   - champion: user `winners[finalMatchId] === correctAnswer.champion` ? 10p : 0p
   - bronze: 0p (vagy `user != correctAnswer.bronzeWinner` → 0p; egyébként 0p)

4. Összeg = összes körből.

## Frontend adatmodell

### Komponens: `BracketRoundTeamList.vue` v2

**Props:**
- `typeId: string` — special type ID
- `template: BracketProgressionTemplate` — bracket template
- `correctGroupStandings: AllGroupsStandingAnswer` — csoport végeredmények (az admin már kiértékelte)
- `currentCorrectAnswer: BracketProgressionCorrectAnswer | null` — mentett correct answer (vagy legacy alakú)
- `allTeams: Team[]` — összes csapat (pool-hoz)
- `onSave: (correctAnswer: BracketProgressionCorrectAnswer) => Promise<void>` — save callback

**Megjelenítés: 7 szekció (körök)**

1. **Last 32**: pool = `GET /api/teams` teljes lista. Admin max 32 csapatot választ.
2. **Last 16**: pool = az előző körben (last_32) kiválasztott 32 csapat. Admin max 16-ot.
3. **QF**: pool = last_16 csapatai. Admin max 8-at.
4. **SF**: pool = qf csapatai. Admin max 4-et.
5. **Final**: pool = sf csapatai. Admin max 2-t.
6. **Champion** (ÚJ): pool = final 2 csapata. Admin 1-et választ (a bajnok).
7. **Bronze** (ÚJ): pool = auto-derive (sf \ final, azaz az SF-ben voltak de a finalba nem kerültek). Admin 1-et választ (bronzérmes). Read-only pool (a szelektor lezárva).

Minden szekciónál:
- Chip-lista (név + logó).
- Kattintható toggle select/deselect.
- Korlátozás enforce: túllépés után gomb disabled + toast.
- Cascade: ha az előző kör nincs mentett, a jelenlegi disabled + üzenet.
- Mentés gomb szekciónként: `PATCH /api/admin/special-types/:typeId/correct-answer` + kiértékelés POST.

### Helper: `buildCorrectAnswer`

```typescript
export function buildCorrectAnswer(
  round: BracketRound,
  selectedTeamIds: string[],
  prevCorrectAnswer: BracketProgressionCorrectAnswer | null
): BracketProgressionCorrectAnswer
```

Frissíti az adott körben az `participants[round]` mezőt, megtartja az előzőeket.

## Backend validáció

### Függvény: `validateBracketProgressionCorrectAnswer`

```typescript
export function validateBracketProgressionCorrectAnswer(
  answer: BracketProgressionCorrectAnswer
): ValidationResult
```

**Ellenőrzések:**

- `participants.last_32.length === 32`
- `participants.last_16.length === 16` és `⊆ last_32`
- `participants.qf.length === 8` és `⊆ last_16`
- `participants.sf.length === 4` és `⊆ qf`
- `participants.final.length === 2` és `⊆ sf`
- `champion ∈ final`
- `bronzeWinner === null` vagy `bronzeWinner ∈ (sf \ final)`

**Error messages (human-readable):**
- `'bracket_round_size_mismatch:last_16'` — 16-tól eltérő darabszám
- `'bracket_subset_violation:qf'` — qf ⊄ last_16
- `'bracket_champion_not_in_final'` — champion nincs a final halmazban
- `'bracket_bronze_invalid'` — bronzeWinner az SF-ben vagy a final-ban van

## Migráció (backend)

### SQL: `00XX_bracket_correct_answer_shape.sql`

Átkonvertálja a meglévő legacy `winners`-map alakú `correctAnswer`-eket az új `participants` alakra az adott `special_type_definition`-nél, ahol a type `bracket_progression`:

```sql
UPDATE special_type_definitions
SET correct_answer = jsonb_build_object(
  'participants', jsonb_build_object(
    'last_32', (correct_answer->'_derived_participants'->'last_32'),
    'last_16', (correct_answer->'_derived_participants'->'last_16'),
    ...
  ),
  'champion', (correct_answer->>'_derived_champion'),
  'bronzeWinner', (correct_answer->>'_derived_bronze')
)
WHERE type_id = 'bracket_progression'
  AND correct_answer ? 'winners'
  AND NOT (correct_answer ? 'participants');
```

**Megjegyzés**: Ha a deriváció sikertelen (pl. hiányzik az `all_groups_standing` correctAnswer), a migráció meghagyja a régit — a backend mindkettőt érti.

## Tesztelés

### Backend unit tesztek

1. **`validateBracketProgressionCorrectAnswer.spec.ts`**:
   - Happy path: érvényes correctAnswer.
   - Round size mismatches (last_16 ≠ 16, stb.).
   - Subset violations (qf ⊄ last_16, stb.).
   - Champion/bronze constraint violations.

2. **`scoreBracketProgression.spec.ts`**:
   - Happy path: user + admin tipp → metszet × pontozás.
   - Legacy winners-map compatibilitás: ugyanaz a felhasználói tipp ugyanazt a pontot kapja, függetlenül attól, hogy az admin new vagy legacy alakú.
   - 0 pont: egyáltalán nincsen metszet.
   - Champion exact match: 10p.
   - Bronze: 0p.

### Frontend unit tesztek

3. **`BracketRoundTeamList.spec.ts`**:
   - 7 szekció render.
   - Chip toggle select/deselect.
   - Exact count enforcement (last_32 = 32, stb.).
   - Cascade: előző kör nincs mentett → jelenlegi disabled.
   - Bronze pool auto-derive (sf \ final).
   - Save emit: teljes `BracketProgressionCorrectAnswer`.

4. **`buildCorrectAnswer.spec.ts`**:
   - Last_32 frissítés: az előző null-ból új 32 csapat.
   - Last_16 frissítés: az előző last_32 + új 16.
   - Champion frissítés: az előző final 2-ből új 1 csapat.

### E2E teszt (Playwright)

5. **`admin-bracket-correct-answer-shape.spec.ts`**:
   - Admin login.
   - `/admin/tournament-evaluation` → `all_groups_standing` correctAnswer mentése (hogy Last 32 pool elérhető).
   - `bracket_progression` → Last 32 körben: 32 csapat chip-grid, admin 32-t választ, gomb enabled.
   - Mentés gomb: PATCH + POST → 200 OK.
   - Toast: „Last 32 kiértékelve: N felhasználó · X pont".
   - Last 16 kör enable-ként jelenik meg (cascade lock feloldódik).
   - Last 16 körben: pool az előző körből (32 csapat), admin 16-ot választ.
   - … (többi körön végigmegy a cascade logikán).
   - Champion kör: pool = final 2, admin 1-et választ.
   - Bronze kör: pool = sf \ final (2 csapat, read-only), admin 1-et választ.
   - Utolsó mentés után: audit log entry `action: 'result_set'`, slice `'bronze'`.

## Elfogadási kritériumok

1. **Backend adatmodell (`BracketProgressionCorrectAnswer`):**
   - Interface definiálva: `participants`, `champion`, `bronzeWinner`.
   - Validátor függvény: round size + subset + champion/bronze constraint.
   - Unit teszt: 5+ eset (happy path + 4 constraint violation).

2. **Backend scoring (`scoreBracketProgression`):**
   - Új `participants` alakú correctAnswer pontosan pontozódik.
   - Legacy `winners`-map alakú correctAnswer továbbra is működik, ugyanaz az eredmény.
   - Unit teszt: 3+ eset (happy path + legacy kompatibilitás + 0 pont).

3. **Backend evaluate endpoint** (`POST /api/admin/special-types/:typeId/evaluate`):
   - Slice `'bronze'` és `'champion'` szeleteken végigmegy.
   - Score helyes (az új scoring fn alapján).
   - Audit log entry: `slice: 'bronze'` / `'champion'`.

4. **Frontend `BracketRoundTeamList.vue` v2:**
   - 7 szekció (last_32, last_16, qf, sf, final, champion, bronze) renderelnek.
   - Chip toggle kattintható.
   - Exact count enforcement (túllépés után gomb disabled).
   - Cascade lock: előző kör nincs mentett → jelenlegi disabled + üzenet.
   - Bronze pool read-only (csipek nem kattinthatók) és auto-derive (sf \ final).
   - Save emit: teljes `BracketProgressionCorrectAnswer` objektum.
   - Unit teszt: 6+ eset.

5. **Frontend helper `buildCorrectAnswer`:**
   - Egy körön frissít, az előzőeket megtartja.
   - Champion + bronze logikája helyes.
   - Unit teszt: 3+ eset.

6. **Frontend integráció:**
   - `AdminTournamentEvaluationView` az új `BracketRoundTeamList` komponenst használja.
   - Körönkénti PATCH + POST hívások működnek.
   - Toast üzenetek: „{körönév} kiértékelve: N felhasználó · X pont".

7. **Migráció:**
   - SQL script: meglévő winners-map adatok → új `participants` alak.
   - Backend compatibilitás: mindkét alak olvasható a GET API-kon.
   - E2E: migráció után a legacy adatok helyesen pontozódnak.

8. **E2E teszt:**
   - Teljes cascade flow: last_32 → ... → bronze, minden kör enable/disable logikája működik.
   - Audit log: minden körre `action: 'result_set'`.
   - User tipp is új pontozás alapján helyesen értékelődik.

## Technikai megjegyzések

### Backend

- **Scoring fn**: `packages/backend/src/services/scoring.service.ts` → `scoreBracketProgression` frissítése.
- **Validátor**: `packages/backend/src/db/schema/specialTypes.ts` vagy új fájl `bracketValidation.ts`.
- **Adatmodell**: `BracketProgressionCorrectAnswer` type az OpenAPI spec / TypeScript tippeknél.
- **Migráció helye**: `packages/backend/src/db/migrations/` → 0XX_bracket_correct_answer_shape.sql.
- **Route**: meglévő `POST /api/admin/special-types/:typeId/evaluate?slice=<round>` továbbra is működik, ha `slice = 'champion'` vagy `'bronze'`.

### Frontend

- **Komponens**: `packages/frontend/src/components/admin/BracketRoundTeamList.vue` — v2 (7 szekció).
- **Helper**: `packages/frontend/src/lib/bracketRoundMapping.ts` → `buildCorrectAnswer` függvény.
- **AdminTournamentEvaluationView**: az új komponenst integrálja `bracket_progression` típushoz.
- **Tesztek**: `__tests__/` almappák az adott szülő fájlok mellett.

### Backward kompatibilitás

- Régi `{ winners: {...} }` alakú correctAnswer-eket a BE validálja és pontozza.
- FE új requestet küld az új alakkal, de ha a BE régi adattal jön vissza, azt is megnyitja (read-only vagy warning).
- Migráció: opcionális, de ajánlott (a GET API már az új alakot adja).

## Kizárások

- User-facing `BracketProgressionPicker` — marad az eddigiek szerint (user továbbra is winners-map-et küld).
- Más admin típusok (group, upset) — csak `bracket_progression`-ra vonatkozik.
- Auto-trigger match-eredményekből — PLAN-001 feladata.
- Frontend csapatlistás cache + refresh (az UX-042 already has this).
