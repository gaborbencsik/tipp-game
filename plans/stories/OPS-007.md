# OPS-007: Polymarket raw_payload tárolás kivezetése

## Probléma

A `match_market_data` tábla `raw_payload` JSONB oszlopát csak **debug célból** tartottuk meg, amikor a Polymarket sync-et bevezettük. A produkcióban semmilyen kód nem olvassa, és nem is megy ki API válaszban. Mégis minden insert tárolja a teljes Polymarket event objektumot — eseményenként ~5-15 KB JSONB → TOAST-olt tárolás → DB méretnövekedés és egress (ha valaha visszaolvasnánk).

A [[OPS-004]] előtt már elindítottuk a snapshot retention-t (5 / meccs), ami megfékezte a növekedést, de a `raw_payload` továbbra is **felesleges tömeg**:

- Új inserteknél: feleslegesen tárolt ~10 KB / sor (270 meccs × 5 snapshot = 1350 sor × 10 KB = ~13 MB)
- Lekérdezéseknél (ha valaha `SELECT *` lefut): kockázat hogy egy debug/admin endpoint véletlenül visszaadja → MB-os response

**Becsült megtakarítás:** közvetlen egress 5-10% (preventív, ne legyen olyan endpoint ami ezt visszaadja), DB méret ~50-100 MB hosszabb távon.

---

## Story

Mint **platform-üzemeltető**, szeretném, hogy **a Polymarket sync ne mentse el a teljes raw_payload-ot a `match_market_data` táblába**, hogy **ne pazaroljunk DB tárhelyet és ne kockáztassuk hogy egy jövőbeli endpoint visszaadja**.

---

## Elfogadási kritériumok

### A — Insert-oldali kivezetés

1. [ ] A `polymarket.service.ts` `syncAllMatchOdds()` függvénye **ne küldjön** `rawPayload` mezőt a `db.insert(matchMarketData)`-ba
2. [ ] A típusosság miatt: a Drizzle schema `rawPayload` mezőjét tegyük **opcionálissá** (`.default(null)` vagy hagyjuk ahogy van — nullable amúgy is)

### B — Schema kommentár

1. [ ] A schema fájlban (`db/schema/index.ts:243`) tegyünk **kommentárt** a `rawPayload` mezőhöz: `// deprecated: nem írjuk az [[OPS-007]] óta. Drop migration: TBD a következő nagyobb migrációs körben.`
2. [ ] **Ne** dropoljuk az oszlopot ebben a story-ban — külön migration, külön deployment ablak

### C — Adat-cleanup (egyszeri)

1. [ ] SQL query a Supabase SQL Editor-ben futtatva (manual, az implementáció után):
   ```sql
   UPDATE match_market_data SET raw_payload = NULL;
   VACUUM (FULL) match_market_data;  -- külön statement, transaction nélkül
   ```
2. [ ] **NE futtasd** automatizáltan a backend deploy-ban — csak manual a story PR merge után

### D — Tesztek

1. [ ] **Unit teszt** (`polymarket.service.test.ts` bővítés):
   - A `syncAllMatchOdds` által hívott `db.insert()` mock-ját ellenőrizve: a `values()` argumentum **nem tartalmazza** a `rawPayload` kulcsot (vagy explicit `undefined`)
2. [ ] Minden meglévő teszt zöld marad

### E — Smoke

1. [ ] Manual: deploy után 1-2 sync ciklus után SQL query: `SELECT COUNT(*) FROM match_market_data WHERE raw_payload IS NOT NULL AND fetched_at > NOW() - INTERVAL '1 hour'` → eredmény **0**

## Technikai megjegyzések

### Érintett fájlok

**Módosított:**
- `packages/backend/src/services/polymarket.service.ts` — `rawPayload` kihagyása az insert payload-ból
- `packages/backend/src/db/schema/index.ts` — kommentár a `rawPayload` mezőhöz
- `packages/backend/tests/polymarket.service.test.ts` — új teszt

**Migration:** **nincs** új migration ebben a story-ban. Az oszlop drop egy későbbi, koordinált deployment-ben.

### Miért nem dropoljuk most az oszlopot?

- Drizzle migration generation pontatlan lehet és sok DB cleanup közben
- A `VACUUM FULL` után az oszlop NULL-jai már nem foglalnak TOAST helyet
- Egy oszlop drop irreverzibilis a már lefutott migrációból kifelé, és inkább batch-elve csináljuk később

---

## Mit NE csinálj

- Ne dropold az oszlopot ebben a story-ban
- Ne futtasd a `VACUUM FULL`-t automatikusan a deploy pipeline-ban — manual SQL Editor only
- Ne nyúlj a [[OPS-004]]-ben létrehozott `pruneOldSnapshots()` függvényhez — az retention, nem érinti a payload tartalmát

---

## Várt megtakarítás

- DB tárhely: a meglévő ~13 MB raw_payload nullá-zása → 13 MB-os DB méret csökkenés egyszeri
- Inserts forward-going: ~10 KB / sor megtakarítás (270 meccs × 5 snapshot = 13 MB ami nem nő)
- Egress: preventív (ma nem szivárog ki), 0-5% csökkenés ténylegesen
